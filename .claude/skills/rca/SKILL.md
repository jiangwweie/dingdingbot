# 根因诊断七步法

## 核心原则

### 原则 1：不要猜测根因
每一步都必须有代码行 / 调用路径 / git 变更作为证据。

### 原则 2：跨层级读源码（遇到第三方库必遵守）
遇到第三方库的问题，**不能停在 `import` 层面**。必须：
- 读到那个库的实际源码，而不是停留在文档 / 注释 / "应该有"
- 追踪跨层行为：用户代码 → 第三方库 → 底层框架 → 协议/网络
- **每一层都读，不跳层**。很多 bug 不在某一层内部，而在层与层之间

### 原则 3：排除法思维
每发现一个线索，**先问排除了哪些可能性，再决定下一步查哪里**。验证修复时：
- 不是 "看起来修了"，而是 "配置在运行时实际怎么流动的"
- 逐层验证：配置声明 → 合并逻辑 → 条件判断 → 最终执行

## 七步流程

### 1. 正向追踪调用链

从现象出发，逐层追踪代码中实际走的路径：

```
报错 / 现象 → 路由定义 → 处理方法 → 底层调用 → 最终操作
```

**具体操作**:
- 用 Grep 找到入口（API 端点 / 报错文件 / 方法名）
- 追踪到具体 repository / service 的方法调用
- 确认使用的全局变量 / 本地实例 / 临时创建
- 找到最终的底层操作（数据库查询 / 连接调用 / 文件 I/O）

**输出**: 完整的调用链，精确到文件名和行号

### 2. 逆向验证失败条件

不是"可能是 X"，而是"确切的 if 条件"：

- 什么条件下会抛出这个错误？
- 具体的异常类型是什么？（ValueError / AttributeError / ConnectionError）
- 哪一行代码会触发？
- 需要什么状态组合才会走到那行代码？

**输出**: `文件:行号 → if 条件 → 抛出异常 → 错误信息`

### 3. 变更定位

什么代码变更引入了这个问题：

- `git log --oneline -20 -- <相关文件>` 看最近变更
- `git show <commit> -- <文件>` 看具体改了什么
- 对比旧版和新版的行为差异
- 确认是哪个 commit 引入了新的风险

**输出**: `commit hash → 改了什么 → 旧版行为 vs 新版行为`

### 4. 预期-实际对比

很多 bug 的本质不是代码写错了，而是**预期模型和实际模型不匹配**。

```
预期模型: [代码/文档声称的行为]
实际模型: [实际代码执行的行为]
差距:     [不匹配的具体层级]
```

**对比维度**:
- 数据格式：预期传 string，实际传 number？
- 协议方向：预期双向心跳，实际单向？
- 时序关系：预期 A 先于 B，实际 B 先于 A？
- 返回值：预期返回 X，实际返回 None？
- 状态机：预期状态 S1→S2，实际 S1→S3？

**输出**: 对比表，明确标注预期 vs 实际 vs 差距所在

### 5. 确认当前状态

当前代码还满足那个失败条件吗？

- 读当前代码确认（不是 git 历史，是当前工作副本）
- 确认修复是否已经生效
- 确认是否有残留的风险路径

**输出**: "当前代码 X 行是 Y，失败条件 Z 是否仍存在"

### 6. 场景枚举

每个运行模式下逐个验证，不泛泛而谈：

- **实盘模式**：主进程运行中 / 启动 / 关闭时
- **回测模式**：API 端点 / 本地脚本
- **独立服务模式**：uvicorn 独立启动
- **其他模式**：根据你的项目实际情况

每个场景回答：
- 这个场景下，失败条件会不会被触发？
- 什么代码路径会导致触发？
- 有还是没有，给理由

**输出**: 表格形式，每个场景一行，明确 会 / 不会 + 原因

### 7. 风险判断

基于证据判断，不是基于可能性：

- **真实路径**：当前代码中确实存在的可执行路径 → 需要修复
- **假设路径**："未来可能有人改代码" → 不需要预先防御
- 给出明确结论：修了 / 没修 / 不需要修
- 如果要修，给出改动最小的方案（精确到哪个文件哪一行改什么）

## 禁止的输出

| 禁止 | 替代 |
|------|------|
| "我觉得可能是 X" | "代码路径确认是 X" |
| "建议加强防御" | "当前无触发路径，不需要" |
| "可能存在风险" | "场景 A 会触发，场景 B 不会" |
| 泛泛而谈的修复方案 | "文件:行号, 旧代码 → 新代码" |

## 最终输出格式

```
## 根因分析

**根因**: [一句话描述确切根因]
**触发链**: [确切的代码路径]

## 各场景验证

| 场景 | 是否触发 | 原因 |
|------|---------|------|
| ...  | ...     | ...  |

## 结论

[明确的修复状态和是否需要进一步处理]
```
