# 子任务B开发提示词：策略工作台持久化与 CRUD 接口 (SubTask B)

**角色设定**：你是一位精通 FastAPI 和数据库设计的 API 研发工程师。目前我们正在重构一个全动态化的系统监控风控框架。你的职责是打造“策略组装工作台”后端的专属 API 基础库。

## 任务背景
架构师已经在 `src/domain/models.py` 中设计好了高弹性的 `StrategyDefinition` (包含 triggers 和 filters 数组)。目前系统跑的一套总配是直接写在 `user.yaml` 里面的。
为了支撑前端能像“自定义工作流/快捷指令”那样去保存各种预设参数（模板），我们需要一个**数据库支持的独立策略模板管理接口库 (Library)**，以便让用户“一键采用”覆盖到实盘。

## 核心任务要求

### 1. SQLite 数据库模型设计与仓储建立 (Repository)
- **目标**：在基础设施层（可以在现有的 `SignalRepository` 或是新建 `StrategyRepository` 中实现），建立用于存储策略组装模板的表。
- **表结构设想**：
  - 表名：`custom_strategies`
  - 字段：`id` (主键，自增或 UUID), `name` (名称), `description` (说明文案), `strategy_json` (TEXT 字段，用于存放校验通过的序列化 `StrategyDefinition` JSON 数据), `created_at` (时间戳), `updated_at` (时间戳)。

### 2. 策略元数据动态吐出 (Schema API)
- **新增接口**：`GET /api/strategies/meta`
- **目的**：杜绝前端硬编码。返回目前后端支持了哪些 triggers (如 pinbar, engulfing) 和哪些 filters (如 ema, mtf, atr)，以及它们对应的入参格式和名称。
- **示例结构**：
  ```json
  {
      "triggers": [{"type": "pinbar", "paramsSchema": {"min_wick_ratio": "number"}}],
      "filters": [...]
  }
  ```

### 3. RESTful CRUD 路由 (Strategy CRUD API)
在 `src/interfaces/api.py` 中新增这四个模板专属接口：
- `GET /api/strategies`：获得所有保存的自定义策略列表（只返回 id, name, desc）。
- `GET /api/strategies/{id}`：获取某条具体策略的拥有完整参数的 `strategy_json` 文本数据用于回显和“一键采用”。
- `POST /api/strategies`：接收带有完整 `StrategyDefinition` 数据体的请求，校验格式无误后，序列化成 JSON 插入 SQLite 数据库。
- `DELETE /api/strategies/{id}`：从库中移除不再需要的模板。
- `PUT /api/strategies/{id}`（加分项）：如果是修改功能，也可以是新建一个 Version 作为 Snapshot 存储，满足架构师“快照插入而非覆盖”的期许。

**注意事项**：该任务处于边缘基础设施层，只操作新建表和独立的 HTTP 路由，你**不也不应该**侵入和依赖任何长连、实盘判定代码！请将路由整理干净并加上好读的 docstring 用于自动生成 Swagger。
