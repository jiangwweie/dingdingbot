# S6-1 冷却缓存优化任务废弃说明

**废弃日期**: 2026-03-30
**废弃类型**: ❌ 功能冗余 (Redundant)
**替代方案**: S6-2 信号覆盖机制

---

## 废弃原因

### 1. 信号覆盖机制已解决核心问题

**原有需求**: 防止同一信号重复通知

**现有方案**: S6-2 实现的信号覆盖逻辑已确保只有更高分的信号才能触发新通知

```python
# src/application/signal_pipeline.py:813
async def _check_cover(self, kline, attempt, score):
    """
    检查新信号是否应该覆盖现有信号
    返回：(should_cover, superseded_signal_id, old_signal_data)
    """
    # 比较分数：新信号必须分数更高才能覆盖
    if score > old_score:
        logger.info(f"Signal covering: new score ({score:.3f}) > old score ({old_score:.3f})")
        return True, old_signal_id, old_signal_data
    else:
        return False, None, None
```

### 2. 双重限制过度防御

当前实现中，冷却缓存和信号覆盖两层逻辑同时存在：

```python
# Line 429: 冷却检查
if in_cooldown and not should_cover:
    # 在冷却期内 且 不应该覆盖 → 跳过
    continue
```

**问题**:
- 增加了系统复杂度和维护成本
- 边界情况处理困难
- 调试问题时需要排查两层逻辑

### 3. 与核心设计原则冲突

系统设计原则：**One Mechanism, One Responsibility**

- 信号覆盖机制：负责"去重"（只有更好的信号才通知）
- 冷却缓存：历史遗留的临时方案，早期为防止通知轰炸

当两个机制目标重叠时，保留更优雅的那个（信号覆盖）。

---

## 影响分析

### 移除冷却缓存后的行为

| 场景 | 有冷却缓存 | 无冷却缓存 (仅信号覆盖) |
|------|-----------|----------------------|
| 同样分数的信号 | ❌ 被冷却拦截 | ✅ 会通知 |
| 更低分的信号 | ❌ 被冷却拦截 | ✅ 会通知 |
| 更高分的信号 | ✅ 覆盖旧信号 | ✅ 覆盖旧信号 |

**结论**: 移除冷却缓存后，用户会收到更多"同样质量"或"更低质量"的信号通知。

**缓解措施**:
- 如果这成为问题，可以调整信号覆盖的时间窗口（当前按周期差异化）
- 或者引入"最低分数阈值"配置
