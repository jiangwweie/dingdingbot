## 执行摘要 (Executive Summary)

本需求分析对「盯盘狗」系统的配置管理功能进行全面探索，旨在评估从当前静态 YAML 文件管理演进为更现代化配置体系的技术方案与商业价值。

**核心发现**:
- 当前系统已具备基础配置热重载能力，但缺乏版本控制、审计追踪、多环境管理能力
- 历史设计中已实现「配置快照版本化」功能（S4-1），但前端 UI 不完整，用户体验割裂
- 目标用户对配置管理的需求呈现两极分化：开发者追求灵活性，运维追求稳定性，交易员追求便捷性

**MVP 建议**: 聚焦「配置导出/导入」+「基础版本快照」，工时 2-3 人日，满足 80% 核心需求

### 1.3 核心使用场景 (User Stories)

#### 场景 1: 策略参数调优 (Trader)
> 作为交易员，我想快速调整策略参数并立即看到效果，以便优化交易表现。

**当前流程**: 编辑 user.yaml → 保存 → 调用热重载 API → 等待生效 → 观察结果
**理想流程**: UI 调整参数 → 点击「预览」→ 确认无误 → 一键应用 → 自动保存快照

#### 场景 2: 配置故障回滚 (DevOps)
> 作为运维，我想在配置出错时快速回滚到历史稳定版本，以便最小化故障影响。

**当前流程**: 查找备份文件 → 手动替换 → 重启服务 → 验证
**理想流程**: 选择历史版本 → 点击「回滚」→ 系统自动重载 → 验证成功

#### 场景 3: 多环境配置同步 (Developer)
> 作为开发者，我想将测试环境的配置快速同步到生产环境，以便减少配置错误。

**当前流程**: 手动对比两个 YAML 文件 → 复制差异 → 修改生产配置 → 验证
**理想流程**: 选择源环境和目标环境 → 对比差异 → 选择性同步 → 自动验证

#### 场景 4: 配置审计与合规 (All Users)
> 作为团队成员，我想查看配置变更历史和操作记录，以便追溯问题责任。

**当前流程**: 查询 SQLite 快照表 → 人工解读 JSON → 整理报告
**理想流程**: 查看可视化变更时间线 → 筛选操作人/时间 → 导出审计报告

## 2. 功能方案头脑风暴

### 方案 A: 轻量级备份恢复 (Backup & Restore)

**核心功能**:
- YAML 文件下载（脱敏后）
- 配置文件上传（解析验证后应用）
- 自动备份（定时/触发式）

**技术实现**:
```python
# API 端点
GET  /api/config/export        # 导出当前配置（脱敏）
POST /api/config/import        # 导入配置（需验证）
GET  /api/config/backup/latest # 获取最新备份
```

**优点**:
- 实现简单，工时短（1-2 人日）
- 满足基本备份需求
- 与现有架构兼容性好

**缺点**:
- 无版本管理能力
- 无法追溯变更历史
- 手动操作容易出错

**适用场景**: 个人开发者、单节点部署

---

### 方案 B: 版本化配置快照 (Versioned Snapshots) ⭐ 推荐

**核心功能**:
- 手动/自动保存配置快照
- 历史版本列表与对比
- 一键回滚到任意版本
- 版本号与描述标签

**技术实现**:
```python
# API 端点（已部分实现）
POST   /api/config/snapshots           # 创建快照
GET    /api/config/snapshots           # 快照列表
GET    /api/config/snapshots/{id}      # 快照详情
POST   /api/config/snapshots/{id}/activate  # 激活快照
DELETE /api/config/snapshots/{id}      # 删除快照
GET    /api/config/snapshots/{id}/diff # 版本对比（新增）
```

**数据库表（已实现）**:
```sql
CREATE TABLE config_snapshots (
    id          INTEGER PRIMARY KEY,
    version     TEXT NOT NULL,      -- 版本号如 v1.0.0
    config_json TEXT NOT NULL,      -- 序列化配置
    description TEXT,               -- 变更说明
    created_at  TEXT NOT NULL,      -- 创建时间
    created_by  TEXT DEFAULT 'user',-- 操作人
    is_active   INTEGER DEFAULT 0   -- 是否当前生效
)
```

**优点**:
- 完整的版本控制能力
- 支持快速回滚
- 已有后端基础（S4-1 已实现 80%）

**缺点**:
- 需要前端 UI 配合（S5-3 未完成）
- 存储冗余（全量保存配置）
- 无差异对比功能

**适用场景**: 生产环境、多用户协作

---

### 方案 C: 多环境配置管理 (Multi-Environment)

**核心功能**:
- 环境隔离（dev/staging/prod）
- 配置继承与覆盖
- 环境间配置同步
- 环境变量注入

**技术实现**:
```yaml
# 目录结构建议
config/
  base.yaml          # 公共基础配置
  dev.yaml           # 开发环境覆盖
  staging.yaml       # 测试环境覆盖
  prod.yaml          # 生产环境覆盖
```

```python
# API 端点
GET  /api/config/environments        # 列出所有环境
GET  /api/config/{env}               # 获取指定环境配置
PUT  /api/config/{env}               # 更新环境配置
POST /api/config/sync                # 环境间同步
GET  /api/config/diff?from=dev&to=prod  # 环境对比
```

**优点**:
- 环境隔离清晰
- 减少配置错误
- 支持渐进式发布

**缺点**:
- 架构改动大
- 维护成本高
- 学习曲线陡峭

**适用场景**: 中大型团队、多环境部署

---

### 方案 D: 配置审计与变更日志 (Audit Trail)

**核心功能**:
- 变更历史记录（谁、何时、改了什么）
- 配置差异可视化（Diff View）
- 操作日志与告警
- 合规报告导出

**技术实现**:
```python
# 审计日志模型
class ConfigAuditLog(BaseModel):
    id: int
    action: Literal['create', 'update', 'rollback', 'delete']
    user_id: str
    old_value: Optional[str]  # 变更前 JSON
    new_value: Optional[str]  # 变更后 JSON
    diff_summary: str         # 差异摘要
    timestamp: str
    ip_address: Optional[str]
```

**优点**:
- 满足合规要求
- 便于问题追溯
- 增强安全性

**缺点**:
- 存储开销大
- 实现复杂度高
- 对性能有影响

**适用场景**: 金融合规、企业级部署

---

### 方案 E: Git 集成配置管理 (GitOps)

**核心功能**:
- 配置即代码（Config as Code）
- Git 版本控制集成
- PR/MR 审批流程
- 自动 CI/CD 触发

**技术实现**:
```python
# Git 集成架构
config/
  .git/                # Git 仓库
  user.yaml            #  tracked file
  snapshots/           # 自动提交的快照

# Webhook 触发
POST /api/config/commit  # 提交到 Git
POST /api/config/push    # 推送到远程
POST /api/config/pull    # 从远程拉取
```

**优点**:
- 完整的版本历史
- 支持代码审查流程
- 与现有 DevOps 工具链集成

**缺点**:
- 实现复杂度最高
- 需要 Git 知识
- 不适合非技术用户

**适用场景**: 成熟 DevOps 团队、开源项目

---

### 方案 F: 配置中心服务 (Centralized Config Server)

**核心功能**:
- 独立配置服务
- 动态推送更新
- 配置订阅机制
- 灰度发布支持

**技术实现**:
```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  Config UI  │───>│ Config Server│<───│ DingDingBot │
└─────────────┘    └──────────────┘    └─────────────┘
                          │
                          ▼
                   ┌──────────────┐
                   │ Redis/PubSub │
                   └──────────────┘
```

**优点**:
- 解耦配置与应用
- 支持实时更新
- 可扩展性强

**缺点**:
- 架构复杂度最高
- 运维成本高
- 单点故障风险

**适用场景**: 大规模分布式系统

## 7. 关键问题与决策

### 问题 1: 快照存储策略
- **选项 A**: 全量存储（当前方案）- 简单直接，冗余高
- **选项 B**: 增量存储 - 节省空间，复杂度高

**建议**: 维持全量存储，用户规模>1000 后再优化

### 问题 2: 快照自动创建时机
- **选项 A**: 不自动创建 - 用户手动控制
- **选项 B**: 每次配置变更自动创建 - 安全但冗余
- **选项 C**: 定时创建（如每日） - 平衡方案

**建议**: 选项 C - 每日首次配置变更时自动创建

### 问题 3: 导入配置验证
- **选项 A**: 宽松验证 - 只检查基本结构
- **选项 B**: 严格验证 - 完整 Pydantic 校验

**建议**: 选项 B - 严格验证，确保配置安全
