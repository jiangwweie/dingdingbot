# 复杂任务强制检查点清单 v2.0

> **适用范围**: 所有涉及前端 + 后端 + 测试的复杂功能开发
> **违反处理**: 跳过任一检查点将被 Code Reviewer 标记为 P0 问题

| **阶段 -1** | PdM | brainstorming 探索需求 | 需求 brief |

### ⭐  Architect 必须完成以下步骤 (按顺序)：

```
步骤 1: 阅读 PRD
   ↓
步骤 2: 【强制】启动 brainstorming 技能探索技术方案
```

## 🟡 阶段 1: 任务分解 - 强制检查点

### PM 必须完成以下步骤：

```
步骤 1: 阅读架构设计文档 (docs/arch/<feature>-design.md)
   ↓
步骤 2: 阅读接口契约表 (docs/designs/<feature>-contract.md)
   ↓
步骤 3: 【强制】识别可并行执行的任务簇
   ↓
步骤 4: 【强制】标注任务依赖关系 (addBlockedBy)
   ↓
步骤 5: 【强制】输出关键路径分析
   ↓
步骤 6: 请求用户确认产品范围/技术方案/任务计划
```

### 并行任务簇识别规则 ⭐

```python
# 规则 1: 无依赖的任务可先行
先行任务 = [t for t in 任务列表 if not t.依赖]

# 规则 2: 依赖同一前置任务的任务可并行
for 前置任务 in 先行任务:
    可并行 = [t for t in 任务列表 if t.依赖 == [前置任务]]
    if len(可并行) > 1:
        标记为并行簇 (可并行)

# 规则 3: 前后端独立任务优先并行
后端任务 = [t for t in 任务列表 if t.角色 == "backend"]
前端任务 = [t for t in 任务列表 if t.角色 == "frontend"]
if 无共同依赖:
    标记为并行簇 (后端任务 + 前端任务)
```

### 任务计划输出模板

```markdown
## 任务分解 (WBS)

### 后端任务
| 任务 ID | 任务描述 | 预计工时 | 依赖 | 负责人 |
|--------|----------|---------|------|--------|
| B1 | Schema 定义 | 2h | - | backend |
| B2 | 业务逻辑 | 3h | B1 | backend |

### 前端任务
| 任务 ID | 任务描述 | 预计工时 | 依赖 | 负责人 |
|--------|----------|---------|------|--------|
| F1 | 类型定义 | 1h | B1 | frontend |
| F2 | UI 组件 | 3h | F1 | frontend |

### 测试任务
| 任务 ID | 任务描述 | 预计工时 | 依赖 | 负责人 |
|--------|----------|---------|------|--------|
| Q1 | 单元测试 | 2h | B2 | qa |
| Q2 | 集成测试 | 2h | F2 | qa |

## 关键路径
B1 → B2 → Q1 (关键路径)
F1 → F2 → Q2 (可并行)

## 并行任务簇
簇 1 (先行): B1
簇 2 (并行): B2, F1 ← 依赖 B1
簇 3 (并行): Q1, Q2 ← 依赖 B2, F2
```

## 🟢 阶段 2: 并行开发 - 强制检查点

### Coordinator 必须完成以下步骤：

```
步骤 1: 阅读任务计划和契约表
   ↓
步骤 2: 【强制】按并行任务簇调度执行
   ↓

   ↓
步骤 4: 检测任务完成 → 自动调度下一簇
   ↓
步骤 5: 所有开发完成后 → 自动调度 Reviewer 审查
```

```markdown
| B1 | Backend | Schema 定义 | ✅ 已完成 | 无 | 100% |
| B2 | Backend | 业务逻辑 | 🔄 进行中 | B1 | 50% |
```

## 🔵 阶段 4: 测试执行 - 强制检查点

### QA 必须完成以下步骤：

```
步骤 1: 阅读契约表和测试计划
   ↓
步骤 2: 【强制】测试前通知用户确认 (测试非常耗时)
   ↓
步骤 3: 等待用户回复"开始测试"
   ↓
步骤 4: 执行测试 (单元 + 集成+E2E)
   ↓
步骤 5: 生成测试报告
```

### 测试前确认模板

```markdown
📋 审查阶段已完成，准备进入测试执行阶段。

测试任务:
- 单元测试：tests/unit/ (预计 5-10 分钟)
- 集成测试：tests/integration/ (预计 10-20 分钟)
- E2E 测试：如需要 (预计 20-30 分钟)

预计总耗时：30-60 分钟

请回复"开始测试"确认执行，或"跳过测试"直接交付。
```

## 🔔 Code Reviewer 检查清单

**审查代码前必须检查以下文档是否存在**:

```
- [ ] docs/products/<feature>-brief.md (PRD)
- [ ] docs/arch/<feature>-design.md (架构设计)
- [ ] docs/designs/<feature>-contract.md (契约表)
- [ ] docs/planning/<feature>-task-plan.md (任务计划)
- [ ] docs/planning/task-board.md (状态看板)
```

**审查重点**:

```
- [ ] 架构师是否调用 brainstorming?
- [ ] 是否提供 2 个技术方案选项？
- [ ] 用户是否确认技术方向？
- [ ] 任务分解是否识别并行簇？
- [ ] 任务依赖是否正确标注？
- [ ] 测试前是否有用户确认？
```

**违反处理**:
- 发现任一检查点缺失 → 标记为 P0 问题
- 跳过架构师直接编码 → 标记为 P0 问题
- 未调用 brainstorming → 标记为 P0 问题
