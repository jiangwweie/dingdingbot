## 🔍 发现的问题

### 1. 文档命名混乱（已有实践但不统一）

**现状统计**：
```
docs/planning/交接文档：
- 2026-04-02-handoff.md (3.2K)
- 2026-04-03-handoff-morning.md (5.1K)
- 2026-04-03-handoff.md (7.1K)
- 2026-04-03-profile-handoff.md (7.3K)
- restart-handoff.md (4.6K)
```

**问题**：
- 同一天有多个交接文档（`handoff-morning`、`handoff`、`profile-handoff`）
- 命名不统一（`restart-handoff` 不符合日期+序号规范）
- 说明用户已经有交接需求，但缺少统一规范

---

### 2. 内容重复（和 planning-with-files 冗余）

**对比分析**：

| 信息类型 | progress.md (119K) | task_plan.md (51K) | 交接文档 (7.1K) | Git 历史 |
|---------|-------------------|--------------------|----------------|----------|
| **完成任务** | ✅ 详细日志 | ✅ 任务状态表 | ✅ 任务摘要 | ✅ 提交记录 |
| **待办任务** | ✅ 待办事项 | ✅ P0/P1 任务表 | ✅ 待办清单 | ❌ |
| **修改文件** | ✅ 文件列表 | ❌ | ✅ 文件清单 | ✅ git diff |
| **技术决策** | ❌ | ❌ | ✅ 决策记录 | ❌ |
| **问题清单** | ❌ | ✅ P0/P1 问题 | ✅ 问题详情 | ❌ |
| **下一步计划** | ❌ | ✅ 任务优先级 | ✅ 计划详情 | ❌ |

**重复度分析**：
- **70% 信息重复**：完成任务、待办任务、修改文件清单在多个文档中重复
- **30% 独特价值**：技术决策记录、问题详情、阻塞项分析是交接文档的独特价值

**根本原因**：
- `progress.md` 是"进度日志"（按时间记录所有工作）
- `task_plan.md` 是"任务计划"（按优先级记录任务状态）
- `交接文档` 是"会话摘要"（按会话记录工作+问题+计划）

三者的维度不同，但信息重叠。

---

### 3. 文档膨胀问题（新的上下文风险）

**当前文档大小**：
```
findings.md:    82K  ⚠️
progress.md:   119K  ⚠️⚠️
task_plan.md:   51K  ⚠️
交接文档总数:   ~30K (5 个文档)
```

**风险分析**：
- 如果每天生成 1-2 个交接文档（每个 5-7K），一个月后累积 **150-300K**
- 加上 `progress.md` (119K) 和 `findings.md` (82K)，planning 文档总大小可能超过 **500K**
- **新的上下文窗口问题**：AI 在 `/kaigong` 时读取所有这些文档，可能再次超限！

**恶性循环**：
```
上下文不够用 → 生成交接文档
↓
交接文档累积 → 文档膨胀
↓
读取所有文档 → 再次超限
↓
需要新的交接文档...
```

---

### 4. 信息冗余（可以从其他来源获取）

**冗余信息**：
1. **修改文件清单**：可以从 `git diff --stat` 直接获取
2. **Git 提交记录**：可以从 `git log --since="00:00"` 直接获取
3. **任务状态变更**：可以从 `task_plan.md` 或 `tasks.json` 直接获取

**独特信息**（真正需要交接的）：
1. **技术决策记录**：为何选择方案 A 而非方案 B（无法从 Git 推导）
2. **问题详情分析**：P0 问题的根因、影响范围、修复建议（无法从代码推导）
3. **阻塞项与依赖**：任务间的依赖关系（无法从 Git 推导）
4. **影响范围分析**：修改的代码对其他模块的影响（需要 AI 分析）

---

### 5. AI 上下文管理能力不足

**根本问题**：
- AI 不会主动"忘记"无关信息（如早期的决策、已完成的任务）
- AI 不会主动"压缩"重复信息（如相似的工作描述）
- AI 不会主动"清理"历史文档（如归档旧交接文档）

**导致后果**：
- 上下文累积越来越长
- 重要决策被淹没在大量无关信息中
- AI 需要读取所有文档才能恢复上下文

### 方案 2: 自动归档旧交接文档（防止膨胀）

**核心理念**：定期清理旧交接文档，归档到压缩文件。

**归档策略**：
```python
# 每周执行一次（在 /kaigong 或 /shougong 时）
def archive_old_handoffs():
    """归档超过 7 天的交接文档"""
    planning_dir = Path("docs/planning")
    archive_dir = planning_dir / "archive"
    archive_dir.mkdir(exist_ok=True)

    # 找出超过 7 天的交接文档
    cutoff_date = datetime.now() - timedelta(days=7)
    old_handoffs = [
        f for f in planning_dir.glob("*-handoff.md")
        if datetime.fromtimestamp(f.stat().st_mtime) < cutoff_date
    ]

    # 移动到归档目录
    for handoff in old_handoffs:
        handoff.rename(archive_dir / handoff.name)

    # 创建压缩包（可选）
    if len(old_handoffs) > 10:
        create_archive_zip(archive_dir)
```

**归档后目录结构**：
```
docs/planning/
├── 2026-04-03-handoff.md         ← 最近 7 天的交接文档
├── archive/                      ← 归档目录
│   ├── 2026-03-25-to-2026-04-02-handoffs.zip  ← 压缩包
│   └── README.md                 ← 归档说明
├── progress.md                   ← 保留（进度日志）
├── findings.md                   ← 保留（技术发现）
└── task_plan.md                  ← 保留（任务计划）
```

**优点**：
- 防止文档膨胀
- 保留历史记录（可追溯）
- 减少上下文占用（AI 不读取归档文档）

**缺点**：
- 丢失历史交接文档（虽然可以解压查看）
- 需要额外的归档逻辑

### 方案 3: AI 智能上下文压缩（根本解决方案）

**核心理念**：AI 主动识别重要信息，压缩无关信息。

**压缩策略**：
```python
def compress_context(context):
    """AI 主动压缩上下文"""

    # 1. 识别重要信息（保留）
    important = {
        "active_tasks": extract_active_tasks(),      # 当前进行中的任务
        "p0_issues": extract_p0_issues(),            # P0 问题清单
        "tech_decisions": extract_decisions(),       # 关键技术决策
        "blocking_deps": extract_blocking_deps(),    # 阻塞依赖
    }

    # 2. 识别历史信息（压缩）
    history = {
        "completed_tasks": compress_completed(),     # 已完成任务（仅保留摘要）
        "old_decisions": archive_old_decisions(),    # 旧决策（归档）
        "old_issues": archive_resolved_issues(),     # 已解决问题（归档）
    }

    # 3. 返回压缩后的上下文
    return {
        "important": important,   # 详细保留
        "history": summary(history),  # 仅保留摘要
        "archive": archive(history)   # 彻底归档
    }
```

**AI 行为示例**：
```
AI 在 /kaigong 时：

1. 读取 progress.md (119K)
   → 压缩为：
   - "最近 3 天完成 5 个任务（详见 archive）"
   - "今日待办：P0 参数校验修复"

2. 读取 task_plan.md (51K)
   → 压缩为：
   - "P0 任务：3 个待办（详见摘要）"
   - "P1 任务：1 个待办"

3. 读取最新交接文档 (7.1K)
   → 保留（包含决策和问题分析）

4. 不读取归档文档（减少上下文）
```

**优点**：
- 根本解决上下文问题
- AI 主动管理上下文
- 无需用户手动操作

**缺点**：
- 需要 AI 能力提升（识别重要信息）
- 实现复杂度高

### 方案 4: 任务拆分（从源头解决问题）

**核心理念**：如果任务大到需要多个会话，应该拆分成更小的任务。

**任务拆分原则**：
```
单次会话可完成的任务：
- ✅ 单个功能点实现（预计 < 2 小时）
- ✅ 单个 Bug 修复（预计 < 1 小时）
- ✅ 单个测试编写（预计 < 30 分钟）

需要拆分的大型任务：
- ❌ Phase 5 实盘集成（预计 3 周）→ 拆分为 15 个子任务
- ❌ OpenClaw MVP-1（预计 6-8 小时）→ 拆分为 6 个子任务
- ❌ 前端重构（预计 2 周）→ 拆分为 10 个子任务
```

**拆分后的工作流**：
```
大型任务 → 拆分为子任务 → 每个子任务单次会话完成
↓
无需交接文档（上下文足够）
↓
仅在子任务间记录状态（task_plan.md + tasks.json）
```

**优点**：
- 从源头解决上下文问题
- 无需交接文档（单次会话完成）
- 任务粒度更细（易于追踪）

**缺点**：
- 不适用所有场景（有些任务确实需要多会话）
- 拆分成本（需要提前规划）

---

### 方案 5: 现状优化（保守改进）

**核心理念**：保留当前设计，但优化命名和归档。

**改进措施**：

1. **统一命名规范**：
   - 强制使用 `<YYYYMMDD>-<序号>-handoff.md` 格式
   - 删除不符合规范的交接文档（如 `restart-handoff.md`）

2. **限制每日交接数量**：
   - 每天最多生成 2 个交接文档（如 `20260403-001`、`20260403-002`）
   - 超过限制时提示用户"今日交接文档已超限，建议合并"

3. **自动归档机制**：
   - `/shougong` 时自动归档超过 7 天的交接文档
   - 保留最近 7 天的交接文档，其余移到 `archive/`

4. **文档大小限制**：
   - 单个交接文档不超过 5K
   - 超过限制时提示用户"内容过长，建议精简"

**优点**：
- 最小改动（不破坏现有习惯）
- 解决命名混乱问题
- 防止文档膨胀

**缺点**：
- 未解决内容重复问题
- 未解决上下文根本问题
