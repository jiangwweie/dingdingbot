# Backtest Studio PRD

> 日期：2026-04-22
> 阶段定位：**低优先级并行线**
> 项目形态：独立回测前端（建议目录：`apps/backtest-studio`）
> 当前决策：采用 **B 方案**，先做设计与静态壳子，再接真实 API

---

## 一、背景与目标

当前策略主线已经收敛到：

- `ETH/USDT:USDT`
- `1h`
- `LONG-only`
- 基准参数已冻结

现有 `web-front` 中虽然已有 `PMSBacktest`、`BacktestReports` 等页面，但其信息架构混有：

- 旧策略工作台语义
- 旧导航迁移包袱
- 模板导入/策略配置混合职责

因此，后续不再把现有 `web-front` 作为回测效率工具的主演进载体，而是新建一个**只服务回测**的独立前端：

- 名称暂定：`backtest-studio`
- 定位：个人回测工作台
- 职责：参数输入 -> 执行回测 -> 查看结果 -> 与基准参数比较

---

## 二、目标

1. 提供一个专门服务个人研究的回测前端，不再承接旧策略工作台职责
2. 让基准参数回测、参数微调回测、结果对比进入更顺手的闭环
3. 在不打断订单对账 / 执行链主线的前提下，以独立子项目方式并行推进
4. 第一版先完成 UI 壳子与交互结构，后续再接真实 API

---

## 三、用户画像

### 核心用户

- 你自己
- 典型使用方式：
  - 频繁运行单品种回测
  - 围绕基准参数做小范围调参
  - 快速看结果、看成本、看回撤、看和基准的差异

### 非目标用户

- 通用策略配置用户
- 实盘交易执行用户
- 订单/仓位运营用户
- 需要复杂模板管理与多角色协作的团队用户

## 四、用户故事

### US-001：快速加载基准回测参数

**Description:** 作为策略研究者，我希望一进入页面就能加载当前基准参数，这样我能直接在此基础上做微调。

**Acceptance Criteria:**
- [ ] 页面提供“加载基准参数”入口
- [ ] 基准参数加载后，表单可见且可编辑
- [ ] 基准参数加载不会依赖旧策略工作台页面
- [ ] 页面会明确显示当前是否处于“基准参数模式”

### US-002：编辑最小参数集并发起回测

**Description:** 作为策略研究者，我希望只编辑当前真正在调的参数，并快速运行回测，这样我不会被旧前端的大量无关字段干扰。

**Acceptance Criteria:**
- [ ] 第一版仅展示最小参数集
- [ ] 参数区至少包含：交易对、周期、时间范围、初始资金、关键策略参数、风险参数
- [ ] 用户可一键执行回测
- [ ] 执行中有明确的 loading 状态
- [ ] 执行失败时能显示后端错误信息

### US-003：查看当前回测结果总览

**Description:** 作为策略研究者，我希望快速看到回测核心指标，这样我能第一时间判断本次调整是否值得继续。

**Acceptance Criteria:**
- [ ] 展示总收益、总盈亏、胜率、交易数、最大回撤
- [ ] 展示成本相关指标：手续费、滑点、资金费
- [ ] 指标布局在桌面端和移动端都可读
- [ ] 空状态和错误状态明确可见

### US-004：将当前结果与基准参数结果比较

**Description:** 作为策略研究者，我希望直接看到当前结果相对基准参数的变化，这样我能避免只盯单次结果做错误判断。

**Acceptance Criteria:**
- [ ] 页面支持“当前结果 vs 基准结果”对比
- [ ] 至少比较：总收益、总盈亏、胜率、交易数、最大回撤
- [ ] 差异字段有正负方向提示
- [ ] 当基准结果不可用时有清晰提示

### US-005：保留报告与旧前端资产作为参考，而不是耦合依赖

**Description:** 作为开发者，我希望新前端只复用字段契约和少量展示逻辑，而不继续继承旧模块结构，这样新项目边界才会稳定。

**Acceptance Criteria:**
- [ ] 新项目不直接依赖旧 `PMSBacktest` 页面结构
- [ ] 可参考现有 `backtest.ts` 类型定义和图表组件语义
- [ ] 不再接入旧策略工作台路由和模板管理入口

## 五、功能需求

### FR-1：独立前端项目

系统必须以独立子项目形式承载新前端，建议目录：

- `apps/backtest-studio`

### FR-2：仅覆盖回测场景

系统必须只服务回测，不承接以下职责：

- 实盘执行
- 订单管理
- 仓位管理
- 旧策略工作台
- 策略模板管理

### FR-3：提供最小回测配置区

第一版配置区至少包含：

- `symbol`
- `timeframe`
- `start_time`
- `end_time`
- `initial_balance`
- 关键策略参数
- 关键风险参数

### FR-4：提供基准参数加载能力

系统必须支持加载当前主线基准参数，并允许在此基础上修改后运行回测。

### FR-5：接入 PMS 回测主接口

后续真实 API 接线时，优先接入：

- `POST /api/backtest/orders`

对应现有前端契约：

- `PMSBacktestRequest`
- `PMSBacktestReport`

### FR-6：展示核心结果总览

结果页第一版至少展示：

- 总收益率
- 总盈亏
- 胜率
- 总交易次数
- 最大回撤
- 手续费
- 滑点成本
- 资金费

### FR-7：支持基准对比

第一版必须支持当前结果与基准结果的指标比较。

### FR-8：静态壳子优先

在真实 API 未冻结前，第一阶段允许采用 mock 数据完成：

- 页面布局
- 交互流程
- 状态管理
- 对比视图

### FR-9：历史报告作为后续迭代项

历史报告列表、详情弹窗、复杂分析图表不进入第一版 MVP 必选项。

## 七、信息架构与页面范围

### 7.1 MVP 页面建议

#### 页面 1：Backtest Studio 主页面

包含 4 个主区块：

1. **配置区**
   - 标的
   - 周期
   - 时间范围
   - 初始资金
   - 成本模式（后续可扩展）

2. **参数区**
   - 当前主线策略的最小参数集
   - 基准参数加载
   - 恢复默认/基准

3. **结果区**
   - 总览指标卡片
   - 成本卡片
   - 可选：简版权益图

4. **对比区**
   - 当前 vs 基准
   - 差异提示

#### 页面 2：Reports（可后置）

后续迭代再接：

- 历史报告列表
- 报告详情
- 高级图表分析

### 7.2 推荐布局

- 左侧：配置 + 参数
- 右侧：结果总览 + 对比
- 桌面端优先优化
- 移动端保证可读，不追求完整桌面体验

## 八、技术方案

### 8.1 方案选择

已确认采用 **B 方案**：

- 与订单对账主线并行推进
- 先做设计
- 再做静态壳子
- 最后接真实 API

### 8.2 目录建议

建议新建：

- `apps/backtest-studio`

不建议继续把新需求塞入：

- `web-front/src/pages/PMSBacktest.tsx`

### 8.3 可复用资产

允许参考但不直接继承的资产：

- `web-front/src/types/backtest.ts`
- `web-front/src/lib/api.ts` 中的回测请求/响应契约
- `web-front/src/components/v3/backtest/*`

### 8.4 状态管理建议

第一版建议从轻量开始：

- 本地组件状态 + 一个简洁的 API 封装层即可
- 不急着引入复杂全局状态

### 8.5 UI 风格建议

新前端不应再沿用“旧后台工具页”的松散组合风格，建议：

- 更像研究工作台，而不是运维后台
- 强调信息密度和快速比较
- 优先服务桌面使用场景
