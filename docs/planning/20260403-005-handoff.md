## 💡 技术决策记录（独特价值）

### 决策 1: 两阶段修复 asyncio.Lock + lifespan（DEBT-6 + DEBT-7）

**背景**: API 503 错误，诊断发现 lifespan 缺少初始化，但直接修复会引入 asyncio.Lock 死锁风险

**选择方案**: PR-1（修复 asyncio.Lock）→ PR-2（实施 lifespan 初始化）

**拒绝方案**: 直接实施 lifespan 初始化（有死锁风险）

**理由**:
- SignalRepository 和 ConfigEntryRepository 的 lock 在 `__init__` 中创建
- uvicorn --reload 热重载会创建新事件循环
- 新事件循环无法使用旧循环的 lock → 死锁

**影响**:
- 所有 Repository 统一使用 `_ensure_lock()` 延迟创建模式
- 架构一致性提升，支持热重载

**后续**: 无（已完成）
