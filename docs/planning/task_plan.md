### ADR-2026-04-06: 模块边界重新划分

**决策内容**: 重构 application 层服务模块

**新架构**:
```
application/
  - OrderLifecycleService (新增：订单全生命周期管理)
  - ConfigService (新增：统一配置访问)
  - BacktestService (重构：回测沙盒)
  - SystemHealthMonitor (新增：系统健康监控)

domain/
  - OrderStateMachine (订单状态转换)
  - SlippageModel (动态滑点计算)
  - FundingRateModel (资金费率模拟)
  - AttributionAnalyzer (策略归因分析)
```

**理由**:
1. 解决订单状态管理分散问题
2. 解决配置访问同步/异步混用问题
3. 统一服务层，避免 Repository 直连 API

### ADR-2026-04-06: 任务执行策略

**决策内容**: 采用关联性驱动的任务执行策略

**核心原则**:
1. **共享数据模型的任务一起实施** - 避免重复修改相同文件
2. **有依赖关系的任务连续实施** - 减少上下文切换
3. **一次重构解决多个问题** - 识别重构机会，节省工时

**关键关联关系**:
```
订单状态机 → 对账机制 → 外部订单关联 → K 线图展示
    ↓              ↓              ↓
order_repository.py (5 次修改) - 集中重构
```
