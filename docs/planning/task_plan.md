## 📌 架构决策记录 (ADR - Architecture Decision Record)

### ADR-2026-04-06: 实施方案决策

**决策日期**: 2026-04-06  
**决策者**: 架构师 + 用户  
**决策内容**: 采用 **方案 C（混合方案）** 作为实施路线图

**决策理由**:
1. 平衡短期交付和长期架构设计
2. 4 周内交付用户最需要的功能（对账 + 外部订单关联+K 线图）
3. 包含飞书风险问答 MVP，展示移动端差异化价值
4. 每周有可交付成果，风险可控

**实施方案**:

| 阶段 | 时间 | 任务 | 工时 |
|------|------|------|------|
| **第 1 周** | 实盘安全 + 回测准确性 | 订单状态机 + 滑点模型 + 资金费率 | 18h |
| **第 2 周** | 订单状态同步 | 对账机制核心 + 审计日志表 | 17.5h |
| **第 3 周** | 用户体验完整 | 外部订单关联 + K 线图 + 批量删除 | 16h |
| **第 4 周** | 差异化价值 | 飞书风险问答 MVP + 测试验证 | 12h |

**延后任务** (第 5-6 周):
- 组合回测 (16h)
- 归因分析 (12h)
- 交互式订单确认 (12-16h)
- 实时监控 (8h)

**关键架构改进**:
1. 创建 `OrderLifecycleService` 统一服务层 - 受益任务：ORD-1,2,3,5，节省约 9.5h
2. 创建 `BacktestConfig` 配置化重构 - 受益任务：BT-1,2,3，节省约 6h

**替代方案**:
- **方案 A（最小改动）**: 2 周/33.5h - 聚焦核心，但技术债累积
- **方案 B（系统性重构）**: 6 周/93.5h - 技术债清零，但周期过长

### ADR-2026-04-06: 模块边界重新划分

**决策内容**: 重构 application 层服务模块

**新架构**:
```
application/
  - OrderLifecycleService (新增：订单全生命周期管理)
  - ConfigService (新增：统一配置访问)
  - BacktestService (重构：回测沙盒)
  - SystemHealthMonitor (新增：系统健康监控)

domain/
  - OrderStateMachine (订单状态转换)
  - SlippageModel (动态滑点计算)
  - FundingRateModel (资金费率模拟)
  - AttributionAnalyzer (策略归因分析)
```

**理由**:
1. 解决订单状态管理分散问题
2. 解决配置访问同步/异步混用问题
3. 统一服务层，避免 Repository 直连 API

### ADR-2026-04-06: 任务执行策略

**决策内容**: 采用关联性驱动的任务执行策略

**核心原则**:
1. **共享数据模型的任务一起实施** - 避免重复修改相同文件
2. **有依赖关系的任务连续实施** - 减少上下文切换
3. **一次重构解决多个问题** - 识别重构机会，节省工时

**关键关联关系**:
```
订单状态机 → 对账机制 → 外部订单关联 → K 线图展示
    ↓              ↓              ↓
order_repository.py (5 次修改) - 集中重构
```
