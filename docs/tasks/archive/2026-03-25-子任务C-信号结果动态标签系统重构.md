# 子任务C开发提示词：彻底移除硬编码标签与构建动态 Tag 系统 (SubTask C)

**角色设定**：你是一位精通 Python/FastAPI/SQLite 的核心后端研发工程师。我们目前处于实盘引擎动态化的最后收尾阶段。尽管核心引擎已能执行完全自定义配置的“任意组合策略”（Triggers + Filters），但报警和落库管道 (`application` 与 `infrastructure` 层) 的契约中仍然遗留了旧时代强绑定的标签（如固定且死板的 `ema_trend` 与 `mtf_status`）。
你需要把这颗雷排掉，还给系统一个100%动态的信号标签展现能力。

## 任务痛点
每当 `SignalPipeline.process_kline` 获取到了一个报警 (SIGNAL_FIRED) 记录时，由于下游 `SignalResult` 模型契约的历史包袱，管道仍然会被迫去跑一趟 `self._legacy_engine.get_ema_trend()` 并且硬塞进发往飞书/TG/前端的数据体里。这导致即使一个策略根本没用 EMA 和 MTF 过滤，发送的卡片上还是会贴着“看涨/已确认”的错误或无关标签，而且新增的指标（例如 ATR / 成交量突增）则完全无法在卡片上展示出来。

## 核心重构任务要求

### 1. 重塑领域模型 (`src/domain/models.py`)
- **目标**：在 `SignalResult` (和相关的 API Request/Response 模型中)，移除历史的 `ema_trend` 和 `mtf_status` 字段。
- **新增**：替换为一个通用的动态字段，例如 `tags: List[str]` 或者结构化的 `tags: List[Dict[str, str]]`（如 `{"name": "EMA", "value": "Bullish"}`）。
*(提示：记得兼顾修改向后兼容，或者保证 Pydantic 解析不再依赖旧字段)*

### 2. 重写风险计算契约 (`src/domain/risk_calculator.py`)
- **目标**：修改 `RiskCalculator.calculate_signal_result()` 函数的签名。
- **调整**：将入参 `ema_trend` 和 `mtf_status` 拔掉，替换为接收 `tags: List[xxx]`。

### 3. 解除对 Legacy 引擎的依赖 (`src/application/signal_pipeline.py`)
- **目标**：在 `process_kline` 中组装发生信号的数据时，彻底移除为了硬塞标签而去调用 `_legacy_engine` 的代码。
- **动态标签生成逻辑**：
  从刚刚判定过的 `attempt.filter_results` 里面去提取“成功经过的拦截网”。
  比如：如果 `attempt` 走过了名为 `ema` 和 `volume_surge` 的两个指标。就把这两个动态组合成 `tags = ["EMA: Bullish", "VOL: Surge"]` 丢给下游。如果是自带获取趋势逻辑的指标也可以封装进动态标签中。

### 4. 仓储层落库升级 (`src/infrastructure/signal_repository.py`)
- **目标**：由于 `SignalResult` 去掉了老结构。`save_signal` 和 `get_signals` 函数中涉及到的 `ema_trend` 和 `mtf_status` 的 SQL 语句都要删掉。
- **新增**：新增一列（如果需要 ALTER TABLE）如 `tags_json TEXT`；在落库时将 `signal.tags` json 序列化塞进去；读取 `get_signals` 时再反序列化返回给前端。

### 5. 报警卡片渲染重构 (`src/infrastructure/telegram_notifier.py` / `feishu_notifier.py`)
- **目标**：打开通知组装器，你肯定会看到类似 `*EMA 趋势:* {signal.ema_trend}` 的死代码。
- **调整**：删掉它们！用一个 for 循环或者 `join()` 遍历 `signal.tags` 并炫酷地渲染在 Markdown 面板的“指标标签”区域就好。

**行动指导**：这是对通知闭环的最后一块拼图。修改范围分布广，但逻辑极其清晰：**“凡遇 EMA/MTF 硬编码变量，尽数拔除换 Tags”**。请写完切实验证测试后再提交。
