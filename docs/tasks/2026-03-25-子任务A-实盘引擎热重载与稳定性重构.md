# 子任务A开发提示词：实盘引擎热重载与稳定性重构 (SubTask A)

**角色设定**：你是一位精通 Python 异步编程、系统架构以及并发控制的高级后端工程师。当前系统的底层核心域（`models.py` 与 `strategy_engine.py`）已经由架构师升级完毕以支持多组合策略与属性环境隔离。你需要完成上层应用管道的核心改造任务。

## 任务背景
在实盘监控管道 (`src/application/signal_pipeline.py`) 中，当前代码依然硬编码依赖了旧的 `StrategyConfig` 和废弃的 `StrategyEngine`，并且在 K 线上报和状态持久化上存在多个严重的性能和并发漏洞。

## 核心任务要求

### 1. 动态引擎注入与热更新观测 (Hot-Reload Observer)
- **目标**：重写 `__init__` 函数接收 `config_manager`。
- **职责**：
  - 初始化时，通过向 `ConfigManager` 注册异步监听器 `self.config_manager.add_observer(self.on_config_updated)` 建立热重载通道。
  - 实现 `self._build_and_warmup_runner()` 函数，从 `config_manager.user_config.active_strategies` 构造 `create_dynamic_runner()` 实例。
  - **预热 (Warmup)**：重建 Runner 后，必须立马用内存缓存的 `self._kline_history` 循环喂入 `runner.update_state()`，以秒级瞬间恢复 EMA 等有状态指标，实现无缝切换。

### 2. 补漏1：热重载的并发竞争锁 (Asyncio Lock)
- **漏洞表现**：高频推流的 K 线在执行 `process_kline` 时，刚好遇上前台修改配置触发了 `on_config_updated`，导致刚建好的半成品 Runner 被拿去做误判。
- **方案**：引入 `self._runner_lock = asyncio.Lock()`。在 `process_kline` 的策略判定区间与 `on_config_updated` 彻底互斥！

### 3. 补漏2：SQLite 同步背压剥离 (Async Queue Worker)
- **漏洞表现**：目前 `process_kline` 内对每一个策略检查都执行了 `await self._repository.save_attempt()`，这会让弱并发的 SQLite 直接锁死主交易事件循环，引发 WebSocket 断线。
- **方案**：
  - 在 `SignalPipeline` 内维护一个 `asyncio.Queue` (如 `self._attempts_queue`)。
  - 在初始化时启动一个后台任务 `asyncio.create_task(self._flush_attempts_worker())`，定时或者满批次 (Batch) 进行批量落盘入库。
  - `process_kline` 里面对 Attempt 的收集改为无锁的内存 `put_nowait()` 以实现“发后即忘 (Fire-and-Forget)”。

### 4. 补漏3：哈希缓存清理 (Stale Cache)
- **漏洞表现**：同名策略（如 Pinbar）的参数被修改热重载后，受限于原来的 Dedup Key 冷却算法，该策略在接下来几小时内可能处于老旧的“假沉默冷却期”。
- **方案**：在 `on_config_updated` 中直接 `self._signal_cooldown_cache.clear()` 彻底清空所有信号冷却。

### 5. 入口升级 (main.py)
- **目标**：最后到 `src/main.py` 修改 `create_signal_pipeline()` 工厂函数，淘汰遗留的 `StrategyConfig`，将 `config_manager` 传入 Pipeline 并启动！

**注意事项**：务必保证核心流程稳定运行不可 Crash！完成任务后请提交测试。
