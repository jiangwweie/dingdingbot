# 子任务 S6-1：冷却缓存优化

**创建日期**: 2026-03-28
**优先级**: 中
**状态**: 待启动
**依赖**: 无

---

## 背景

冷却缓存（Cooldown Cache）用于防止相同币种/周期/方向/策略的信号重复发送。当前实现使用内存字典缓存上次信号时间，默认冷却期 4 小时。

在完整业务流程演示的讨论中，我们发现冷却缓存的设计需要重新评估：

### 原始设计目的
1. 防止通知轰炸（极端情况下连续多根 K 线符合形态）
2. 保护用户注意力（避免信息过载）
3. 避免过度交易（同一方向重复信号诱导频繁开仓）

### 潜在问题
1. **冷却期过长可能错过机会** - 4 小时内同一币种只能发一次信号
2. **状态复杂度** - 内存缓存、重启丢失、Key 冲突
3. **策略无状态性的矛盾** - 策略引擎是纯函数式，冷却作为外部状态破坏了这一原则

---

## 目标

实现灵活、可配置的冷却机制，支持多种冷却策略供用户选择。

---

## 考虑方案

### 方案 A：动态冷却配置

```yaml
# config/core.yaml
signal_pipeline:
  cooldown:
    enabled: true
    default_seconds: 3600  # 默认 1 小时

    # 按周期差异化
    by_timeframe:
      "15m": 1800    # 30 分钟
      "1h": 3600     # 1 小时
      "4h": 7200     # 2 小时
      "1d": 86400    # 24 小时

    # 按策略差异化
    by_strategy:
      "pinbar": 1800     # Pinbar 较常见
      "engulfing": 3600  # 吞没较少见
```

**实现改动**:
```python
# src/application/signal_pipeline.py
def _get_cooldown_seconds(self, timeframe: str, strategy_name: str) -> int:
    # 1. 查找策略特定配置
    strategy_cooldown = self._config.core_config.cooldown.by_strategy.get(strategy_name)
    if strategy_cooldown:
        return strategy_cooldown

    # 2. 查找周期特定配置
    timeframe_cooldown = self._config.core_config.cooldown.by_timeframe.get(timeframe)
    if timeframe_cooldown:
        return timeframe_cooldown

    # 3. 返回默认值
    return self._config.core_config.cooldown.default_seconds
```

**优点**: 灵活，适应不同场景
**缺点**: 配置复杂度增加

### 方案 C：移除冷却，交给策略逻辑

在策略层面增加条件来自然减少重复信号：

```python
# 示例：增加"突破前 K 线"条件
class PinbarStrategy:
    def detect(self, kline, kline_history):
        if not is_pinbar(kline):
            return None

        # 新增：突破前一根 K 线
        if len(kline_history) > 0:
            prev_high = kline_history[-1].high
            if kline.close <= prev_high:
                return None  # 未突破，不触发

        return PatternResult(...)
```

**优点**: 策略逻辑透明，用户可配置
**缺点**: 增加策略复杂度，不是所有策略都适用

### 方案 D：冷却作为可插拔过滤器（推荐）

将冷却设计为用户可选的过滤器，可添加到任意策略：

```python
# src/domain/filters/cooldown_filter.py
class CooldownFilter(FilterBase):
    """
    冷却过滤器 - 检查最近是否已发送相同信号
    """

    def __init__(self, cooldown_seconds: int, repository: SignalRepository):
        self._cooldown_seconds = cooldown_seconds
        self._repository = repository

    @property
    def name(self) -> str:
        return "cooldown"

    @property
    def is_stateful(self) -> bool:
        return False  # 状态在数据库，不在内存

    async def check(self, pattern: PatternResult, context: FilterContext) -> TraceEvent:
        last_signal = await self._repository.get_last_fired_signal(
            symbol=context.kline.symbol,
            timeframe=context.current_timeframe,
            direction=pattern.direction,
        )

        if last_signal is None:
            return TraceEvent(passed=True, reason="no_recent_signal")

        elapsed = time.time() - (last_signal.timestamp / 1000)
        if elapsed >= self._cooldown_seconds:
            return TraceEvent(passed=True, reason="cooldown_expired")
        else:
            return TraceEvent(
                passed=False,
                reason="cooldown_active",
                expected=f"wait_{self._cooldown_seconds}s",
                actual=f"elapsed_{int(elapsed)}s"
            )
```

**策略配置示例**:

```yaml
active_strategies:
  - id: "pinbar_with_cooldown"
    name: "Pinbar + 冷却过滤"
    logic_tree:
      gate: "AND"
      children:
        - type: "trigger"
          config:
            type: "pinbar"
            params:
              min_wick_ratio: "0.6"
        - type: "filter"
          config:
            type: "cooldown"
            params:
              cooldown_seconds: 1800  # 30 分钟
```

**优点**:
- 用户可自由选择是否启用
- 符合动态化架构设计
- 有明确的 Trace 记录

**缺点**:
- 开发量较大
- 需要数据库查询（需优化性能）
