# BRC Owner Console v2 需求文档

> [!IMPORTANT]
> 2026-06-08 当前产品口径覆盖说明：
> 本文档是早期 Owner Console v2 需求稿。当前权威产品模型见
> `docs/ops/knowledge-pack/CURRENT_PRODUCT_OPERATING_MODEL.md`。
> Console 现在是 Owner bounded-live operations surface，不是单纯只读观察台；
> 动作必须通过官方授权、FinalGate、Operation Layer、保护和 Review 路径。

## 1. 背景

当前 Owner Console 已完成一轮功能性验证，但界面形态存在明显问题：

- 页面像文档，不像工作台；
- 后端字段、技术状态、审计说明平铺过多；
- 英文和技术术语过多；
- 页面信息密度高，单页滚动长；
- Owner 第一眼难以判断“现在系统到底处于什么状态”；
- 安全文案重复，但不够通俗；
- 旧页面导航按后端模块组织，不符合实际使用路径。

因此，Owner Console v2 的目标不是继续补字段，而是重构为一个面向 Owner 的主控工作台。

## 2. 产品定位

Owner Console v2 是一个：

```text
策略试验主控工作台
```

它不是：

```text
文档渲染页
后端状态调试页
交易终端
下单面板
LLM 自动交易入口
```

它需要帮助 Owner 快速回答：

```text
系统现在安全吗？
账户现在怎么样？
当前策略组是什么？
有没有持仓 / 挂单 / 盈亏？
最近有没有执行意图？
现在卡在哪里？
我下一步能做什么？
我绝对不能做什么？
```

## 3. 用户角色

主要用户：

```text
Owner 本人
```

用户特征：

- 简体中文用户；
- 有程序员背景；
- 能理解必要的交易术语；
- 不希望被当成小白教育；
- 不希望页面像 Markdown 文档；
- 需要快速判断系统状态；
- 需要能展开技术详情，但不希望技术详情抢占主界面。

## 4. 核心设计原则

### 4.1 中文优先

界面文案以简体中文为主。

可保留必要英文：

```text
SOL
ETH
BTC
USDT
API
PG
TradingView
execution intent
```

技术字段不应作为主文案。

错误示例：

```text
blocked_startup_guard_runtime_coupled
```

推荐表达：

```text
运行时启动保护还没有完成预检，试验暂不能启动。
```

技术字段可作为标签或折叠详情展示。

### 4.2 单屏优先

首页应尽量一屏看清核心信息。

避免：

- 长滚动；
- 大段说明；
- 密集小字；
- 多层卡片堆叠；
- 把验收报告直接搬到页面。

### 4.3 技术详情默认隐藏

技术字段、raw JSON、source refs、operation ids、PG refs 等应默认折叠。

主界面展示通俗结论。

技术详情通过：

```text
技术详情
查看来源
查看链路
展开审计
```

等方式进入。

### 4.4 页面必须有功能性

每个页面都要回答一个明确问题：

| 页面     | 回答的问题                           |
| -------- | ------------------------------------ |
| 首页     | 当前系统和当前策略组是什么状态？     |
| 策略组   | 当前有哪些策略组和具体策略？         |
| 执行意图 | 策略本来想做什么？为什么没执行？     |
| 账户订单 | 账户、持仓、挂单、盈亏是否正常？     |
| 复盘分析 | 当前策略证据、风险、复盘结论是什么？ |
| 链路追踪 | 这件事是如何一步步走到当前状态的？   |

## 5. 新导航结构

Owner Console v2 使用左侧导航。

一级导航为：

```text
首页
策略组
执行意图
账户订单
复盘分析
链路追踪
```

旧导航全部移除，不再显示：

```text
Command Center
Markets & Orders
Campaign
Review / Evidence
Strategy Families
Fixed Rehearsal
Runtime Control
LLM Copilot
Workflow
Operator
Guide
Dashboard
Ledger
Developer Detail
Audit Trail
```

旧页面可以暂时保留代码文件，但不能出现在用户入口中。

旧路由应跳转到新首页或显示“页面已下线”。

## 6. 全局状态按钮

顶部应保留一个全局状态胶囊。

示例：

```text
● 只读观察中
```

或：

```text
● 实盘只读 · 记录意图
```

状态含义：

| 状态 | 含义                   |
| ---- | ---------------------- |
| 绿色 | 系统处于安全可观察状态 |
| 黄色 | 有阻断或需要处理       |
| 红色 | 风险阻断               |
| 灰色 | 状态未上报或未连接     |

鼠标悬停时显示短提示：

```text
当前连接真实市场，只记录信号和执行意图。不会创建执行指令，也不会下单。
```

点击后显示小弹层：

```text
当前模式：实盘只读
权限：记录意图
下单：禁止
执行指令：禁止
最近检查：xx:xx
```

不要在页面中反复写长段安全说明。

## 7. 首页需求

首页是主控面板，不是文档页。

首页第一屏应展示：

```text
当前结论
系统状态
账户状态
当前策略组
当前阻断
最近执行意图
下一步动作
```

### 7.1 首页顶部结论

示例：

```text
MI-001 SOL 已完成试验前准备，当前因启动保护未预检而阻断，试验未启动。
```

### 7.2 首页核心状态卡

至少包含：

```text
系统状态：安全 / 只读 / 记录意图 / 禁止下单
账户状态：总权益、可用余额、持仓、挂单、未实现盈亏
当前策略组：MI-001 SOL 多头
当前阻断：启动保护未预检
最近执行意图：暂无 / 最近一条
下一步：查看证据 / 查看策略组 / 查看技术详情
```

首页禁止出现：

```text
Start Trial
Place Order
Enable Live
Start Runtime
Create Execution Intent
```

## 8. 信息输入入口

控制台需要一个“信息输入入口”，但不应优先做成聊天框。

推荐交互：向导式。

示例：

```text
我想观察什么市场？
-> 趋势延续 / 震荡回归 / 波动突破 / 事件驱动
-> 选择标的
-> 系统推荐策略组
-> 展示风险和可用模式
-> Owner 确认进入观察
```

输入入口应帮助 Owner 表达市场意图，而不是填写技术参数。

初期可仅做入口占位，不必实现完整推荐逻辑。

## 9. 策略组页面需求

前端统一使用：

```text
策略组
```

技术详情中可标记：

```text
strategy family
```

策略组页面应展示：

```text
策略组列表
具体策略列表
当前状态
最近信号
最近执行意图
下一步建议
```

示例结构：

```text
MI-001 动量冲击
  - MI-001 SOL 多头
  - MI-001 BNB 多头
  - MI-001 ETH 候选
```

列表字段：

```text
策略组
具体策略
标的
方向
状态
最近信号
最近意图
下一步
```

技术字段如 family_id、version_id、PG refs 默认放入技术详情。

## 10. 执行意图页面需求

执行意图是一级页面。

它用于展示：

```text
策略本来想做什么
为什么没执行
后续走势如何
是否值得复盘
```

第一版以表格为主：

```text
时间
策略
标的
方向
意图
状态
原因
后续表现
```

执行意图不是订单。

页面必须明确：

```text
这里记录的是策略意图，不是订单，不会触发交易。
```

后续增强可接 TradingView 类图表：

```text
点击一条意图
-> 打开图表
-> 标记信号点 / 意图点
-> 查看 1h / 3h / 24h 后走势
```

图表不是 v2 首版必须项。

## 11. 账户订单页面需求

账户订单页应像交易所简化版。

主界面展示：

```text
总权益
可用余额
保证金
未实现盈亏
持仓
挂单
异常敞口
```

数据来源、truth_level、reconciliation、unknown exposure 放入折叠详情。

页面不能提供：

```text
下单
撤单
平仓
加仓
减仓
flatten
```

## 12. 复盘分析页面需求

复盘分析页展示：

```text
试验前复盘
风险披露
Owner 接受
账户事实
信号证据
执行意图证据
当前结论
```

它不是技术日志页。

如果数据缺失，应显示：

```text
暂未上报
暂无数据
未进入该阶段
```

不能伪造数据。

## 13. 链路追踪页面需求

“审计详情”改为“链路追踪”。

它用于解释：

```text
这件事是怎么一步步走到现在的？
谁确认了？
哪里阻断了？
有哪些证据？
```

推荐表现形式：

```text
纵向流程线
```

示例：

```text
● 策略候选形成        已完成
│
● Owner 风险接受      已完成
│
● PG 注册             已完成
│
● 最终复核            已完成
│
● 启动保护            阻断
```

每个节点可展开技术详情。

技术字段和 JSON 可以在此页展示。

## 14. 视觉要求

整体风格参考：

```text
现代 SaaS 工作台
交易所简化面板
Raycast / Linear / Vercel Dashboard 的克制风格
```

要求：

```text
少边框
大留白
圆角卡片
状态胶囊
清晰图标
主信息大号
技术信息折叠
避免大段文字
避免密集小字
```

不强制引入新 UI 库。优先使用现有 Tailwind 和组件体系。

## 15. 安全边界

v2 不得实现或展示可执行入口：

```text
启动试验
启动运行时
创建执行指令
下单
撤单
平仓
提现
转账
启用实盘交易
自动执行
```

如果未来某些动作需要展示，也必须：

```text
禁用
需要单独授权
有明确说明
```

当前 v2 只做展示、理解、复盘和只读观察入口。

## 16. 验收标准

### 16.1 页面验收

必须能访问：

```text
/home
/strategy-groups
/intents
/account-orders
/analysis
/trace
```

旧路由应无法打开旧页面。

### 16.2 首页验收

首页一屏应能看懂：

```text
当前系统安全状态
账户状态
当前策略组
当前阻断
最近意图
下一步动作
```

### 16.3 文案验收

页面主文案必须中文优先。

不得把技术字段作为主要标题。

### 16.4 安全验收

全站不得出现可点击的：

```text
Start Trial
Place Order
Enable Live
Execute Order
Create Execution Intent
Withdraw
Transfer
Flatten
Cancel Order
```

### 16.5 浏览器截图验收

必须截图：

```text
home.png
strategy-groups.png
intents.png
account-orders.png
analysis.png
trace.png
legacy-route-redirect.png
```

截图用于 Owner 手工验收。

## 17. 不做事项

v2 首版不做：

```text
TradingView 图表
Feishu
LLM selector
复杂调度
多策略自动路由
执行指令创建
订单链路
完整高保真设计系统
```

当前目标是先把控制台从“文档式状态页”改为“Owner 主控工作台”。
