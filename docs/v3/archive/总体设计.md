这份总体设计方案将作为我们后续模块化开发的“总纲”。我们严格聚焦于**“以仓位为核心的 PMS 模型（Position-Centric Model）”**的架构升级，剥离具体的代码实现，从顶层视角梳理业务流转。

盯盘狗 🐶 核心交易架构升级总体设计 (v3.0 蓝图)
一、 架构升级目标与核心理念
核心目标：将系统从“纯粹的信号生成与观测沙箱”，平滑演进为具备“机构级回测精度”与“实盘自动化执行能力”的订单/仓位管理系统（OMS/PMS）。

核心理念（三重解耦）：

意图与执行解耦：信号（Signal）只表达策略意图，订单（Order）负责真实的物理交互。

执行与状态解耦：订单（Order）是一次性的动作，仓位（Position）才是持久化的资产状态。

回测与实盘统一：上层策略引擎和下层仓位管理完全一致，中间通过“环境适配器（沙箱悲观撮合引擎 vs CCXT 真实网关）”进行热插拔。

二、 核心领域模型设计 (Domain Driven Design)
系统将建立四大核心实体，它们构成整个 PMS 模型的基石：

Account/Wallet (资产账户)：

职责：管理基础本金（可用现金）。

行为：接受平仓产生的已实现盈亏（Realized PnL），扣除手续费/滑点。

Signal (策略信号)：

职责：记录“为什么交易”。包含策略逻辑、触发时间、建议方向、预期止盈止损点位、形态评分等。

生命周期：生成 → 转化为订单 → 终结。

Order (交易订单)：

职责：记录“如何交易”。包含外部交易所单号（实盘用）、订单类型（市价/限价/条件单）、角色（入场/TP1/尾仓追踪）。

生命周期：PENDING → OPEN → PARTIALLY_FILLED → FILLED / CANCELED。

Position (核心仓位)：

职责：记录“持有资产现状”。包含持仓方向、开仓均价（绝对静止）、当前体积（随分批止盈动态缩减）。

生命周期：开仓（体积>0） → 减仓（体积缩小，均价不变） → 平仓（体积归零）。

三、 完整业务流程与逻辑流转 (TP1 + TP2 场景)
以下是系统应对一笔典型的“入场 -> 50% 止盈 (TP1) -> 移动追踪止损 (TP2)” 的完整生命周期流程：

阶段 1：信号生成与订单编排 (Signal to Orders)
数据摄入：系统接收最新 K 线流（回测中为批量导入，实盘为 WebSocket 流）。

严格防漏评估：策略引擎评估当前 K 线。若涉及多周期（MTF），严格执行收盘时间左侧对齐，获取已闭合的大周期数据，彻底杜绝未来函数。

信号触发：策略条件满足，生成 Signal 对象。

订单编排（风控层介入）：基于设定的多级别风控策略，系统一次性裂变出三个逻辑 Order：

Order_Entry：入场单（市价）。

Order_TP1：固定止盈单（限价，体积占比 50%）。

Order_SL：初始防守止损单（条件触发，体积占比 100%）。

阶段 2：建仓与初始状态确立 (Position Initialization)
撮合执行：回测环境调用“极端悲观撮合引擎”（加滑点/手续费），实盘环境调用 CCXT 发单。

订单成交：Order_Entry 状态变为 FILLED。

仓位实例化：系统正式生成一个 Position 对象。钱包 (Account) 扣除预估摩擦成本或冻结保证金。此时，Order_TP1 和 Order_SL 处于挂单监控状态。

阶段 3：分批止盈与动态风控 (Partial TP & Risk Mutation)
触及 TP1：行情发展触及 TP1 限价单。Order_TP1 状态翻转为 FILLED。

仓位缩容：Position 体积缩减 50%，开仓均价保持不变。计算这 50% 仓位的纯利润，划拨回 Account。

防守单突变（核心状态转移）：系统拦截底层的 Order_SL，执行两个动作：

将触发价从初始极值上移至开仓均价（推保护损，确保该笔交易立于不败之地）。

赋予其“移动止盈（Trailing）”属性，开启高水位线追踪。

阶段 4：尾仓追踪与清算 (Trailing & Liquidation)
高水位更新：行情继续推进，只要符合“防频控阶梯阈值”，系统不断向上修改 Order_SL 的触发价。

触及移动止损：行情回撤打穿动态止损线。Order_SL 状态变为 FILLED，并被标记出局语义（如 Trailing_Profit 或 Breakeven_Stop）。

仓位终结：Position 体积归零，生命周期结束。剩余资金与利润结算至 Account。关联的原始 Signal 彻底闭合。

四、 核心技术选型与落地方案
基于你现有的 v2.0 技术栈，架构升级的技术方案如下：

数据库与持久化：继续使用 SQLite，但为了支持实盘的高并发状态更新和事务一致性，建议在当前基础上引入轻量级的异步 ORM（如 SQLAlchemy 2.0 的 async 模式或 SQLModel）。

领域模型基建：深度利用 Pydantic v2 构建 Account, Signal, Order, Position 的强类型验证。

回测引擎架构：保留现有的流式驱动外壳，在内部重写 mock_match_orders()。废弃原有的信号状态直接判定，转而遍历和撮合当前处于 OPEN 状态的 Orders。

交易所网关：充分利用 CCXT.Pro 的 WebSocket 能力。实盘中，订单状态的翻转必须依赖 watch_orders 的真实推送，而非本地主观臆断。

五、 核心注意事项与风险阻断 (架构红线)
在后续的详细开发中，必须时刻警惕以下系统级风险：

并发与脏写（Race Condition）：当 TP1 成交的瞬间价格极速反向插针，极易导致系统一边在结算利润，另一边又触发了止损。规则：对同一 Position 的修改必须加异步锁（Asyncio Lock）或数据库行级锁。

实盘网关频率雪崩（API Rate Limits）：移动止损的频繁更新是封号元凶。规则：强制植入“阶梯阈值（Step Threshold）”，订单修改指令的触发必须满足最小价格变动差。

数据防漏红线（MTF Leakage）：回测的准确性命根子。规则：大级别 K 线获取函数必须通过 assert (htf_open_time + duration <= current_time) 的强制单元测试。

订单残留与孤儿订单：系统重启或断网导致本地与交易所状态脱节。规则：实盘启动阶段必须强制执行一次对账（Reconciliation），清理或同步交易所远端的所有挂单。
