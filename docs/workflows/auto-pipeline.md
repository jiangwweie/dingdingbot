# 盯盘狗全自动复杂任务交付流水线 v1.0

> **创建日期**: 2026-03-29
> **适用范围**: 涉及前端 + 后端 + 测试的复杂功能开发
> **核心原则**: 契约先行、并行执行、自动审查、无人值守

---

## 流水线总览

```
【阶段 0】需求接收 ──→ 【阶段 1】契约设计 ──→ 【阶段 2】任务分解
                                                  ↓
【阶段 6】提交汇报 ←─【阶段 5】测试执行 ←─【阶段 4】审查验证 ←─【阶段 3】并行开发
```

---

## 各阶段详细说明

### 【阶段 0】需求接收与澄清

**触发条件**: 用户提出功能需求

**Coordinator 执行步骤**:

1. **判断任务复杂度**
   - 涉及前端 + 后端 + 测试 → 启动全自动工作流
   - 仅单一模块 → 直接分配对应角色

2. **需求澄清**（如需要）
   - 调用 `brainstorming` 技能探索需求边界
   - 识别隐性依赖和风险点

3. **输出**: 需求澄清摘要（如调用过 brainstorming）

---

### 【阶段 1】契约设计（对齐的关键！）

**目标**: 创建《接口契约表》，作为后续所有工作的 SSOT（Single Source of Truth）

**Coordinator 执行步骤**:

1. 创建契约文件：`docs/designs/<task-name>-contract.md`
2. 使用模板：`docs/templates/contract-template.md`
3. 填写以下内容：
   - API 端点定义（路径、方法）
   - 请求/响应 Schema（字段名、类型、必填、说明）
   - 错误码定义
   - 前端组件 Props 定义
4. 将契约表分发给 Backend/Frontend/QA 角色

**契约表示例结构**:

```markdown
# <功能名称> - 接口契约表

## API 端点
| 端点 | 方法 | 说明 |
|------|------|------|
| /api/xxx | POST | 说明 |

## 请求 Schema
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| field_name | string | 是 | 说明 |

## 响应 Schema
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| field_name | string | 是 | 说明 |

## 前端 Props
| 组件 | Prop | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| ComponentName | propName | string | 是 | 说明 |
```

---

### 【阶段 2】任务分解与依赖

**目标**: 将需求拆分为可并行执行的独立任务

**Coordinator 执行步骤**:

1. **任务分解**
   ```python
   # 使用 TaskCreate 创建任务
   - Task 1: 后端 - 实现 API Schema 定义
   - Task 2: 后端 - 实现业务逻辑
   - Task 3: 后端 - 编写单元测试
   - Task 4: 前端 - 定义 TypeScript 类型
   - Task 5: 前端 - 实现 UI 组件
   - Task 6: 前端 - 编写组件测试
   - Task 7: 测试 - 编写集成测试
   ```

2. **标注依赖关系**
   ```python
   # 使用 addBlockedBy 设置依赖
   TaskUpdate(taskId="5", addBlockedBy=["1", "2"])  # 前端实现依赖后端 Schema
   TaskUpdate(taskId="7", addBlockedBy=["3", "6"])  # 集成测试依赖单元测试
   ```

3. **识别并行任务簇**
   - 后端 Schema 定义 → 先行
   - 前端类型定义 → 等待 Schema 完成后
   - 后端业务逻辑 + 前端 UI 实现 → 可并行

**输出**: TaskList（带依赖关系）

---

### 【阶段 3】并行开发

**目标**: 前后端并行开发，定期同步状态

#### Backend Dev 任务清单

```
1. 阅读契约表，理解接口定义
2. 实现 Pydantic Schema（src/domain/models.py 或新建）
3. 实现业务逻辑（领域层 + 应用层）
4. 实现 API 端点（interfaces/api.py）
5. 编写单元测试（TDD 优先）
6. 完成后调用 code-simplifier 优化代码
7. 标记任务完成，通知 Coordinator
```

**必须调用的技能**:
- `code-simplifier`: 代码完成后优化
- `systematic-debugging`: 遇到 Bug 时调试

#### Frontend Dev 任务清单

```
1. 阅读契约表，理解接口定义
2. 定义 TypeScript 类型（对照契约表）
3. 实现 UI 组件（契约驱动）
4. 调用 ui-ux-pro-max 设计样式（如需要）
5. 编写组件测试
6. 完成后调用 code-simplifier 优化代码
7. 标记任务完成，通知 Coordinator
```

**必须调用的技能**:
- `ui-ux-pro-max`: UI 设计时
- `code-simplifier`: 代码完成后优化

#### QA Tester 任务清单

```
1. 阅读契约表，理解测试范围
2. 准备测试数据
3. 编写集成测试用例
4. 等待前后端完成后执行测试
5. 调用 webapp-testing（E2E 测试如需要）
```

**必须调用的技能**:
- `webapp-testing`: E2E 测试时
- `systematic-debugging`: 测试失败时分析

---

### 【阶段 4】审查与验证

**目标**: 确保代码质量，契约对齐

**Coordinator 执行步骤**:

1. 通知 Reviewer 介入
2. Reviewer 对照契约表检查：
   - [ ] API 字段命名与契约表一致
   - [ ] 必填/可选字段对齐
   - [ ] 错误响应格式统一
   - [ ] 前端类型定义与契约表一致
   - [ ] 代码质量（复杂度、可读性）

3. **审查问题处理**:
   - 小问题 → 直接通知对应角色修复
   - 大问题 → 创建修复任务，重新进入审查循环

4. **审查通过** → 进入测试阶段

**Reviewer 输出**: 审查报告（包含问题列表和修复状态）

---

### 【阶段 5】测试执行

**目标**: 验证功能完整性，确保无回归

**QA Tester 执行步骤**:

1. **运行单元测试**
   ```bash
   pytest tests/unit/ -v --cov=src
   ```

2. **运行集成测试**（如有）
   ```bash
   pytest tests/integration/ -v
   ```

3. **运行 E2E 测试**（如有）
   ```bash
   # 使用 webapp-testing 技能
   ```

4. **生成测试报告**:
   - 通过率
   - 覆盖率
   - 失败用例分析

5. **测试结果处理**:
   - 全部通过 → 进入提交阶段
   - 有失败 → 创建修复任务，返回对应角色

---

### 【阶段 6】提交与汇报

**目标**: 生成 Git 提交和交付报告

**Coordinator 执行步骤**:

1. **生成 Git 提交**
   ```bash
   git add <修改的文件>
   git commit -m "$(cat <<'EOF'
   feat: <功能描述>

   - 后端：实现 API 端点和业务逻辑
   - 前端：实现 UI 组件和类型定义
   - 测试：编写单元测试和集成测试

   契约文件：docs/designs/<task>-contract.md
   审查报告：docs/reviews/<task>-review.md
   EOF
   )"
   ```

2. **生成交付报告**

```markdown
## 任务完成汇总

### ✅ 已完成任务
| 角色 | 任务 | 状态 |
|------|------|------|
| 后端 | 实现 API 接口 | ✅ |
| 前端 | 实现 UI 组件 | ✅ |
| 测试 | 编写测试用例 | ✅ |

### 📦 交付物
- 后端代码：`src/...`
- 前端代码：`web-front/src/...`
- 测试代码：`tests/...`

### ✅ 验证结果
- 单元测试：通过 (X/X)
- 集成测试：通过 (Y/Y)
- 覆盖率：Z%

### 🔗 契约对齐情况
- 字段命名：一致 ✅
- 类型定义：一致 ✅
- 错误处理：一致 ✅

### 📝 相关文档
- 契约表：docs/designs/<task>-contract.md
- 审查报告：docs/reviews/<task>-review.md
```

3. **通知用户验收**
   - 输出交付报告
   - 等待用户确认或修改意见

## 异常处理机制

### 问题分级

| 级别 | 类型 | 示例 | 处理方式 |
|------|------|------|----------|
| **L1 简单** | 语法错误、小 Bug | 拼写错误、类型错误 | 角色自行修复 |
| **L2 中等** | 接口理解偏差 | 字段类型不匹配 | Coordinator 协调对齐 |
| **L3 严重** | 技术阻塞、需求变更 | 依赖缺失、架构冲突 | 标记 blocked，最后汇报 |

### 异常处理流程

```
发现问题
    │
    ▼
角色尝试自解 (L1)
    │
    ├─ 成功 → 继续工作
    │
    └─ 失败 → 通知 Coordinator
              │
              ▼
        Coordinator 分析
              │
        ├─────┼─────┤
        ▼     ▼     ▼
       L1    L2    L3
        │     │     │
        │     │     └─→ 标记 blocked，继续其他任务
        │     │
        │     └─→ 协调对齐，更新契约表
        │
        └─→ 角色自行修复
```

### 阻塞任务恢复

当 L3 问题解决后：
1. Coordinator 解除 blocked 状态
2. 重新分配任务
3. 继续流水线
