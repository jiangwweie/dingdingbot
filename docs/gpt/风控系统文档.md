# 风控系统文档

> **文档类型**: 全局系统体检核心材料
> **生成日期**: 2026-04-29
> **数据来源**: 代码静态分析（未运行测试）
> **结论摘要**:

1. 单笔风险计算独立、完整，使用 `available_balance * max_loss_percent / stop_distance` 公式，三重约束（风险/敞口/杠杆）取 min。
2. 最关键的缺陷：**日内限制（daily_max_loss、daily_max_trades）从未真正生效** —— `record_trade()` 和 `reset_if_new_day()` 无任何外部调用点，daily stats 形同虚设。
3. 风控与策略已实现架构分离，但信号计算侧（SignalPipeline）无独立风控拒绝机制，风控仅在下单执行前拦截（CapitalProtection）。
4. 敞口用 `size * entry_price` 名义价值口径，基于 `total_balance`，多 symbol 统一收口。
5. 杠杆限制在 position sizing 中生效，但无强平距离检查、无保证金安全检查、无仓位持有时间限制实现。
6. 当前风控达到"Sim-1 可用"程度，但日内熔断和回撤暂停均为死代码，实盘前必须修复。

## 1. 单笔风险

### 1.1 Position Size 核心公式

**文件**: `src/domain/risk_calculator.py` → `RiskCalculator.calculate_position_size()`

```
核心公式:
Position_Size = Risk_Amount / Stop_Distance

其中:
- Risk_Amount = available_balance * max_loss_percent
- Stop_Distance = |entry_price - stop_loss| （绝对价差，非百分比）
```

**基数选择**: 使用 `account.available_balance`（可用余额），而非 `total_balance`（总权益）。代码注释明确标注 "Scheme B"。

**三重独立约束**（第 166-191 行，取 min）:

| 约束层 | 公式 | 含义 |
|--------|------|------|
| Layer 1: 风险约束 | `available_balance * max_loss_percent / stop_distance` | 单笔最大可亏金额 / 止损距离 |
| Layer 2: 敞口约束 | `(total_balance * max_total_exposure - total_position_value) / entry_price` | 剩余可用敞口 / 入场价 |
| Layer 3: 杠杆约束 | `available_balance * max_leverage / entry_price` | 最大杠杆对应的仓位 / 入场价 |

最终 `position_size = min(Layer1, Layer2, Layer3)`。

### 1.2 止损计算

**文件**: `src/domain/risk_calculator.py` → `RiskCalculator.calculate_stop_loss()`

- LONG: `stop_loss = kline.low`（K 线最低价）
- SHORT: `stop_loss = kline.high`（K 线最高价）

无缓冲距离，止损即 K 线极值。

### 1.3 杠杆计算

**文件**: `src/domain/risk_calculator.py` 第 213-220 行

```python
leverage_required = position_value / available_balance
leverage_to_use = min(int(leverage_required) + 1, max_leverage)
```

杠杆是试算结果，非独立输入。先算仓位大小，再反推杠杆需求，最后 cap 到 `max_leverage`。

### 1.4 边界处理

- `stop_distance == 0` → 抛出 `DataQualityWarning("W-001")`
- `total_balance <= 0` → 返回 `(0, 1)`
- `available_balance <= 0` → 返回 `(0, 1)`
- `position_size <= 0` → 返回 `(0, 1)`

## 2. 配置真实含义

### 2.1 `max_loss_percent`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `RiskConfig.max_loss_percent` |
| **单位** | 小数形式（0.01 = 1%） |
| **代码语义** | 单笔最大亏损占 `available_balance` 的比例 |
| **Runtime 生效** | 是。通过 `RiskRuntimeConfig.to_risk_config()` 传递到 `RiskCalculator` |
| **作用范围** | 信号计算阶段，决定 position_size 上限 |
| **风险点评** | 硬限制，三层约束中最基础的一层。实际执行前还有 `CapitalProtection._check_single_trade_loss()` 二次校验（使用 `balance * max_loss_percent / 100`）。两层都生效，但口径略有差异：RiskCalculator 用 `available_balance`，CapitalProtection 用 `balance`（通过 `_get_balance()` 获取） |

### 2.2 `max_total_exposure`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `RiskConfig.max_total_exposure` |
| **单位** | 倍数（0.8 = 80% of total_balance，2.5 = 250%） |
| **代码语义** | 所有持仓名义价值之和占 `total_balance` 的上限 |
| **Runtime 生效** | 是 |
| **作用范围** | 信号计算阶段，三层约束中的 Layer 2 |
| **口径** | `sum(pos.size * pos.entry_price) / total_balance` — 名义价值口径 |
| **风险点评** | 有效硬限制。当所有持仓名义价值达到 `total_balance * max_total_exposure` 时，新仓位 size 归零。默认 0.8 意味着无杠杆时最多 80% 仓位 |

### 2.3 `max_position_percent`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `CapitalProtectionConfig.single_trade["max_position_percent"]` |
| **单位** | 百分比（20 = 20%） |
| **代码语义** | 单笔订单名义价值（amount * price）不得超过 balance 的此比例 |
| **Runtime 生效** | 仅在 `CapitalProtection.pre_order_check()` 中检查，执行阶段 |
| **作用范围** | 执行前拦截，与 RiskCalculator 的三层约束独立 |
| **风险点评** | 与 `max_total_exposure` 口径不同：`max_position_percent` 基于 `balance`（可用余额），`max_total_exposure` 基于 `total_balance`。两者不冲突但有重叠。默认 20% 是合理的单笔上限 |

### 2.4 `max_leverage`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `RiskConfig.max_leverage` |
| **单位** | 整数倍（10 = 10x） |
| **代码语义** | 硬限制，三层约束中的 Layer 3，同时限制杠杆试算结果 |
| **Runtime 生效** | 是 |
| **作用范围** | 信号计算 + 执行前检查（`CapitalProtection.account["max_leverage"]`） |
| **风险点评** | 硬限制，但仅在 position sizing 中通过 `min()` 生效。不检查强平距离，不检查保证金充足率 |

### 2.5 `daily_max_loss`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `RiskConfig.daily_max_loss` |
| **单位** | USDT 绝对金额（可选） |
| **代码语义** | 每日已实现亏损的上限 |
| **Runtime 生效** | **否** — 仅在 `CapitalProtection._check_daily_loss()` 中读取 `self._daily_stats.realized_pnl`，但 `record_trade()` 从未被调用，`_daily_stats` 永远为零 |
| **作用范围** | 执行前拦截（死代码） |
| **风险点评** | **未生效**。`record_trade()` 无外部调用点，daily stats 从不更新。即使配置了此值，检查永远通过 |

### 2.6 `daily_max_trades`

| 属性 | 值 |
|------|-----|
| **定义位置** | `src/domain/models.py` → `RiskConfig.daily_max_trades` |
| **单位** | 次数（整数） |
| **代码语义** | 每日交易次数上限 |
| **Runtime 生效** | **否** — 同上，`record_trade()` 从未调用，`_daily_stats.trade_count` 永远为零 |
| **作用范围** | 执行前拦截（死代码） |
| **风险点评** | **未生效**。即使配置了此值，检查永远通过 |

### 2.7 其他配置

| 配置 | 代码语义 | 状态 |
|------|----------|------|
| `max_position_hold_time` | 最大持仓时间（分钟） | **未实现**。`RiskConfig` 中有字段定义，但无任何代码读取并执行 |
| `min_balance` (CapitalProtection) | 最低余额保留（100 USDT） | 有效。`_check_min_balance()` 在执行前检查 |

## 3. 风险聚合

### 3.1 敞口口径

**文件**: `src/domain/risk_calculator.py` 第 135-144 行

```python
total_position_value = sum(pos.size * pos.entry_price for pos in account.positions)
current_exposure_ratio = total_position_value / account.total_balance
```

- **口径**: 名义价值（notional value = size * entry_price）
- **基数**: `total_balance`（总权益，非可用余额）
- **包含**: 所有持仓（多 symbol、多方向）统一聚合

### 3.2 多 symbol 同向

- 是。`account.positions` 包含所有 symbol 的持仓，统一加总
- 无方向区分（LONG/SHORT 名义价值都加总）
- 这意味着多空对冲不被抵消，全仓名义价值受统一上限

### 3.3 多 strategy / 多 symbol 并发

- 风控统一收口于 `RiskCalculator.calculate_position_size()`
- 每次信号计算时，读取当前 `AccountSnapshot.positions` 快照
- 快照由 `SignalPipeline.update_account_snapshot()` 更新
- 并发信号串行化：`_runner_lock`（asyncio.Lock）保护策略执行
- **但**: 两个信号在同一 K 线周期内串行执行时，后一个信号看到的 `positions` 快照可能未包含前一个信号刚创建的仓位（取决于 AccountSnapshot 更新频率）

### 3.4 组合层风险

**不存在**。当前系统只有单笔交易级风控，无组合层风控：
- 无最大持仓数量限制
- 无最大亏损金额上限（daily_max_loss 未生效）
- 无最大回撤暂停机制
- 无 correlation 风险检查
- 无 sector/market risk 聚合

## 4. 日内限制与状态机

### 4.1 日内亏损限制

**实现位置**: `src/application/capital_protection.py` → `_check_daily_loss()`

```python
# 优先使用显式金额上限；否则回退到百分比口径
max_loss_amount = self._config.daily.get("max_loss_amount")
if max_loss_amount is not None:
    max_daily_loss = Decimal(str(max_loss_amount))
else:
    max_daily_loss = balance * (max_loss_percent / Decimal("100"))
passed = daily_pnl >= -max_daily_loss
```

**实际生效状态**: **未生效**。`record_trade()` 无外部调用点（全文搜索确认），`_daily_stats.realized_pnl` 永远为 `Decimal("0")`。

### 4.2 每日交易次数限制

**实现位置**: `src/application/capital_protection.py` → `_check_daily_trade_count()`

```python
trade_count = self._daily_stats.trade_count
max_trade_count = self._config.daily["max_trade_count"]
passed = trade_count < max_trade_count
```

**实际生效状态**: **未生效**。同上原因。

### 4.3 连续亏损后自动降仓 / 熔断 / 暂停

**不存在**。代码中：
- 无连续亏损检测逻辑
- 无自动降仓机制
- 无基于 PnL 的暂停开关
- `circuit_breaker` 仅在执行失败时触发（execution error），不是基于亏损的风控熔断

### 4.4 Drawdown State

**不存在**。`strategy_optimizer.py` 中有 `calculate_max_drawdown()` 用于回测后分析，但不影响实盘执行。无运行时 drawdown 状态机。

### 4.5 重置逻辑

`reset_if_new_day()` 方法存在但未被调用。即使每日统计在某些边缘场景被更新，也不会自动重置。

## 5. 杠杆与安全距离

### 5.1 `max_leverage` 是硬限制还是建议

**硬限制**。在两个层面生效：

1. **RiskCalculator Layer 3**: `leverage_based_position_size = available_balance * max_leverage / entry_price`
2. **RiskCalculator 杠杆试算**: `leverage_to_use = min(..., max_leverage)`
3. **CapitalProtection**: `account["max_leverage"]` 虽然存储了值，但 `pre_order_check()` 中没有显式杠杆检查（杠杆限制已通过 position sizing 间接保证）

### 5.2 强平距离安全阈值

**不存在**。代码中：
- `liquidation_price` 字段在 `PositionInfo` 和 `PositionInfoV3` 模型中有定义
- 但在 `RiskCalculator` 和 `CapitalProtection` 中从未使用
- 无任何代码检查 "止损价是否离强平价太近"

### 5.3 是否检查 stop 太近

**部分检查**。`stop_distance == 0` 时抛出 `DataQualityWarning`。但 `stop_distance` 过小（如 0.001）不会被拦截 —— 只会产生一个极大的 position_size，然后被杠杆约束 cap 住。

### 5.4 保证金充足

**不检查**。`CapitalProtection._get_balance()` 获取的是 `account_service.get_balance()` 返回的余额，但未检查维持保证金率、保证金占用比例等。

## 6. 策略与风控分离度

### 6.1 架构分离

风控逻辑已独立于具体策略：

| 层 | 文件 | 职责 |
|----|------|------|
| 策略层 | `src/domain/strategies/pinbar.py`, `engulfing.py` | 仅输出 pattern + direction + score |
| 风险计算层 | `src/domain/risk_calculator.py` | 读取 RiskConfig + AccountSnapshot，计算 position_size / stop / leverage |
| 执行风控层 | `src/application/capital_protection.py` | 下单前二次校验（单笔损失/仓位/日内/余额） |
| 编排层 | `src/application/execution_orchestrator.py` | 调用 CapitalProtection + 熔断检查 |

策略不持有任何风控参数，不计算仓位大小，不设置止损。风控参数通过 `RiskConfig` 从配置注入。

### 6.2 仍存在的耦合

- **止损来源**: `calculate_stop_loss()` 硬编码为 K 线极值（Pinbar low/high），与策略形态隐含耦合。如果策略输出的不是 Pinbar 形态（如 Engulfing），止损仍取当前 K 线的 high/low，这在逻辑上是正确的但缺乏通用性验证
- **信号去重**: `cooldown_seconds` 和 score covering 逻辑在 `SignalPipeline` 中，与风控无关但影响交易频率
- **执行策略构建**: `_build_execution_strategy()` 从 SignalResult 的 `take_profit_levels` 派生 OrderStrategy，这部分与策略输出有耦合

### 6.3 整体判断

当前系统更接近 **"统一风控控制层"**：策略只负责发现机会，风控独立计算仓位和止损，执行前独立拦截。但控制层不完整（日内限制失效、无组合层风险）。

## 7. 系统性结论

### 7.1 真正有效的硬限制

| 限制 | 实现状态 | 有效性 |
|------|----------|--------|
| 单笔风险上限 (`max_loss_percent`) | RiskCalculator + CapitalProtection 双重检查 | **有效** |
| 总敞口上限 (`max_total_exposure`) | RiskCalculator Layer 2 | **有效** |
| 单笔仓位上限 (`max_position_percent`) | CapitalProtection | **有效** |
| 杠杆上限 (`max_leverage`) | RiskCalculator Layer 3 + 杠杆试算 | **有效** |
| 最低余额保留 (`min_balance`) | CapitalProtection | **有效** |
| 最小订单金额 (`min_notional`) | CapitalProtection | **有效** |
| 价格偏差检查 | CapitalProtection | **有效** |
| 数量精度检查 | CapitalProtection | **有效** |
| 熔断机制（执行失败） | ExecutionOrchestrator | **有效**（但仅针对执行异常，非风险熔断） |

### 7.2 未生效 / 死代码

| 限制 | 实现状态 | 风险 |
|------|----------|------|
| 每日最大亏损 (`daily_max_loss`) | `record_trade()` 未调用 | **P0 缺陷** |
| 每日最大交易次数 (`daily_max_trades`) | `record_trade()` 未调用 | **P0 缺陷** |
| 每日统计重置 (`reset_if_new_day()`) | 未调用 | **P0 缺陷** |
| 最大持仓时间 (`max_position_hold_time`) | 仅有字段定义，无实现 | **P1 缺陷** |

### 7.3 缺失的能力

| 能力 | 现状 | 建议优先级 |
|------|------|-----------|
| 强平距离检查 | 无 | P1 |
| 保证金充足率检查 | 无 | P1 |
| 连续亏损降仓 | 无 | P2 |
| 运行时 drawdown 暂停 | 无 | P2 |
| 组合层风险（最大持仓数量、最大亏损金额） | 无 | P2 |
| 日内限制状态机 | 无 | P1 |

### 7.4 可用性判断

**当前风控达到"Sim-1 小资金观察"程度**：
- 单笔风险控制完整且有效
- 杠杆上限有效
- 敞口聚合有效（多 symbol 统一收口）
- 执行前拦截链路完整（CapitalProtection 5 项检查）

**但不适用于实盘**：
- 日内熔断完全失效（record_trade 未调用是关键缺陷）
- 无强平/保证金安全检查
- 无最大持仓时间限制
- 无运行时回撤暂停

### 7.5 代码与文档差异

| 差异点 | 文档/注释说法 | 代码实际 |
|--------|-------------|----------|
| CapitalProtection 注释 | "单笔最大损失 2% of balance" | RiskConfig 默认 0.01（1%），但 CapitalProtectionConfig 默认 2%。Runtime 模式下会覆盖为 RiskConfig 的值 |
| 主流程 fallback | `max_loss_percent` + `max_leverage` | 跳过了 `max_total_exposure`（`main.py` 第 518-521 行），fallback 模式下敞口约束缺失 |
| daily_max_loss | "每日最大回撤 5% of balance" | 实际从未生效 |
| RiskConfig.max_loss_percent | 描述为 "% of balance" | 实际代码中是小数形式（0.01 = 1%），`CapitalProtection` 内部转换为百分比（乘 100） |
