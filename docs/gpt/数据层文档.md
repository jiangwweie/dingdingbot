## 二、每类数据详细说明

### 2.1 OHLCV K-line — 核心数据源（完整）

**数据来源**:
- REST: `ExchangeGateway.fetch_historical_ohlcv()` (启动时预热, limit=100~3000, 支持分页)
- WebSocket: `ExchangeGateway.subscribe_ohlcv()` (实时推送, 仅 `is_closed=True` 触发回调)
- 本地持久化: `HistoricalDataRepository` → SQLite `klines` 表 (INSERT OR IGNORE 幂等写入)

**数据模型** (`domain/models.py:KlineData`):
```python
class KlineData(BaseModel):
    symbol: str           # "BTC/USDT:USDT"
    timeframe: str        # "15m", "1h", "4h", "1d", "1w"
    timestamp: int        # 毫秒时间戳
    open/high/low/close: Decimal
    volume: Decimal       # 基础资产成交量
    is_closed: bool
    info: Optional[Dict]  # 交易所原始数据
```

**是否进入主流程**: ✅ 是。这是系统唯一实时订阅的市场数据流。

**被哪些模块使用**:
- `SignalPipeline.process_kline()` → 策略引擎 → 过滤器链 → 风控 → 通知
- `Backtester.run_v3_pms_backtest()` → 撮合引擎 → 仓位管理
- `RiskManager.check_trailing_exit()` → 水位线更新
- `FilterFactory` 下所有 filter: EMA / MTF / ATR / Donchian

**主要用途**: Strategy (形态检测 + 过滤器) / Risk (ATR 止损 / 水位线) / Execution (撮合) / Research (回测)

### 2.2 Mark Price — 被动获取，不参与决策

**数据来源**: REST `fetch_positions()` 中 `pos['markPrice']` 字段被动获取

class PositionInfo(BaseModel):

**是否进入主流程**: ⚠️ 仅在 PositionInfoV3 API 响应中展示，不参与策略/风控/执行决策。

**被哪些模块使用**:

- `PositionInfoV3` API 响应模型 (展示字段)

**主要用途**: Observability (前端展示持仓标记价格)

**代码位置**: `exchange_gateway.py:867` (REST 被动获取), `readmodels/runtime_positions.py:76` (展示读取)

### 2.4 Funding Rate — 回测用固定值，实盘无获取

**数据来源**:
- 回测: 配置固定值 `Decimal('0.0001')` (0.01%/8h)，来源于 `config_entries_v2` KV 表
- 实盘: **无任何 fetch_funding_rate / fetch_funding_history 调用**

**当前实现** (`backtester.py:1559-1563`):
```python
funding_rate = (
    kv_configs.get('funding_rate')
    if kv_configs and kv_configs.get('funding_rate') is not None
    else Decimal('0.0001')
)  # 默认 0.01% 每 8 小时
```

**是否进入主流程**: ⚠️ 仅在回测中以固定值模拟资金费用扣除

**被哪些模块使用**:
- `Backtester._calculate_funding_cost()` — 按 8h 周期计算资金费用
- `Position.total_funding_paid` — 累计资金费用追踪

**主要用途**: Research (回测成本模型)

**关键限制**: 使用固定费率而非历史实际费率，回测结果在 funding cost 维度上不精确。

### 2.9 Order Book / Spread — 完全没有

**数据来源**: 无实时 order book 订阅。`fetch_ticker()` 仅获取 `last` 价格用于市价单预估。

**数据模型** (`exchange_gateway.py:1251`):

price = ticker.get('last') or ticker.get('close') or ticker.get('bid') or ticker.get('ask')

**是否进入主流程**: ⚠️ ticker price 仅用于 `CapitalProtectionManager` 的价格偏差检查，不用于策略或执行决策

**主要用途**: Execution (下单前价格偏差检查)

**缺失影响**: 无法做 market microstructure 研究，无法评估滑点影响

## 四、关键点特别说明

### 4.1 回测使用的数据与缺失

**当前回测使用的数据**:
- OHLCV K-lines (多时间框架: 15m/1h/4h/1d/1w)
- 固定费率模型: slippage_rate=0.1%, fee_rate=0.04%, funding_rate=0.01%/8h
- 交易所元数据: tick_size/step_size (用于精度校验)

**回测缺失的关键数据**:
1. **实际 funding rate 历史** — 使用固定 0.01%/8h，在极端行情下严重失真
2. **实际滑点分布** — 使用固定比率，不反映真实市场深度
3. **实际成交量分布** — 仅用 volume 判断 volume_surge（且该 filter 是 PlaceholderFilter）
4. **Order book depth** — 无法模拟大单冲击
5. **Funding rate regime** — 无法区分 contango/backwardation 市场状态

### 4.2 实盘执行使用的市场数据

实盘执行仅使用以下市场数据辅助决策/保护:
1. **OHLCV K-lines** — 策略信号生成的唯一输入
2. **Account Balance** — 风控计算 (仓位大小, 每日限额)
3. **Position Info** — 持仓状态 (entry_price, mark_price, unrealized_pnl)
4. **Ticker Price** — 下单前价格偏差检查 (CapitalProtection)
5. **exchangeInfo** — 精度检查 (tick_size, step_size, min_notional)

**不使用的市场数据**: order book depth, funding rate, OI, liquidation, taker volume

### 4.3 实盘成交、滑点、订单状态轨迹记录

**已记录**:
- ✅ 订单全生命周期: `PGOrderORM` (CREATED → SUBMITTED → OPEN → PARTIALLY_FILLED → FILLED/CANCELED)
- ✅ 成交均价: `average_exec_price`
- ✅ 成交时间: `filled_at`
- ✅ 出场原因: `exit_reason` (INITIAL_SL / BREAKEVEN_STOP / TRAILING_PROFIT)
- ✅ 净 PnL: `close_pnl` = gross_pnl - fee
- ✅ 手续费: `close_fee`
- ✅ 订单审计日志: `OrderAuditLog` (全状态变更追溯)
- ✅ Position Close Events: `position_close_events` 表 (每次 TP/SL 平仓事件)

**未记录**:
- ❌ 实盘实际滑点 (signal_price vs actual_exec_price 差异)
- ❌ 订单从下单到成交的延迟
- ❌ 实际执行时的 market depth snapshot

### 4.4 是否有足够数据支持 funding/regime/derivative-specific 研究

**结论: 严重不足**

| 研究类型 | 所需数据 | 当前状态 |
|---------|---------|---------|
| Funding rate arb | 实际 funding rate 历史 + mark/index price | ❌ 完全没有 |
| Regime detection | OI + funding + volume profile | ❌ 仅有 OHLCV |
| Volatility regime | 实际 IV vs RV (期权数据) | ❌ 完全没有 |
| Crowd sentiment | Long/Short ratio + funding rate | ❌ 完全没有 |
| Liquidation cascade | Liquidation data + OI | ❌ 完全没有 |
| Order flow analysis | Taker buy/sell volume | ❌ 完全没有 |
| Market microstructure | Order book + trade tape | ❌ 完全没有 |

**唯一的"regime"研究**: `run_market_regime_experiments.py` 使用 OHLCV 计算 daily EMA250 slope 作为 regime gate，这是纯价格形态方法，不涉及衍生品数据。

### 4.5 "理论上有接口，但实际没有进入策略/风控/回测"的数据

| 数据 | 接口状态 | 实际使用状态 |
|------|---------|-------------|
| `volume` (K-line volume) | ✅ 在 KlineData 中 | ⚠️ 仅在 PlaceholderFilter `volume_surge` 中声明，但该 filter 是占位符 (不执行任何逻辑) |
| `mark_price` | ✅ 在 PositionInfo 中 | ⚠️ 仅用于 API 展示，不参与策略/风控决策 |

**关键发现**: `ExtremeVolatilityConfig` 模型定义了完整的价格波动/成交量放大/深度失衡检测逻辑（5 分钟波动率、volume surge threshold、depth imbalance threshold），但**没有任何代码实例化或调用该配置**。这是一个完整的未激活设计。

## 五、系统层级判断

### 当前层级: **A. OHLCV 驱动的价格形态策略 + 订单执行平台**

### 判断证据

**1. 唯一实时数据源是 OHLCV**
- `main.py:623-630`: WebSocket 仅订阅 `subscribe_ohlcv()`
- `main.py:569-576`: Warmup 仅调用 `fetch_historical_ohlcv()`
- 没有任何其他 WebSocket 订阅 (如 order book, trades, funding)

**2. 策略引擎完全基于 OHLCV**
- `strategy_engine.py`: Pinbar/Engulfing 检测仅使用 open/high/low/close/volume
- `filter_factory.py`: EMA/MTF/ATR/Donchian 均从 OHLCV 计算
- 没有任何 filter 使用 funding rate / OI / volume profile 等衍生品数据

**3. 风控基于 OHLCV + 账户状态**
- `risk_manager.py`: trailing stop / breakeven / TP trailing 均基于 K-line high/low
- `capital_protection.py`: 仓位限制 / 每日限额 / 价格偏差检查
- 不使用 funding rate 做资金成本风控

**4. 回测完全基于 OHLCV**
- `backtester.py`: 输入仅为 K-line 序列
- `matching_engine.py`: 撮合仅基于 K-line high/low/close
- funding cost 使用固定值而非实际历史

**5. 缺失所有衍生品核心数据**
- 没有 funding rate 实时/历史获取
- 没有 open interest
- 没有 liquidation data
- 没有 taker buy/sell volume
- 没有 order book depth
- 没有 index price

**6. 架构设计不支持衍生品数据扩展**
- `KlineData` 模型没有 funding_rate / OI 等字段
- `FilterBase` 的 `FilterContext` 没有衍生品数据上下文
- 不存在任何衍生品数据的 Repository 或 Service

### 从 A 升级到 B 最缺的数据

如果要从 **A (OHLCV 驱动)** 升级到 **B (带部分衍生品语义的量化执行系统)**，需要:

| 优先级 | 数据 | 升级影响 |
|-------|------|---------|
| **P0** | **Funding Rate (实际历史 + 实时)** | 回测成本模型精确化 + funding regime 检测 |
| **P0** | **Taker Buy/Sell Volume** | Order flow 分析 + volume_surge filter 从占位符升级为真实实现 |
| **P1** | **Open Interest** | OI divergence 信号 + regime detection |
| **P1** | **实盘滑点记录** | 校准回测滑点模型 |
| **P2** | **Long/Short Ratio** | 散户情绪分析 |
| **P2** | **Liquidation Data** | Cascade liquidation 预警 |

如果要从 **B** 升级到 **C (真正多因子 regime system)**，还需要:
- Order book depth + L2 data
- Index price + basis calculation
- Options data (IV, greeks)
- Cross-exchange data aggregation
- Real-time funding rate streaming

## 六、"有但未真正使用"的数据小节

### 6.1 Volume (K-line volume) — 有数据，filter 是占位符

`volume_surge` filter 类型已注册在 `FilterFactory` 中，但实现为 `PlaceholderFilter`，执行时直接返回 `passed=True`：

```python
# filter_factory.py:952
"volume_surge": lambda **kw: PlaceholderFilter("volume_surge", **kw),
```

### 6.2 Mark Price — 有数据，不参与决策
