# 实盘 / 模拟盘日志样本文档

> **产出日期**: 2026-04-29
> **目的**: 全局系统体检用日志样本与日志结构分析
> **数据来源**: 仓库源码日志语句 + 真实运行日志文件 + 通知格式模板
> **运行状态**: 有真实日志（logs/dingdingbot.log, logs/dingdingbot.log.2026-04-24.log, backend.log），但日志内容主要为回测和启动阶段，**未捕获到实盘信号触发 / 拒绝 / 执行的完整生命周期日志**。文中明确标注了"真实样本"和"推断样本"。

## 一、结论摘要

### 1.1 日志体系成熟度判断

| 维度 | 评级 | 说明 |
|------|------|------|
| **日志基础设施** | **B+** | 统一 logger + 脱敏 + 按日轮转 + 30天保留 + 压缩清理，机制完善 |
| **启动阶段日志** | **A** | Phase 1~9 全覆盖，真实样本可还原完整启动链路 |
| **信号检测日志** | **B-** | strategy_engine 仅有 debug 级别日志（波幅过滤），filter rejection 通过 signal_pipeline WARNING 输出，但日志中**未捕获到真实信号触发样本** |
| **风控拒绝日志** | **C+** | CapitalProtection 的 pre_order_check 有 WARNING 日志，但**实盘未出现过风控拒绝**（日志文件中无证据），且拒绝原因只到 reason_code 级别，缺少结构化 JSON 输出 |
| **执行链路日志** | **B** | ExecutionOrchestrator 有完整的状态转换日志，但**实盘未捕获到执行链路运行样本** |
| **对账日志** | **A-** | StartupReconciliationService 有完整结构化摘要，真实样本可验证 |
| **通知格式日志** | **B+** | 三套通知模板（普通信号/覆盖/反向），格式清晰，但**无真实通知发送成功/失败的生产日志样本** |
| **PnL / Equity Curve** | **D** | **完全缺失**。无累计盈亏曲线、无每日收益汇总、无净值快照日志 |
| **结构化可检索性** | **C** | 大部分日志为自由文本，缺少统一的 structured log（JSON），难以 grep/聚合分析 |

### 1.2 关键发现

1. **日志存在但缺乏生产信号样本**：日志文件中只有回测运行（0 trades）和启动对账（0 候选订单），**未捕获到任何实盘信号触发、过滤拒绝、订单执行的完整日志链**。
2. **FILTER_REJECTED 日志格式设计良好但无实盘样本**：signal_pipeline.py 中定义了 `[FILTER_REJECTED]` 标签日志，格式包含 symbol/timeframe/pattern/direction/filter/reason/metadata，但日志文件中无此内容。
3. **Rejected Trades 没有专门的 rejected log 落地**：信号被过滤标记为 `FILTERED`（存入 SignalAttempt），但不生成独立的"拒绝日志文件"或结构化报告。唯一输出是 `logger.warning` 格式的文本日志。
4. **PositionProjectionService 零日志**：该模块是仓位投影的核心，但内部**无任何 logger 调用**，是日志链路中的一个盲区。
5. **PnL 追踪仅限于 PerformanceTracker 的 WON/LOST 标记**，无每日 PnL 汇总、无曲线快照。

### 1.3 实盘前日志缺口清单

| 缺口 | 严重度 | 说明 |
|------|--------|------|
| 无结构化 signal attempt 日志 | P0 | FILTERED/SIGNAL_FIRED 的 attempt 应有 JSON 结构化日志落盘 |
| PositionProjectionService 无日志 | P0 | 仓位投影是核心状态机，必须有变更日志 |
| 无每日 PnL 汇总 | P1 | Sim-1 观察期必须有每日收益快照 |
| 无 rejection reason 结构化输出 | P1 | 被拒绝信号需要可检索的结构化记录 |
| CapitalProtection 拒绝无通知 | P1 | pre_order_check 拒绝时只写 WARNING，不发飞书通知 |
| 域层 logger 缺失 | P2 | strategy_engine / risk_calculator / filter_factory 仅用 debug 级别 |

## 二、日志来源盘点

### 2.1 日志基础设施

**来源模块**: `src/infrastructure/logger.py`
**配置**:
```
格式: [%(asctime)s] [%(levelname)s] [%(name)s] %(message)s
日期格式: %Y-%m-%d %H:%M:%S
输出: stdout (INFO+) + 文件 (DEBUG+) 按日轮转
文件路径: logs/dingdingbot.log → dingdingbot-YYYY-MM-DD.log
保留策略: 30天自动清理，7天自动 gzip 压缩
脱敏: SecretMaskingFormatter 自动脱敏注册的 secret
```

**真实样本** (来源: `/Users/jiangwei/Documents/final/logs/dingdingbot.log`):
```

```

### 2.2 日志输出目的地汇总

| 输出 | 内容 | 落地方式 |
|------|------|----------|
| **控制台** | INFO+ 全部模块 | stdout StreamHandler |
| **日志文件** | DEBUG+ 全部模块 | logs/dingdingbot.log (按日轮转) |
| **飞书/企业微信通知** | 信号通知 + 系统告警 | NotificationService → FeishuWebhook/WeComWebhook |
| **SQLite (signal_attempts)** | 信号尝试记录 | SignalRepository.save_attempt() (异步队列) |
| **PG (orders/intents/signals/recovery)** | 订单/意图/信号/恢复 | 各 PG Repository |
| **SQLite (order_audit_log)** | 订单审计事件 | OrderAuditLogger → OrderAuditLogRepository |
| **SQLite (reconciliation_reports)** | 对账报告 | ReconciliationRepository |

## 四、Rejected Trades 专项分析

### 4.1 拒绝类型全谱

系统中存在以下拒绝场景，按拒绝点从上游到下游排列：

| 拒绝点 | 模块 | 拒绝原因 | 日志级别 | 结构化程度 | 实盘样本 |
|--------|------|----------|----------|------------|----------|
| 形态未检测到 | strategy_engine | NO_PATTERN | DEBUG | 低（仅debug日志） | 有（回测） |
| 波幅过小 | strategy_engine | min_range_filter | DEBUG | 低 | 有（回测） |
| EMA 距离不足 | signal_pipeline | FILTER_REJECTED | WARNING | 中（含metadata JSON） | **无** |
| MTF 不确认 | signal_pipeline | FILTER_REJECTED | WARNING | 中 | **无** |
| Runtime方向策略 | signal_pipeline | direction_not_allowed | INFO | 低 | **无** |
| 信号去重 | signal_pipeline | cooldown | DEBUG | 低 | 有（回测） |
| position_size=0 | risk_calculator | risk_constraint | WARNING | 低 | 有（回测） |
| 风控前置检查 | capital_protection | 7种reason_code | WARNING | 中 | **无** |
| 熔断拦截 | execution_orchestrator | CIRCUIT_BREAKER | WARNING | 中 | **无** |
| 交易所拒绝 | execution_orchestrator | REJECTED | ERROR | 中 | **无** |
| 交易所取消 | execution_orchestrator | CANCELED | ERROR | 中 | **无** |

### 4.2 能否看清信号为什么被拒绝？

**结论：理论上可以，但实际无法验证**

**设计层面**：
- FILTER_REJECTED 日志包含 symbol/timeframe/pattern/direction/filter/reason/metadata，**理论上可以定位每个拒绝原因**
- CapitalProtection 有 7 种 reason_code（BELOW_MIN_NOTIONAL, PRICE_DEVIATION_TOO_HIGH, SINGLE_TRADE_LOSS_LIMIT, POSITION_LIMIT, DAILY_LOSS_LIMIT, DAILY_TRADE_COUNT_LIMIT, INSUFFICIENT_BALANCE），每种都有明确的 reason_message
- ExecutionOrchestrator 的 BLOCKED 状态包含 blocked_reason + blocked_message

**实际层面**：
- 日志文件中**没有任何实盘拒绝/过滤的样本**
- 拒绝原因是自由文本，不是结构化 JSON，grep 聚合困难
- 部分拒绝（如 NO_PATTERN, 波幅过小）仅在 DEBUG 级别输出，生产环境看不到

### 4.3 拒绝日志能否支持发现系统设计问题？

**部分支持，但有关键缺口**：

**能支持的**：
- CapitalProtection 拒绝的 reason_code 分布分析（如果记录到结构化日志）
- 单笔损失/仓位/每日亏损超限的频率统计
- 信号去重率分析（通过 filter rejection 的 pattern 分布）

**不能支持的**：
- 缺少 rejection count dashboard（需要结构化日志才能聚合）
- 缺少 rejection rate 时间序列（"最近 1 小时拒绝率突增"的检测）
- 缺少拒绝原因的自动归因（哪个 filter 最常拒绝、拒绝率随参数变化的追踪）
- **SignalAttempt 虽然有 final_result 字段，但没有定期统计输出日志**

| **RiskCalculator 日志为 DEBUG 级别** | P1 | 生产环境看不到风控决策过程 | 关键约束提升到 INFO |

### 5.4 "系统做了决策，但日志看不懂原因" 的问题

**已识别的场景**：

1. **position_size=0 的信号被静默丢弃**
   - 代码路径：risk_calculator 返回 (0, 1) → backtester 输出 `[BACKTEST_SKIP] 跳过信号 sig_xxx：position_size=0`
   - 但实盘路径（signal_pipeline → signal_executor）**不输出 position_size=0 的跳过日志**，信号直接消失
   - **严重度**: P0 — 实盘中信号可能因风控约束被静默丢弃，无法从日志中发现

2. **Runtime direction policy 过滤**
   - 代码路径：_apply_runtime_direction_policy 将不匹配方向的 SIGNAL_FIRED 转为 FILTERED
   - 有日志输出，但**仅 INFO 级别**，容易在大量日志中被忽略

3. **SignalAttempt 持久化但不分析**
   - 每个 signal attempt（含 final_result=NO_PATTERN/FILTERED/SIGNAL_FIRED）都通过异步队列写入 SQLite
   - 但**没有定期的汇总统计日志**输出，如"过去 1 小时：100 个 K 线 → 85 个 NO_PATTERN → 12 个 FILTERED → 3 个 SIGNAL_FIRED"
