## 结论摘要（10 行）

当前回测引擎定位为**形态策略粗评估器**，不是交易仿真器。它有严肃的撮合逻辑（优先级队列、same-bar 冲突可配置、Decimal 精度），但缺少几个对实盘研究关键的要素：入场同 bar 成交有轻微 look-ahead 偏差（MARKET 单用 open 成交但信号基于 close 生成）、没有模拟强平/爆仓、没有 mark-to-market equity curve（只在 bar close 更新）、不支持同 symbol 同方向仓位合并（加仓时 entry_price 被覆盖）。已严肃的部分：手续费 + 滑点 + funding rate 全链路逐笔计算；多级 TP（TP1-TP5）+ SL + Trailing Stop + Breakeven + Trailing Exit 完整状态机；same-bar 冲突可配置（pessimistic / random）。**定位：形态策略粗评估器**，适合比较不同形态策略的相对表现，不适合精确预测实盘收益率。

## 一、入场与成交时机

### 1.1 Entry 是 next-bar 还是 same-bar

**结论：next-bar，且有明确的代码防护。**

信号在 bar N 的 close 生成，但 ENTRY 订单被放入 `pending_entry_orders` 列表，直到 bar N+1 开始时才移入 `active_orders` 参与撮合。

代码路径：`src/application/backtester.py`

```python
# bar N+1 开始时（循环顶部）
if pending_entry_orders:
    active_orders.extend(pending_entry_orders)
    pending_entry_orders = []

# bar N 处理完撮合后，运行策略
runner.update_state(kline)

# 信号生成后，创建订单放入 pending_entry_orders
pending_entry_orders.extend(entry_orders)
```

注释明确写道：
> "The closed bar can generate a signal, but the resulting ENTRY order should only become active on the next bar to avoid same-bar lookahead."

**这消除了 entry 的 same-bar look-ahead 风险。**

### 1.2 不同订单类型的成交价格

| 订单类型 | 触发判定 | 执行价格 | 代码位置 |
|----------|----------|----------|----------|
| MARKET (ENTRY) | **无条件成交** | `kline.open * (1 +/- slippage_rate)` | `matching_engine.py` |
| STOP_MARKET (SL) | LONG: `k_low <= trigger_price`; SHORT: `k_high >= trigger_price` | `trigger_price * (1 -/+ slippage_rate)` | `matching_engine.py` |
| TRAILING_STOP (SL) | 同上 | 同上 | `matching_engine.py` |
| LIMIT (TP1-TP5) | LONG: `k_high >= price`; SHORT: `k_low <= price` | `price * (1 -/+ tp_slippage_rate)` | `matching_engine.py` |

**注意不一致性**：
- ENTRY MARKET 单使用 `kline.open` 成交（next-bar open），这是合理的。
- SL 使用 `trigger_price`（不是 kline.low/high），这是保守的——实际可能在更差的价格成交。
- TP 使用 `price`（不是 kline.high/low），同理。
- 滑点方向：入场买单向上滑（多付钱），止盈卖单向下滑（少收钱），符合实盘逻辑。

### 1.3 ENTRY 订单仅支持 MARKET 类型

代码中 ENTRY 订单的成交判定只处理 `OrderType.MARKET`，LIMIT/STOP 类型的 ENTRY 不会被撮合引擎处理。

## 二、同一根 K 线内冲突处理

### 2.1 TP / SL 同 bar 同时触发的判定

引擎通过 `_detect_same_bar_conflicts()` 检测冲突：对每个 signal_id，检查是否 TP 和 SL 都会被当前 K 线触发。如果都触发，标记为冲突。

### 2.2 优先级机制

通过 `_sort_orders_by_priority()` 实现两种策略：

**策略 1: `pessimistic`（默认）**
- SL 优先级 = 1（最高）
- TP 优先级 = 2
- ENTRY 优先级 = 3（最低）
- 效果：同一根 K 线 SL 优先成交，TP 被取消

**策略 2: `random`**
- 对每个冲突的 signal_id，以 `same_bar_tp_first_prob`（默认 0.5）概率决定 TP 还是 SL 优先
- 每个 signal 只抽签一次，同一 signal 内的 TP1-TP5 保持同一优先级
- 可通过 `random_seed` 复现结果

### 2.3 SL 成交后的级联取消

SL 成交后，引擎调用 `_cancel_related_orders()` 取消该 signal_id 下所有 OPEN 状态的订单（包括 TP1-TP5）。

### 2.4 Entry 与 SL/TP 同 bar 相遇

由于 ENTRY 订单是 next-bar 才进入 active_orders，所以**同 bar 不会发生 entry 与 SL/TP 冲突**。

### 2.5 Look-ahead bias 风险评估

**已防护的风险**：
- 信号生成使用 `kline_history`，在当前 kline 追加到 history 之前传入，确保 multi-candle 策略（如 Engulfing）不会看到未来数据。
- MTF 使用已收盘的高周期 K 线，有明确的 `candle_close_time <= timestamp` 检查。

**残余偏差**：
- ENTRY 使用 next-bar open 成交，信号基于 bar N close 生成——这意味着策略假设你能在 bar N+1 open 以市价成交。对大周期（4h/1d）这可能不现实。

## 三、仓位与并发

### 3.1 多仓位并发

**支持。** 每个 signal_id 对应一个独立的 Position 对象，存储在 `positions_map: Dict[str, Position]` 中。不同 symbol 或同 symbol 不同方向的仓位互不干扰。

### 3.2 同 symbol 同方向仓位叠加

**技术上允许，但实现有问题。** 代码中有加仓逻辑：

```python
# 已有仓位（加仓场景）
position.current_qty += order.requested_qty
# 简单平均：实际场景中可能需要更复杂的均价计算
position.entry_price = exec_price
```

**问题**：加仓时 `entry_price` 被直接覆盖为新成交价，而不是计算加权平均。这意味着之前的持仓均价信息丢失。

### 3.3 多 symbol 并发

回测本身是**单 symbol** 串行运行的。`run_backtest()` 每次调用只处理一个 symbol。多 symbol 需要多次调用，各自维护独立的 equity curve，**不共享权益与风险占用计算**。

### 3.4 Pyramiding / DCA / 分批建仓

**不支持分批建仓。** 每个 signal 只生成一个 ENTRY 订单。虽然代码有加仓分支，但回测主循环中不会为同一 signal 创建第二个 ENTRY 订单。

## 四、成本模型

### 4.1 手续费

**逐笔记账。** 每笔订单成交时计算：

```python
trade_value = exec_price * order.requested_qty
fee_paid = trade_value * self.fee_rate
```

默认费率 0.04%（`fee_rate=Decimal('0.0004')`）。

手续费在入场时从 `account.total_balance` 扣除，出场时从 gross_pnl 中扣除。

### 4.2 滑点

**独立配置入场滑点和止盈滑点**：

| 场景 | 默认滑点率 | 方向 |
|------|-----------|------|
| 入场（ENTRY MARKET） | 0.1% (`slippage_rate`) | LONG 向上, SHORT 向下 |
| 止损（SL） | 0.1% (`slippage_rate`) | LONG 向下, SHORT 向上 |
| 止盈（TP） | 0.05% (`tp_slippage_rate`) | LONG 向下, SHORT 向上 |

Trailing Exit 有独立的滑点率配置（`trailing_exit_slippage_rate`，默认 0.1%）。

### 4.3 Funding Rate

**支持，默认开启。** 代码路径：`_calculate_funding_cost()` 方法。

计算公式：
```
position_value = entry_price * |current_qty|
funding_events = timeframe_hours / 8
funding_cost = position_value * funding_rate * funding_events
```

关键特性：
- 使用**固定费率**（默认 0.01% 每 8 小时），不是动态费率
- LONG 支付正成本，SHORT 收取负成本（即收益）
- 每根 K 线计算一次，直接从 `account.total_balance` 扣除
- 不使用 mark price，使用 entry price 计算持仓价值（简化处理）

### 4.4 成本汇总

`PMSBacktestReport` 分别输出：
- `total_fees_paid`: 所有订单手续费总和
- `total_slippage_cost`: 入场滑点成本（MARKET 订单）
- `total_funding_cost`: 资金费用总和

## 五、成交粒度

### 5.1 Partial Fill 支持

**支持，但为简化模拟。** TP/SL 订单可以只平部分仓位（通过 `requested_qty` 控制）。代码中有防超卖保护：

```python
actual_filled = min(order.requested_qty, position.current_qty)
```

### 5.2 不支持的部分

- **不模拟部分成交**：ENTRY 订单总是全额成交（MARKET 单无条件成交）
- **不模拟 order book 深度**：不检查 liquidity 是否足够
- **不模拟 partial fill 的多次回调**：一个订单要么全额成交，要么不成交
- **TP 订单的 requested_qty 超过剩余仓位时自动截断**

## 六、权益与风控基础

### 6.1 Equity Curve

**有 mark-to-market equity curve，但更新频率为每根 K 线收盘。**

```python
# 每根 K 线末尾
unrealized_pnl = sum(
    (kline.close - pos.entry_price) * pos.current_qty  # LONG
    ...
)
true_equity = account.total_balance + unrealized_pnl
equity_curve.append((kline.timestamp, true_equity))
```

**注意**：只在 K 线收盘时更新，不会在 K 线中间更新。这意味着 max drawdown 计算可能遗漏 K 线内的极端价格。

### 6.2 Unrealized PnL vs Realized PnL

| 类型 | 计算方式 | 更新时机 |
|------|----------|----------|
| Realized PnL | `(exec_price - entry_price) * filled_qty - fee` | 每笔 TP/SL 成交时 |
| Unrealized PnL | `(kline.close - entry_price) * current_qty` | 每根 K 线收盘时 |

`account.total_balance` 只包含已实现盈亏，不包含未实现盈亏。

### 6.3 Liquidation Price / 强平 / 爆仓

**不模拟。** 代码中没有任何 liquidation price 计算、维持保证金检查、或强制平仓逻辑。`RiskConfig` 中有 `max_leverage` 参数，但只用于仓位计算时的杠杆限制，不模拟实际的保证金不足场景。

### 6.4 动态风控状态机

虽然不模拟强平，但实现了完整的动态风控状态机（`DynamicRiskManager`）：

| 功能 | 触发条件 | 效果 |
|------|----------|------|
| Breakeven | TP1 成交 | SL 移至 entry_price |
| Trailing Stop | 水位线更新 | SL 阶梯式上移/下移 |
| Trailing TP | 水位线达到激活阈值 | TP 价格阶梯式上移/下移 |
| Trailing Exit | 水位线达到激活阈值 | 价格回撤到追踪退出价时市价平仓 |

所有风控修改遵循 T+1 时序：本 K 线撮合后修改的订单价格，在下一根 K 线才生效。

## 七、已知差异与局限

### 7.1 代码与文档差异

| 差异点 | 代码实际行为 | 备注 |
|--------|-------------|------|
| `_build_account_snapshot` 中 unrealized_pnl | 硬编码为 `Decimal('0')` | 注释写"回测中无浮动盈亏概念"，但 equity curve 有 unrealized PnL |
| 加仓 entry_price | 直接覆盖为新价格 | 代码注释承认需要"更复杂的均价计算" |
| Funding rate 计算基础 | 使用 entry_price * qty | 应使用 mark_price * qty |
| `_execute_trailing_exit` | 取消所有 OPEN 订单（不限于当前 signal） | 遍历所有 active_orders |

### 7.2 系统性高估/低估因素

**高估策略表现**：
1. **强平缺失**：不会模拟爆仓，极端行情下策略可能看起来"还活着"，实际已经归零
2. **Funding rate 使用固定费率**：如果资金费率波动大，实际成本可能更高
3. **Entry 用 open 成交**：假设能在 bar open 以市价成交，实际可能有额外滑点
4. **Equity curve 只在 bar close 更新**：可能遗漏 K 线内的极端 drawdown

**低估策略表现**：
1. **SL 使用 trigger_price 成交**（不是 kline.low），实际可能更差
2. **TP 使用 price 成交**（不是 kline.high），实际可能更好也可能更差
3. **不考虑 order book 深度**：大仓位可能有更大的市场冲击

## 八、系统性结论

### 当前定位

**形态策略粗评估器**，介于"信号质量计数器"和"交易仿真器"之间。

### 已经足够严肃的部分

1. **撮合逻辑**：优先级队列 + same-bar 冲突可配置 + Decimal 精度
2. **成本模型**：手续费 + 滑点 + funding rate 全链路逐笔计算
3. **动态风控**：Breakeven + Trailing Stop + Trailing TP + Trailing Exit 完整状态机
4. **多级 TP**：TP1-TP5 分批平仓 + 防超卖保护
5. **归因分析**：四维度归因（形态质量、过滤器贡献、时间分布、策略对比）
6. **Look-ahead 防护**：next-bar entry + kline_history 传入时机 + MTF 已收盘检查

### 需要谨慎对待的部分

1. **强平/爆仓未模拟**：这是最大的风险盲区
2. **Equity curve 更新粒度**：每根 K 线一次，不是 tick 级别
3. **Funding rate 计算基础**：使用 entry_price 而非 mark_price
4. **加仓逻辑**：entry_price 覆盖而非加权平均
5. **单 symbol 隔离**：多 symbol 回测不共享权益与风险

### 使用建议

- 适合比较不同形态策略的**相对表现**
- 适合评估策略的**信号质量和基本胜率**
- 不适合精确预测实盘收益率
- 不适合评估高杠杆策略的真实风险
- 建议将回测结果作为**下界估计**（保守方向）
