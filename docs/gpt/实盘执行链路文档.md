## 结论摘要（10 行以内）

1. **执行主链路完整可用**：Signal -> ExecutionIntent -> Order -> ENTRY -> TP/SL 保护单的完整链路已实现，所有核心对象（Intent/Order/Position/RecoveryTask）均已持久化到 PostgreSQL。
2. **WebSocket 订单推送未启动**：`watch_orders` 从未在 `main.py` 中被调用，已注册的 `global_order_callback` 实际上从未被 WebSocket 触发。这是当前最大的执行层风险 -- 系统依赖 REST 同步返回 + 启动对账来更新订单状态，缺少实时成交推送。
3. **保护单（SL/TP）已实现交易所原生挂单**：SL/TP 均为交易所原生 STOP_MARKET / LIMIT 单（reduce_only=True），交易所侧天然保护，不依赖系统在线。
4. **SL 替换失败有 best-effort 恢复**：包含单次同步重试 + best-effort 恢复旧 SL + PG recovery task + 熔断 + 飞书告警，但无自动重试 worker。
5. **启动对账覆盖订单状态推进 + PG recovery task 推进**：但不做仓位全量对账，不做 TP/SL 挂载修复。
6. **Circuit breaker 持久化到 PG**：重启后从 `execution_recovery_tasks` 表重建熔断集合，阻塞有未恢复任务的 symbol 新开仓。
7. **CapProtection 检查完整**：单笔损失 / 仓位 / 日亏 / 日次数 / 最低余额 / 名义价值 / 数量精度 / 价格偏差共 8 项检查。
8. **Sim 与 Live 共用执行主干**：区别仅在 RuntimeProfile 配置（testnet flag、风控参数、策略定义），执行代码路径完全相同。
9. **PositionProjectionService 仅做本地投影**：不与交易所持仓对账，仅跟踪 realized_pnl / remaining_qty。
10. **当前阶段定位**：系统偏"Sim-ready"，距离小资金实盘需补上 WS order watch 启动和周期性对账。

## 1. Signal -> Intent -> Order

### 1.1 Signal 如何变成 Intent

**入口**: `SignalPipeline.process_kline()` -> `signal_executor(signal, strategy)` -> `ExecutionOrchestrator.execute_signal(signal, strategy)`

**文件**: `src/application/signal_pipeline.py:656-665`, `src/application/execution_orchestrator.py:636-969`

**流程**:

```
KlineData (WebSocket 收到收盘 K 线)
  -> SignalPipeline.process_kline()
    -> _run_strategy() -> DynamicStrategyRunner.run_all()
    -> 如果 attempt.final_result == "SIGNAL_FIRED"
      -> _calculate_risk() -> SignalResult（含 entry_price, stop_loss, position_size）
      -> _signal_executor(signal, strategy)
        -> ExecutionOrchestrator.execute_signal(signal, strategy)
```

在 `execute_signal()` 内部：

1. **创建 ExecutionIntent**（PENDING 状态），生成 `intent_id` 和 `signal_id`
2. **深拷贝 strategy** 到 `intent.strategy` 快照（P1 修复：确保 partial fill 回调使用创建时的策略快照）
3. **保存到内存缓存 + PG**（`_save_intent` -> `intent_repository.save()`）

**真源**: `ExecutionIntent` 的真源是 `PgExecutionIntentRepository`（`pg_execution_intents` 表），内存缓存 `_intents` 仅做快速回读。

### 1.2 Intent 如何变成 Order

在 `execute_signal()` 中，创建 Intent 后依次执行：

1. **熔断检查**：`is_symbol_blocked(symbol)` -- 如果 symbol 在 `_circuit_breaker_symbols` 集合中，Intent 直接标记为 `BLOCKED`
2. **CapitalProtection 前置检查**：`pre_order_check()` 执行 8 项检查，任一不通过 Intent 标记为 `BLOCKED`
3. **创建本地主单**：`OrderLifecycleService.create_order()` -> `OrderManager.create_order_chain()` -> 保存到 PG（`PgOrderRepository`）
4. **提交 ENTRY 到交易所**：`ExchangeGateway.place_order(order_type="market", ...)` -- 市价单
5. **回填 exchange_order_id**：`OrderLifecycleService.submit_order(order.id, exchange_order_id=...)`
6. **按交易所返回状态推进本地订单**（见下文 1.3）

### 1.3 检查发生在哪些阶段

| 阶段 | 检查项 | 失败后果 |
|------|--------|----------|
| Signal -> Intent | 熔断检查 (`is_symbol_blocked`) | Intent -> BLOCKED |
| Signal -> Intent | CapitalProtection (8 项) | Intent -> BLOCKED |
| Signal -> Intent | Intent 状态机合法性 | 静默忽略非法转换 |
| Intent -> Order | OrderManager.create_order_chain 异常 | Intent -> FAILED |
| Intent -> Order | place_order() 交易所返回失败 | Intent -> FAILED |
| Intent -> Order | 市价单无成交均价（average_exec_price is None） | Intent -> FAILED ("拒绝挂载保护单") |
| Intent -> Order | 交易所取消/拒绝 | Intent -> FAILED |

### 1.4 执行主链主对象与真源

| 对象 | 真源 | 文件 |
|------|------|------|
| ExecutionIntent | `PgExecutionIntentRepository` (PG `pg_execution_intents`) | `src/infrastructure/pg_execution_intent_repository.py` |
| Order | `PgOrderRepository` (PG `pg_orders`) | `src/infrastructure/pg_order_repository.py` |
| Position | `PgPositionRepository` (PG `pg_positions`) | `src/infrastructure/pg_position_repository.py` |
| RecoveryTask | `PgExecutionRecoveryRepository` (PG `pg_execution_recovery_tasks`) | `src/infrastructure/pg_execution_recovery_repository.py` |
| Signal (live) | `PgSignalRepository` (PG `signals` + `signal_take_profits`) | `src/infrastructure/hybrid_signal_repository.py` |
| Signal (attempts) | `SignalRepository` (SQLite) | `src/infrastructure/signal_repository.py` |
| Circuit Breaker | 内存 `_circuit_breaker_symbols`（启动时从 PG recovery tasks 重建） | `src/application/execution_orchestrator.py` |

**仓储工厂**: `src/infrastructure/core_repository_factory.py` 中 `create_runtime_order_repository()` 硬编码 PG，`create_runtime_position_repository()` 硬编码 PG，`create_runtime_signal_repository()` 返回 `HybridSignalRepository`。

## 2. 订单成交与状态推进

### 2.1 Order 如何确认成交

**当前实际路径**（两条，但第二条未激活）：

1. **REST 同步返回**（激活）：`ExchangeGateway.place_order()` 返回 `OrderPlacementResult`，在 `execute_signal()` 中根据 `placement_result.status` 同步推进本地订单状态。对于市价 ENTRY，通常直接返回 FILLED。

2. **WebSocket 推送**（**未激活**）：`ExchangeGateway.watch_orders()` 已实现，`global_order_callback` 已注册到 `_order_lifecycle_service.update_order_from_exchange`，但 `main.py` 中 **从未调用** `watch_orders()`。这意味着：
   - ENTRY 市价单的成交状态在下单时就能拿到（REST 同步），问题不大
   - **TP/SL 条件单的成交状态无法实时更新** -- 这是核心风险点

### 2.2 Partial Fill 处理

**代码位置**: `src/application/execution_orchestrator.py:1171-1585`

**机制**:

当 WebSocket（或启动对账）推进 ENTRY 到 PARTIALLY_FILLED 时，`OrderLifecycleService.update_order_from_exchange()` 触发 `_on_entry_partially_filled` 回调。

回调执行增量补挂逻辑：
1. 计算 `filled_qty_total`（ENTRY 已成交量）
2. 计算 `protected_qty_total`（已存在保护单覆盖的数量）
3. `delta_qty = filled_qty_total - protected_qty_total`
4. 只有 `delta_qty > 0` 时才补挂新 TP/SL
5. **SL 替换策略**：如果有旧 SL，先撤旧 SL（交易所 + 本地），再创建新 SL 覆盖全仓

**幂等保证**：相同 `filled_qty` 的重复回调，`delta_qty = 0`，直接返回。

### 2.3 Order Status Transition

**文件**: `src/domain/order_state_machine.py`

```
CREATED -> SUBMITTED -> OPEN -> PARTIALLY_FILLED -> FILLED
   |          |           |            |
   v          v           v            v
CANCELED   REJECTED   CANCELED     CANCELED
                         REJECTED
```

**终态**: FILLED, CANCELED, REJECTED, EXPIRED
**非终态**: CREATED, SUBMITTED, PENDING, OPEN, PARTIALLY_FILLED

**Intent 状态机**（`src/domain/execution_intent.py` + `execution_orchestrator.py:119-154`）:

```
PENDING -> BLOCKED (终态)
PENDING -> FAILED (终态)
PENDING -> SUBMITTED -> PROTECTING -> COMPLETED (终态)
                       PARTIALLY_PROTECTED -> PROTECTING
                                             FAILED
                                             COMPLETED
```

### 2.4 TP / SL 是交易所原生单还是系统托管

**结论：交易所原生单**。

- SL 使用 `stop_market` 类型，`reduce_only=True`
- TP 使用 `limit` 或 `stop_market` 类型，`reduce_only=True`
- 均通过 `ExchangeGateway.place_order()` 提交到 Binance 合约

**这意味着**：即使系统完全下线，交易所侧的 TP/SL 单仍然有效，会在价格触发时自动执行。这是系统安全性的核心保障。

## 3. 保护与恢复

### 3.1 系统断线时保护单是否仍在交易所

**是的**。TP/SL 单以 `reduce_only=True` 提交到 Binance 合约，交易所侧独立管理，不依赖系统在线。

**但是**：当前 TP/SL 机制存在以下限制：
- SL 替换失败时，旧 SL 可能已被撤销但新 SL 未成功提交（有 best-effort 恢复，见 3.2）
- 部分 TP 模式下，需要系统在线来触发级联撤单（OCO 逻辑），但当前代码中 OCO 逻辑未在主链中激活

### 3.2 SL 替换失败时的恢复机制

**文件**: `src/application/execution_orchestrator.py:510-634, 1075-1139`

当 SL 替换失败（partial fill 增量补挂场景），系统执行以下恢复链：

```
SL 提交失败
  -> _place_protection_order_with_single_retry()  // 单次同步重试
    -> 成功 -> continue
    -> 失败 -> _best_effort_restore_previous_sl()  // 尝试恢复旧 SL
      -> 成功 -> return（旧 SL 已补回，缩小裸奔窗口）
      -> 失败 -> _trigger_unprotected_recovery()
        -> 创建 PG recovery_task (type="replace_sl_failed")
        -> 熔断 symbol（circuit_breaker_symbols.add(symbol)）
        -> 飞书告警 [P0] Pending Recovery Triggered
```

**关键代码路径**:
- `_place_protection_order_with_single_retry()`: 创建新的 Order 对象，生成新的 order_id，提交到交易所
- `_best_effort_restore_previous_sl()`: 深拷贝旧 SL，生成新 order_id，尝试重新提交
- `_trigger_unprotected_recovery()`: 写入 `pg_execution_recovery_tasks` 表 + 熔断 + 告警

### 3.3 重启后如何恢复

**启动流程** (`src/main.py`):

```
Phase 4.2: 初始化执行运行时
  -> PgOrderRepository, PgExecutionIntentRepository, PgPositionRepository
  -> OrderLifecycleService.start()
  -> PgExecutionRecoveryRepository 初始化

Phase 4.3: 启动对账 (StartupReconciliationService)
  -> 扫描 SUBMITTED/OPEN/PARTIALLY_FILLED 订单
  -> 通过 REST fetch_order() 查询交易所真实状态
  -> update_order_from_exchange() 推进本地状态
  -> 清除 WS 待恢复标记

Phase 4.4: Circuit Breaker 重建
  -> 从 PG execution_recovery_tasks 读取 blocking tasks
  -> 重建 _circuit_breaker_symbols 集合
```

### 3.4 Startup Reconciliation 实际做什么，不做什么

**做** (`src/application/startup_reconciliation_service.py`):
- 扫描本地 SUBMITTED / OPEN / PARTIALLY_FILLED 状态的订单
- 对每笔通过 REST `fetch_order()` 查询交易所真实状态
- 用 `update_order_from_exchange()` 推进本地状态
- 读取 WS 失败恢复队列（`_pending_recovery_orders`）并清除标记
- 扫描 PG `execution_recovery_tasks`：已终态订单 -> mark_resolved；未终态 -> 检查重试次数 -> mark_retrying 或 mark_failed

**不做**:
- 不做定期对账（仅启动时执行一次）
- 不做 30s 超时查询
- 不做 TP/SL 挂载修复（已挂的保护单不会重新提交）
- 不做全量仓位对账
- 不做交易所侧孤儿订单清理

### 3.5 Execution Recovery Tasks 链路

**表**: `pg_execution_recovery_tasks`

**字段**: id, intent_id, related_order_id, related_exchange_order_id, symbol, recovery_type, status (pending/retrying/resolved/failed), error_message, retry_count, next_retry_at, context_payload, created_at, updated_at, resolved_at

**重试策略** (`src/application/recovery_retry_policy.py`):
- 最大重试次数: 3
- 基础延迟: 60s
- 指数退避: 60s -> 120s -> 240s（上限 900s）
- 重试判断: `retry_count < 3`

**关键分离**:
- `list_active()`: status in (pending, retrying) AND next_retry_at is null or <= now（"现在就要处理哪些"）
- `list_blocking()`: status in (pending, retrying)，不关心 next_retry_at（"哪些仍应阻止新开仓"）

**当前局限**:
- 启动对账处理 recovery tasks 但不实际执行恢复动作（不重新提交 SL）
- 无后台 worker 持续消费 recovery tasks
- recovery task 被标记为 resolved 仅因"关联订单已终态"或"达到最大重试次数"，没有自动修复逻辑

## 4. 异常、重试、限频、降级

### 4.1 Binance API Rate Limit / Timeout / Error Retry

**文件**: `src/infrastructure/exchange_gateway.py`

**CCXT 配置**:
- `enableRateLimit: True` -- CCXT 内置限频器
- `timeout: 30000` -- 30s 请求超时
- `recvWindow: 30000` -- 30s 接收窗口

**异常分类与处理**:

| 异常类型 | CCXT 异常 | 系统异常 | 处理方式 |
|---------|-----------|---------|---------|
| 保证金不足 | `ccxt.InsufficientFunds` | `InsufficientMarginError` (F-010) | **直接失败，不重试** |
| 订单参数错误 | `ccxt.InvalidOrder` | `InvalidOrderError` (F-011) | **直接失败，不重试** |
| 频率限制 | `ccxt.DDoSProtection` | `RateLimitError` (C-010) | **直接抛出，不重试** |
| 网络错误 | `ccxt.NetworkError` | `ConnectionLostError` (C-001) | **直接抛出，不重试** |
| 其他异常 | `Exception` | `OrderPlacementResult(is_success=False)` | **返回失败结果，不抛异常** |

**WebSocket 重连**:
- 最大重连次数: 10
- 初始延迟: 1s，指数退避，上限 60s
- 超限后抛出 `ConnectionLostError` (C-001)

**Asset Polling 重连**:
- 最大连续失败次数: 5
- 超限后抛出 `ConnectionLostError` (C-002)

### 4.2 重试策略总结

| 场景 | 重试? | 重试策略 |
|------|-------|---------|
| place_order 失败 | **否**（ENTRY 直接 FAILED） | -- |
| place_order SL 失败 | **是**（1 次同步重试） | `_place_protection_order_with_single_retry` |
| 旧 SL best-effort | **否**（单次尝试） | -- |
| recovery task | **是**（最多 3 次） | 指数退避 60s/120s/240s |
| WebSocket 重连 | **是**（最多 10 次） | 指数退避 1s -> 60s |
| Asset polling | **是**（最多 5 次连续失败） | 固定间隔 |

### 4.3 降级逻辑

| 场景 | 降级方式 | 影响 |
|------|---------|------|
| CCXT Pro 未安装 | WebSocket -> REST 轮询 | 延迟增加（60s 轮询），无实时推送 |
| PG execution_recovery_repository 初始化失败 | 降级到无 PG 模式 | Recovery task 不持久化，熔断不重启 |
| RuntimeConfigResolver 失败 | 降级到 ConfigManager | 策略/风控配置来源变化 |
| AccountSnapshot 不可用 | 使用 dummy (10000 USDT) | 风控计算基于虚假余额 |
| _check_quantity_precision 异常 | 跳过精度检查 | 可能提交不合规数量 |
| _check_price_reasonability 异常 | 跳过价格偏差检查 | 可能提交异常价格订单 |

### 4.4 Silent Failure 风险

1. **WebSocket order watch 未启动**（最大风险）：`global_order_callback` 已注册但从未被触发，TP/SL 的成交状态无法实时更新到本地。
2. **Position projection 异常被静默吞掉**: `execution_orchestrator.py:259-264` 中 `project_entry_fill` 失败仅 warning 日志。
3. **WS 订单回调异常被标记但无自动恢复**: `_pending_recovery_orders` 记录了失败的回调，但仅在启动对账时处理。
4. **CapitalProtection daily_stats 仅在内存中**: 重启后 `trade_count` 和 `realized_pnl` 归零，每日亏损/次数限制重置。

## 5. Kill Switch / Safety

### 5.1 是否有 Kill Switch

**有，但形式分散**:

1. **Circuit Breaker（自动）**: SL 替换失败 -> 自动熔断 symbol -> 拒绝该 symbol 所有新开仓信号
2. **CapitalProtection 检查（自动）**: 8 项下单前检查，任一不通过 -> BLOCKED
3. **Intent 状态机（自动）**: 非法状态转换被拒绝
4. **API Key 权限检查（启动期）**: 有 withdraw 权限 -> F-002 FatalStartupError
5. **没有显式的"全局紧急停止开关"**: 没有 API 端点或配置项可以一键停止所有交易

### 5.2 阻止新开仓的条件

| 条件 | 触发方式 | 范围 |
|------|---------|------|
| Symbol 熔断 | SL 替换失败自动触发 | 单 symbol |
| 单笔损失超限 | CapitalProtection pre_order_check | 全局 |
| 仓位占比超限 | CapitalProtection pre_order_check | 全局 |
| 日亏损超限 | CapitalProtection pre_order_check | 全局 |
| 日交易次数超限 | CapitalProtection pre_order_check | 全局 |
| 最低余额不足 | CapitalProtection pre_order_check | 全局 |
| 名义价值过低 | CapitalProtection pre_order_check | 全局 |
| 数量精度不合格 | CapitalProtection pre_order_check | 全局 |
| 价格偏差过大 | CapitalProtection pre_order_check | 全局 |

### 5.3 安全倾向评估

**"出错时宁可不做，也不做错"的倾向评估**:

- **信号到执行**: 有较好的防御倾向。熔断检查在 CapitalProtection 之前，失败路径都会标记 Intent 为 FAILED/BLOCKED。
- **SL 保护**: SL 提交失败有单次重试 + best-effort 恢复旧 SL + 熔断 + 告警，但无持续自动恢复。
- **Daily PnL 追踪**: 仅在内存中，重启归零 -- 这是安全漏洞。
- **仓位投影**: 仅本地投影，不与交易所对账 -- 如果投影有误，系统可能误判仓位状态。

## 6. Sim 与 Live 的差异

### 6.1 执行链是否共用主干

**是的**。Sim（自然模拟盘）和 Live 执行主干完全相同：

- `ExecutionOrchestrator.execute_signal()` 是唯一入口
- `ExchangeGateway.place_order()` 是唯一下单通道
- `OrderLifecycleService` 是唯一状态管理服务
- 区别仅在 `RuntimeConfigResolver` 解析的配置不同

### 6.2 差异点

| 维度 | Sim | Live |
|------|-----|------|
| 交易所端点 | `enable_demo_trading(True)` -> demo-fapi.binance.com | 正式端点 |
| 风控参数 | runtime profile 配置 | runtime profile 配置 |
| 策略定义 | runtime profile 配置 | runtime profile 配置 |
| testnet flag | `exchange_testnet=True` | `exchange_testnet=False` |
| 资金规模 | 模拟盘资金 | 真实资金 |

**代码路径完全相同**，无 `if sim:` / `if live:` 分支。

### 6.3 当前系统定位

**偏 "Sim-ready"**，理由：
- `watch_orders` 未启动（实盘必须有实时订单状态更新）
- 无周期性对账（实盘需要定期校验本地状态与交易所一致性）
- `daily_stats` 内存态（实盘需要持久化）
- `CapitalProtection` 的 `daily_stats` 重启归零（实盘需要 PG 持久化）
- 无显式 kill switch API

## 7. 系统性结论

### 7.1 当前执行链是否具备进入 Sim-1 / 小资金实盘前验证阶段的条件

**具备进入 Sim-1 自然模拟盘观察阶段的条件**，但不完全具备小资金实盘条件。

**已具备**:
- 完整的 Signal -> Intent -> Order -> TP/SL 保护单链路
- 所有核心对象 PG 持久化
- Circuit breaker + Recovery task 机制
- 启动对账
- CapitalProtection 8 项检查
- API Key withdraw 权限拦截

**不完全具备（需要补全才能进入小资金实盘）**:
- `watch_orders` WebSocket 未启动（最大阻塞项）
- 无周期性对账（建议至少每 15 分钟一次）
- daily_stats 内存态（重启归零）
- 无显式 kill switch API
- PositionProjection 不与交易所对账

### 7.2 最大的执行层风险

**P0: WebSocket 订单推送未启动**

`ExchangeGateway.watch_orders()` 已实现但从未在 `main.py` 中调用。已注册的 `global_order_callback`（`_order_lifecycle_service.update_order_from_exchange`）实际上从未被触发。

**影响**:
- TP/SL 条件单成交后，本地 Order 状态不会被更新
- PositionProjection 不会被触发（`_on_exit_progressed` / `_on_exit_filled` 回调不被调用）
- realized_pnl 不会被更新
- 当前系统仅在启动对账时通过 REST 查询来推进状态

**修复建议**: 在 `main.py` Phase 8 中为每个活跃 symbol 启动 `watch_orders()` 任务。

### 7.3 最大的恢复/保护层风险

**P0: Recovery Task 无自动恢复 worker**

当前 `execution_recovery_tasks` 被创建后，仅在启动对账时被处理（标记 resolved/retrying/failed），但没有后台 worker 实际执行恢复动作（如重新提交 SL）。

**影响**:
- SL 替换失败后，recovery task 被记录但不会自动重试提交 SL
- 熔断的 symbol 需要人工介入清除
- 如果系统运行期间 SL 替换失败，该 symbol 将持续被熔断直到重启

**P1: Daily stats 内存态**

`CapitalProtectionManager._daily_stats` 仅在内存中，重启后归零。实盘场景下，这可能导致：
- 重启后当日亏损/交易次数限制失效
- 用户可能在已经亏损的情况下继续开仓
