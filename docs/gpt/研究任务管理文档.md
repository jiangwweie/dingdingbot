# 研究任务管理与实验控制面（Research Control Plane）成熟度评估

> **审计日期**: 2026-04-29
> **审计范围**: research control plane 全链路（模型、存储、API、candidate 流程、防污染、可复现性）
> **审计方法**: 代码静态审计（未运行测试），以代码为准
> **当前系统阶段**: Sim-1 观察期

---

## 一、结论摘要

### 成熟度判定

**B+. 有基本任务管理的研究工作台，但尚未达到"可复现、可治理、可晋升"的完整研究系统。**

当前系统已具备：

- 独立的 Research Job / Run Result / CandidateRecord 三对象模型
- 独立 SQLite 存储（`data/research_control_plane.db`），与 runtime PG 执行链物理隔离
- 完整的 job 生命周期状态机（PENDING / RUNNING / SUCCEEDED / FAILED / CANCELED）
- Candidate 的 create -> review -> REJECTED/RECOMMENDED 流程已实现
- Optuna candidate report 有 JSON schema 契约，包含 git commit、source profile hash、reproduce command
- Review rubric（Strict v1 / Loose v1）已有代码实现（`CandidateArtifactService._evaluate_review()`）
- 详细设计文档明确禁止 "promote candidate to runtime" 和 "research 写 runtime profile"

当前缺失：

- **数据版本/时间范围未被结构化记录**：spec_snapshot 包含 `start_time_ms` / `end_time_ms`，但没有数据源 hash 或数据校验机制
- **代码版本仅记录 git commit**：不记录依赖版本，不能完全复现环境
- **无 IS / validation / OOS 方法学分层**：所有回测都是"单次历史窗口"，无 walk-forward 概念
- **Optuna study 与 Research Job 未打通**：Optuna 走 `StrategyOptimizer` + `optimization_history.db`，Research Control Plane 走 `ResearchJobService` + `research_control_plane.db`，两条链路并行但未统一
- **candidate -> promote -> runtime 流程不存在**：设计文档明确标注 "v1 不实现"，目前只有 review 标记，没有晋升执行路径
- **50 个 research 脚本中绝大多数不走 Research Control Plane**：只有 API 发起的 backtest 才进入 job 系统

### 一句话结论

**Research Control Plane 的骨架（模型 / 存储 / API / candidate review）已落地，但研究方法学（复现性、分层验证、Optuna 集成）仍处于脚本级水平。系统有能力"管理研究任务"，但还不具备"保证研究结论可复现、可安全推进到 runtime"的完整能力。**

## 二、实验唯一性与任务模型

### 2.1 唯一标识体系

系统中存在三层唯一标识，但分别服务于不同子系统：

| 层次 | 前缀格式 | 生成方式 | 持久化位置 | 覆盖范围 |
|------|----------|----------|------------|----------|
| Research Job ID | `rj_{uuid16}` | `ResearchJobService._new_id("rj")` | `research_control_plane.db` | 仅 API 发起的 backtest |
| Run Result ID | `rr_{uuid16}` | `LocalBacktestResearchRunner._new_id("rr")` | `research_control_plane.db` | 仅 API 发起的 backtest |
| Candidate Record ID | `cand_{uuid16}` | `ResearchJobService._new_id("cand")` | `research_control_plane.db` | 仅通过 API 创建的 candidate |
| Optimization Job ID | `opt_{uuid8}` | `StrategyOptimizer.start_optimization()` | 内存（`_jobs` dict） | Optuna 优化任务 |
| Backtest Report ID | 由 `BacktestReportRepository` 生成 | `BacktestReportRepository` | SQLite backtest DB | 兼容旧回测入口 |

**关键发现**：

1. **Research Job / Run Result / CandidateRecord 的唯一标识稳定且持久化**。`research_control_plane.db` 表有 PRIMARY KEY 约束，ID 通过 `ON CONFLICT DO UPDATE` 实现幂等写入。

2. **Optuna job ID 只在内存中**。`StrategyOptimizer._jobs` 是 `Dict[str, OptimizationJob]`，进程重启后丢失。优化历史通过 `OptimizationHistoryRepository` 持久化到 `data/optimization_history.db`，但 job 元数据（status、best_trial）不持久化。

3. **50 个 `scripts/run_*.py` 脚本中，只有通过 `POST /api/research/jobs/backtest` 发起的任务才有 job_id / run_result_id**。其余脚本直接构造 `BacktestRequest` 调用 `Backtester`，产出写入旧 backtest DB 或文件系统，没有统一标识。

4. **CandidateRecord 与 RunResult 通过 `run_result_id` 外键关联**，可以追溯到产生该 candidate 的完整 spec snapshot 和 metrics。

### 2.2 状态机

`ResearchJobStatus` 状态机：

```
PENDING --> RUNNING --> SUCCEEDED
PENDING --> RUNNING --> FAILED
PENDING --> CANCELED
```

状态转移由 `ResearchRepository.mark_job_running()` / `mark_job_succeeded()` / `mark_job_failed()` 实现，使用 `WHERE status = ?` 条件更新，具有基本的并发安全性（单条 SQL 原子操作）。

`CandidateStatus` 状态机：

```
DRAFT --> REVIEWED
DRAFT --> REJECTED
DRAFT --> RECOMMENDED
REVIEWED --> REJECTED
REVIEWED --> RECOMMENDED
```

Review 通过 `ResearchJobService.review_candidate()` 实现，不做状态转移合法性检查（任何状态可以跳到任何状态）。这在 v1 可接受，但长期应加状态机约束。

### 2.3 代码与文档差异

| 差异点 | 代码实际 | 文档描述 |
|--------|----------|----------|
| ResearchJob 是否包含 `spec` 内嵌 | 代码中 `ResearchJob` 包含 `spec: ResearchSpec` 内嵌字段 | 详细设计文档只提到 `spec_ref`，代码同时保留了 `spec_ref`（文件路径）和内嵌 `spec` |
| Job cancel API | 代码中不存在 cancel 端点 | 详细设计预留了 `POST /api/research/jobs/{id}/cancel`，标注返回 501 |
| PromoteDecision 模型 | 代码中不存在 | 详细设计标注 "v1 只设计，不实现" |

## 三、可复现性

### 3.1 参数保存

**已实现**：

- `ResearchSpec` 作为研究输入的完整序列化对象，包含 symbol / timeframe / time window / costs / runtime_overrides / profile_name
- `LocalBacktestResearchRunner.run()` 在执行前将 `spec_snapshot` 写入 `reports/research_runs/{job_id}/spec.json`
- `ResearchRunResult.spec_snapshot` 存储了完整的 spec + resolved_runtime_overrides + resolved_order_strategy
- `OptunaStudySpec` 定义了 Optuna 研究的统一 spec（`src/application/research_specs.py`）
- `StrategyOptimizer.write_candidate_report()` 产出的 candidate JSON 包含：best_trial params、fixed_params、runtime_overrides、constraints、resolved_request

**未实现**：

- 无 `random_seed` 默认固定（`BacktestEngineProfile.random_seed` 默认为 `None`）。当 `same_bar_policy="random"` 时，同一条 K 线上 TP/SL 的匹配结果不可复现
- Optuna study 的采样路径依赖 Optuna 内部随机种子，未在 `OptimizationRequest` 或 `OptunaStudySpec` 中暴露

### 3.2 代码版本

**已实现**：

- `research_artifacts.py` 提供 `get_git_metadata(repo_root)` 函数，返回 `GitMetadata(commit, branch, is_dirty)`
- `StrategyOptimizer.build_candidate_report()` 将 `git_meta.to_dict()` 写入 candidate JSON
- Candidate report contract 要求包含 `git` 字段（`docs/planning/optuna-candidate-report-contract.md`）

**未实现**：

- Research Job / Run Result 不记录 git commit。`ResearchJob` 和 `ResearchRunResult` 模型中没有 git metadata 字段
- 不记录 Python / pip 依赖版本
- 不记录关键库版本（ccxt / optuna / fastapi / pydantic）

### 3.3 数据版本

**已实现**：

- `ResearchSpec.start_time_ms` / `end_time_ms` 明确记录数据时间范围
- `ResearchSpec.symbol` / `timeframe` 明确记录数据品种和周期
- `BacktestProfile.config_hash()` 使用 `stable_config_hash()` 生成 profile 配置 hash

**未实现**：

- 不记录实际拉取的 K 线数据 hash（同样的 time window + symbol + timeframe，不同时间执行可能获取不同数据，因为交易所历史数据可能被修正）
- 不记录数据源（是本地缓存还是实时拉取）
- 不记录历史数据的时间精度（是否存在缺失 K 线）

### 3.4 复现能力评估

| 复现要素 | Research Control Plane (API) | Optuna Candidate Report | 50 个 scripts |
|----------|------------------------------|-------------------------|---------------|
| 参数完整保存 | 是（spec_snapshot） | 是（resolved_request + runtime_overrides） | 部分（取决于脚本实现） |
| 代码版本 | 否 | 是（git commit） | 否 |
| 数据时间范围 | 是 | 是 | 部分 |
| 数据源 hash | 否 | 否 | 否 |
| 随机种子 | 可选（None = 不确定） | 可选 | 可选 |
| reproduce command | 否 | 是（reproduce_cmd 字段） | 否 |
| Artifact 完整保存 | 是（spec.json + result.json + metrics.json） | 是（candidate JSON 文件） | 部分 |

**结论**：当前可复现性处于"有条件复现"水平。在同一代码版本、同一数据源、`same_bar_policy != "random"` 的条件下，Research Control Plane 的 API backtest 可以通过 spec_snapshot 复现。但缺少数据校验、依赖版本记录和脚本级的统一标准。

## 四、研究方法学分层

### 4.1 IS / Validation / OOS 区分

**当前系统不区分 IS / validation / OOS。**

所有回测（无论通过 API 还是脚本）都是"给定一个 time window，跑一遍回测"。`ResearchSpec` 只有 `start_time_ms` / `end_time_ms`，没有 `kind: "in_sample" | "out_of_sample" | "walk_forward"` 的概念。

从实际研究活动看（`docs/planning/2026-04-28-research-evidence-index.md`），研究者人工划分了 IS（2023-2025）和 OOS（2022、2026 Q1），但这种划分体现在脚本的硬编码时间窗口中，而非系统级的方法学约束。

### 4.2 Walk-Forward / Train-Validation-Test

**不存在。**

- 无 walk-forward 滚动窗口机制
- 无 train/validation/test 数据分段定义
- 无数据泄漏检测（lookahead check）
- Optuna study 的 `n_trials` 在同一个 time window 上做参数搜索，没有 out-of-sample 验证步骤

**唯一的反前瞻机制**：MTF Anti-Lookahead Alignment（`backtester.py:1101`）确保信号判断只使用已收盘 K 线，这是数据层面的反前瞻，不是方法学层面的样本分割。

### 4.3 当前研究链的定位

**更偏"有基本任务管理的单次历史回测工具"，而非"有实验方法学约束的研究系统"。**

证据：

1. `ResearchSpec.kind` 只有 `Literal["backtest"]`，没有 experiment / study / sweep
2. Optuna 集成只做参数搜索，不做样本外验证
3. Candidate review（Strict v1 rubric）评估的是单次回测的统计指标（sharpe >= 1.0、trades >= 100 等），不是跨样本稳定性
4. 研究证据索引（H0-H6）全部是人工设计实验、人工划分时间窗口、人工判断结论
5. 没有"实验模板"或"假设-验证-结论"的结构化流程

### 4.4 代码与文档差异

| 差异点 | 代码实际 | 设计文档 |
|--------|----------|----------|
| ResearchSpec.kind 类型 | `Literal["backtest"]`，只支持 backtest | 详细设计预留 `Literal["backtest", "optuna"]`，但代码中 optuna 未实现 |
| Optuna study 管理 | 独立的 `StrategyOptimizer`，不走 Research Job 系统 | 详细设计提到 "optuna 不在 v1 API 开放" |

## 五、Candidate 治理

### 5.1 Candidate 生命周期

当前实现的 candidate 流程：

```
RunResult (research_control_plane.db)
    |
    v
POST /api/research/candidates  --> CandidateRecord (DRAFT)
    |
    v
POST /api/research/candidates/{id}/review  --> CandidateRecord (REVIEWED/REJECTED/RECOMMENDED)
    |
    v
[终点 -- 没有 promote 动作]
```

**已实现的阶段**：

1. **Create candidate**（`ResearchJobService.create_candidate()`）：从 run_result_id 创建 CandidateRecord，初始状态 DRAFT
2. **Review candidate**（`ResearchJobService.review_candidate()`）：更新 status / review_notes / applicable_market / risks / recommendation
3. **List / Get candidate**：支持按 status 筛选和分页查询

**未实现的阶段**：

1. **Promote candidate to runtime**：设计文档明确标注 "v1 不实现自动 promote"，代码中不存在 `promote` API 或 service
2. **PromoteDecision 审计**：设计文档预留了 `PromoteDecision` 模型，代码中不存在
3. **Candidate 到 runtime profile 的写入路径**：完全不存在

### 5.2 Optuna Candidate 的独立治理链

除了 Research Control Plane 的 CandidateRecord，Optuna 有自己的一套 candidate 治理：

```
StrategyOptimizer._run_optimization()
    |
    v
write_candidate_report()  --> reports/optuna_candidates/*.json
    |
    v
CandidateArtifactService (readmodels) --> 自动评估 Strict/Loose rubric
    |
    v
[人工 review] --> 文档级别（optuna-candidate-review-rubric.md）
```

**关键差异**：

| 维度 | Research Control Plane | Optuna Candidate |
|------|------------------------|------------------|
| 存储 | SQLite（`research_control_plane.db`） | 文件系统（`reports/optuna_candidates/*.json`） |
| Review | API 端点 + CandidateRecord | `CandidateArtifactService` 只读评估 + 人工 |
| 状态管理 | `CandidateStatus` 枚举 | 文件中标注 `status: "candidate_only"` |
| 与 runtime 的关系 | 禁止 promote | 禁止 promote（`promotion_policy: "manual_review_required"`） |

这两条链路尚未统一。一个 Optuna 产出的 candidate 和一个 API backtest 产出的 candidate 使用不同的存储、不同的 review 机制和不同的状态模型。

### 5.3 Candidate 与 Runtime Profile 的隔离

**隔离是明确的，且有多层保障**：

1. **模型层**：`CandidateRecord` 和 `ResearchSpec` 都不包含 runtime profile 的写入接口
2. **存储层**：candidate 写入 SQLite（`research_control_plane.db`），runtime profile 存储在另一个 SQLite 表（`runtime_profiles`）或 PG
3. **API 层**：`/api/research/*` 端点不提供任何 runtime profile 写入能力。设计文档明确列出 "禁止接口"：`POST /api/research/candidates/{id}/promote` 不存在
4. **文档层**：`research_models.py` 模块级 docstring 声明 "These models describe research jobs and candidates only. They must not become runtime configuration or execution state models."
5. **candidate report 约束**：`optuna-candidate-report-contract.md` 硬约束 `status: "candidate_only"` 和 `promotion_policy: "manual_review_required"`

## 六、防污染设计

### 6.1 已建立的边界

**Research Control Plane 本身的边界是清晰的**：

1. `ResearchRepository` 使用独立 SQLite 文件 `data/research_control_plane.db`，不与 runtime PG 共享
2. `LocalBacktestResearchRunner` 通过 `BacktestExecutor` 回调执行回测，不直接访问 runtime 配置
3. `BacktestRequest` 的构建路径（`ResearchSpec -> BacktestJobSpec -> BacktestRequest`）是单向的，不反向写入 runtime
4. `BacktestConfigResolver` 使用独立的 `BacktestProfileProvider`，不读取 `RuntimeConfigProvider`

**配置来源优先级已文档化**（`docs/planning/2026-04-27-runtime-and-research-config-source-priority-ssot.md`）：

- Runtime 链：`ResolvedRuntimeConfig` (frozen at startup)
- Research 链：explicit request > runtime_overrides > local config_manager > profile KV > code defaults

### 6.2 仍存在的污染风险

#### 风险 1：Backtester 的 ConfigManager legacy fallback

**位置**：`src/application/backtester.py:512-518`

```python
cm = self._config_manager
if cm is None:
    # legacy_fallback
    from src.application.config_manager import ConfigManager
    cm = ConfigManager.get_instance()
```

当 `config_manager` 未注入时（例如通过 API 路径 `_execute_backtest_request` 创建的 Backtester），会 fallback 到全局 ConfigManager 单例。如果此时有研究脚本通过 `ConfigManager.set_instance()` 注入了修改后的实例，runtime 可能读到被污染的配置。

**缓解状态**：代码注释已标注为 `legacy_fallback`，有 `logger.info` 告警。但不是架构级隔离。

#### 风险 2：50 个研究脚本的旁路

**位置**：`scripts/run_*.py`（50 个文件）

绝大多数研究脚本直接构造 `BacktestRequest` 并调用 `Backtester`，不经过 Research Control Plane。这些脚本可能：

- 使用 `ConfigManager.set_instance()` 注入自定义配置（审计发现 8 个脚本有此行为）
- 直接修改 `pipeline._kline_history` 等私有属性
- 使用 `allow_readonly_update=True` 绕过 profile 只读保护

虽然这些脚本主要在开发/研究环境中运行，但如果在 runtime 进程的同一 Python 环境中执行，可能污染全局状态。

**缓解状态**：2026-04-26 研究链审计提出了方案 A（最小收口），建议移除 `allow_readonly_update=True`、脚本去掉 `set_instance()` 调用。当前是否已修复需运行测试验证。

#### 风险 3：Optuna study 与 Research Job 的双轨并行

**位置**：`src/application/strategy_optimizer.py` vs `src/application/research_control_plane.py`

两条链路使用不同的数据库、不同的模型、不同的 API 路由。一个研究者可能：

1. 通过 API 发起 Research Job 获得 backtest 结果
2. 同时通过脚本运行 Optuna study 获得另一个结果
3. 两者的结果无法统一比较、统一管理

这不是"污染"，但会导致治理混乱：同一次研究活动的产物分散在两个系统中。

#### 风险 4：Candidate -> Runtime 的晋升路径完全缺失

当前 candidate review 后（即使状态变为 `RECOMMENDED`），没有任何机制将参数写入 runtime profile。这在 Sim-1 阶段是正确的安全设计，但意味着：

- 无法从研究结论自动推进到可执行配置
- 所有"promote"都是人工操作（复制参数、手动写入 runtime profile）
- 人工操作不可审计、不可追溯

### 6.3 防污染总结

| 防护层 | 已实现 | 状态 |
|--------|--------|------|
| Research DB 与 Runtime DB 物理隔离 | 是（SQLite vs PG / 独立 SQLite 文件） | 有效 |
| Research API 不提供 runtime 写入 | 是（`/api/research/*` 只有 CRUD） | 有效 |
| Candidate 不自动 promote | 是（无 promote API） | 有效 |
| CandidateRecord 与 runtime profile 无共享字段 | 是 | 有效 |
| Backtester ConfigManager 隔离 | 部分（legacy fallback 仍存在） | 需持续关注 |
| 研究脚本全局状态隔离 | 部分（8 个脚本仍有 `set_instance()` 调用） | 需修复 |
| 数据库文件级隔离 | 是（`research_control_plane.db` 独立文件） | 有效 |
| 配置来源优先级文档化 | 是（SSOT 文档） | 文档级，非代码级强制 |

## 七、研究控制面成熟度判定

### 7.1 判定

**B+**：有基本任务管理的研究工作台。

| 等级 | 定义 | 当前系统是否符合 |
|------|------|------------------|
| A | 可复现、可治理、可晋升的研究系统 | 否：缺少 IS/OOS 分层、Optuna 集成、promote 路径 |
| **B+** | **有基本任务管理的研究工作台** | **是：job/result/candidate 三对象模型 + 独立存储 + review 流程** |
| B | 有统一入口的研究工具 | 是，但超出 B 的范围（有 candidate review + rubric） |
| A- | 介于 B+ 和 A 之间 | 缺少：数据版本校验、代码版本统一记录、promote 门禁 |

### 7.2 判定证据

**支持 B+ 的证据**：

1. **独立对象模型**：`ResearchJob` / `ResearchRunResult` / `CandidateRecord`（`src/domain/research_models.py`）
2. **独立存储**：`ResearchRepository` 使用 `data/research_control_plane.db`（`src/infrastructure/research_repository.py`）
3. **完整 API**：`/api/research/jobs/backtest`、`/api/research/jobs`、`/api/research/runs/{id}`、`/api/research/candidates`、`/api/research/candidates/{id}/review`（`src/interfaces/api_research_jobs.py`）
4. **Job 生命周期**：PENDING -> RUNNING -> SUCCEEDED/FAILED，带 error_code 和 progress_pct
5. **Candidate review**：`CandidateReviewRequest` 支持 status + review_notes + risks + recommendation
6. **Optuna candidate report**：包含 git commit、source profile hash、reproduce command、strict rubric 评估（`src/application/strategy_optimizer.py` + `src/application/readmodels/candidate_service.py`）
7. **Review rubric 代码化**：`CandidateArtifactService._evaluate_review()` 实现了 Strict v1 / Loose v1 门禁
8. **边界保护**：详细设计、SSOT 文档、模块 docstring 多层声明 research/runtime 隔离

**阻止达到 A 的证据**：

1. **无 IS/OOS 分层**：`ResearchSpec.kind` 只有 `Literal["backtest"]`
2. **无 walk-forward**：不存在滚动窗口验证机制
3. **Optuna 未接入 Research Job**：`StrategyOptimizer` 与 `ResearchJobService` 是两套独立系统
4. **无 promote 路径**：candidate review 后没有代码化的晋升机制
5. **代码版本记录不完整**：Research Job / RunResult 不记录 git commit
6. **50 个脚本不在 job 管理范围内**：研究活动的主要载体（scripts）与 control plane 脱节

### 7.3 最影响"将研究结论转为 runtime 候选"的缺口

按影响排序：

1. **Promote 路径完全缺失**（影响：无法从 candidate 自动/半自动推进到 runtime profile。当前只能人工复制参数）
2. **Optuna 与 Research Job 未统一**（影响：Optuna 产出的最优参数没有进入 CandidateRecord 系统，review 流程分散）
3. **无 IS/OOS 方法学约束**（影响：即使 candidate review 通过，也只证明在特定 time window 上表现好，不保证样本外有效）
4. **50 个脚本不在治理范围内**（影响：实际研究活动的大部分产出没有 job 管理和 artifact 追踪）
5. **数据版本不可校验**（影响：复现依赖"数据没有被交易所修正"的假设）
