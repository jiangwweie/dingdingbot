## 一、 系统架构分层设计 (Clean Architecture)

为了保证"Zero Execution (零执行)"的绝对红线以及底层策略引擎的纯净，整个后端采用标准的**整洁架构 (Clean Architecture)**，严格划分四层：

1. **Entities & Domain Layer (领域核心层)**:
   - 全局数据模型 (Pydantic Models)。
   - **策略引擎** (Pinbar、EMA计算、MTF逻辑)。
   - **风控试算引擎** (仓位计算、止损预测)。
   - *（注：本层为纯计算逻辑，绝不允许出现网络请求、CCXT、文件 IO 等任何外部依赖！）*
2. **Use Case Layer (应用服务层)**:
   - **信号管道 (Signal Pipeline)**: 将收到的 K 线放入计算引擎，若产生有效信号，则拉取风控引擎进行试算。
   - **全局配置管理器 (Config Manager)**: 启动时加载和合并 `core.yaml` 和 `user.yaml`。
3. **Interface Adapters (接口适配层)**:
   - **API 端点**: FastAPI 提供给前端（未来的能力）的只读 REST API。
   - **通知适配器**: 将风控结果转换为飞书/企微 Webhook 支持的纯文本 Markdown 格式。
4. **Infrastructure Layer (基础设施层)**:
   - **交易所网关 (Exchange Gateway)**: 深度封装 `ccxt.pro`，负责历史数据预热 (REST)、增量行情流订阅 (WebSocket)、资产总线轮询 (REST)。

## 三、 配置文件 Schema 规范

### 3.1 `core.yaml` — 系统级配置（只读，开发者不得修改）

```yaml
# ============================================================
# 系统核心配置 - 仅架构师有权修改
# ============================================================

# 核心币池 (不可删改)
core_symbols:
  - "BTC/USDT:USDT"
  - "ETH/USDT:USDT"
  - "SOL/USDT:USDT"
  - "BNB/USDT:USDT"

# Pinbar 默认几何参数
pinbar_defaults:
  min_wick_ratio: 0.6          # 影线占全长的最低比例 (required, Decimal)
  max_body_ratio: 0.3          # 实体占全长的最高比例 (required, Decimal)
  body_position_tolerance: 0.1 # 实体位置偏差容许度 (required, Decimal)

# EMA 指标配置
ema:
  period: 60                   # EMA 周期 (required, int)

# MTF 层级校验映射 (低级别 -> 高级别)
mtf_mapping:
  "15m": "1h"
  "1h": "4h"
  "4h": "1d"
  "1d": "1w"

# 冷启动预热
warmup:
  history_bars: 100            # REST 预热拉取的历史 K 线数量 (required, int)
```

### 3.2 `user.yaml` — 用户级配置（可修改，重启生效）

```yaml
# ============================================================
# 用户配置 - 可按需修改，重启服务后生效
# ============================================================

# 交易所配置

  api_key: "your-api-key"      # API Key (required, 只读权限!)

  testnet: true                # 是否使用测试网 (optional, default: false)

# 用户自定义币池 (与 core_symbols 合并后去重)

  - "XRP/USDT:USDT"
  - "DOGE/USDT:USDT"

# 监控的时间周期列表
timeframes:
  - "15m"
  - "1h"
  - "4h"
  - "1d"
  - "1w"

# 策略开关
strategy:
  trend_filter_enabled: true    # EMA60 同周期趋势过滤 (required, bool)
  mtf_validation_enabled: true  # 多周期校验 (required, bool)

# 风控参数
risk:
  max_loss_percent: 0.01       # 单笔最大亏损占余额比例 (required, Decimal, e.g. 0.01 = 1%)
  max_leverage: 10             # 杠杆上限 (required, int)

# 资产快照轮询
asset_polling:
  interval_seconds: 60         # 轮询间隔 (optional, default: 60)

# 通知渠道
notification:
  channels:                    # 至少配置一个 (required, list)
    - type: "feishu"           # "feishu" | "wecom"
      webhook_url: "https://open.feishu.cn/open-apis/bot/v2/hook/xxx"
```

### 3.3 配置合并与校验规则

| 维度 | 规则 |
|------|------|
| **币池合并** | 最终列表 = `core_symbols` ∪ `user_symbols`，自动去重；`core_symbols` 不可被 `user_yaml` 覆盖或删除 |
| **必填校验** | 启动时 Pydantic 强校验所有 `required` 字段，缺失则抛 `FatalStartupError` 中止 |
| **API Key 权限检查** | 启动后立即调用交易所 `fetch_balance()` 等只读接口试探可用性；若 CCXT 返回权限中包含 `trade`、`withdraw`，则抛 `FatalStartupError` 拒绝启动 |
| **启动时打印** | 合并后的最终生效参数（脱敏后）必须完整打印到控制台日志 |

## 四、 标准化数据接口契约 (Interface Contracts)

开发前，A 和 B 必须首先在共用文件 `domain/models.py` 中就以下模型达成共识。**模型锁定后，任何变更须架构师审批。**

```python
from pydantic import BaseModel, Field
from decimal import Decimal
from typing import Optional, List
from enum import Enum

# ============================================================
# 枚举类型
# ============================================================
class Direction(str, Enum):
    LONG = "long"
    SHORT = "short"

class MtfStatus(str, Enum):
    CONFIRMED = "confirmed"     # 高级别趋势与信号方向一致
    REJECTED = "rejected"       # 高级别趋势与信号方向冲突
    DISABLED = "disabled"       # MTF 功能已关闭
    UNAVAILABLE = "unavailable" # 高级别数据尚未就绪

class TrendDirection(str, Enum):
    BULLISH = "bullish"         # 价格在 EMA 上方
    BEARISH = "bearish"         # 价格在 EMA 下方

# ============================================================
# A -> B : 基础数据模型
# ============================================================
class KlineData(BaseModel):
    """单根已闭合 K 线数据"""
    symbol: str                 # 如 "BTC/USDT:USDT"
    timeframe: str              # "15m", "1h", "4h", "1d", "1w"
    timestamp: int              # 收盘时间戳 (ms)
    open: Decimal
    high: Decimal
    low: Decimal
    close: Decimal
    volume: Decimal             # 成交量（基础资产）
    is_closed: bool = True

class PositionInfo(BaseModel):
    """单个合约持仓信息"""
    symbol: str
    side: str                   # "long" | "short"
    size: Decimal               # 持仓数量
    entry_price: Decimal
    unrealized_pnl: Decimal
    leverage: int

class AccountSnapshot(BaseModel):
    """合约账户快照"""
    total_balance: Decimal       # 账户总权益
    available_balance: Decimal   # 可用余额（可开新仓）
    unrealized_pnl: Decimal      # 未实现盈亏
    positions: List[PositionInfo] = Field(default_factory=list)
    timestamp: int               # 快照时间戳 (ms)

# ============================================================
# B -> A : 策略输出模型
# ============================================================
class SignalResult(BaseModel):
    """策略引擎输出的完整信号"""
    symbol: str
    timeframe: str
    direction: Direction
    entry_price: Decimal         # 建议入场价 (即当前 K 线收盘价)
    suggested_stop_loss: Decimal # 建议止损位
    suggested_position_size: Decimal  # 风控试算后的建议仓位
    current_leverage: int        # 实际使用杠杆
    ema_trend: TrendDirection    # 当前同周期 EMA60 趋势
    mtf_status: MtfStatus        # MTF 校验结果
    risk_reward_info: str        # 风控摘要 (如 "Risk 1% = 200 USDT")
```

## 五、 统一异常分类与错误码体系

所有异常定义在 `domain/exceptions.py` 中，全项目统一使用，禁止各模块私自定义异常。

```python
class CryptoMonitorError(Exception):
    """所有业务异常的基类"""
    def __init__(self, message: str, error_code: str):
        self.error_code = error_code
        super().__init__(f"[{error_code}] {message}")
```

### 异常分级表

| 级别 | 异常类 | 错误码前缀 | 触发场景 | 系统行为 |
|------|--------|-----------|---------|---------|
| **FATAL** | `FatalStartupError` | `F-` | API Key 含交易/提现权限；必填配置缺失；交易所初始化连接彻底失败 | 打印错误日志 → **拒绝启动，进程退出** |
| **CRITICAL** | `ConnectionLostError` | `C-` | WebSocket 重连超过最大重试次数；REST 轮询连续 N 次失败 | **飞书/企微发送系统级故障告警**（附带错误堆栈摘要）→ 进入降级休眠，等待重启 |
| **WARNING** | `DataQualityWarning` | `W-` | 单根 K 线数据异常（如 high < low）；某币种数据延迟超过阈值 | 日志记录 → **跳过本次该币种计算**，不影响其他币种 |
| **INFO** | （非异常，仅日志） | `I-` | 正常信号检测、正常推送完成 | 标准 INFO 日志 |

### 常见错误码速查

| 错误码 | 含义 |
|--------|------|
| `F-001` | API Key 包含交易 (trade) 权限 |
| `F-002` | API Key 包含提现 (withdraw) 权限 |
| `F-003` | 必填配置字段缺失 |
| `F-004` | 交易所连接初始化失败 |
| `C-001` | WebSocket 重连达上限 |
| `C-002` | REST 资产轮询连续失败 |
| `W-001` | K 线数据异常 (OHLCV 逻辑校验不通过) |
| `W-002` | 数据延迟超过阈值 |

## 六、 日志规范与敏感信息脱敏规则

### 6.1 强制约束

| 规则 | 要求 |
|------|------|
| **日志库** | 全项目统一使用 Python `logging` 模块（或 `loguru`），**严禁裸 `print()`** |
| **统一格式** | `[%(asctime)s] [%(levelname)s] [%(name)s] %(message)s` |
| **模块名** | 每个文件使用 `logger = logging.getLogger(__name__)` 获取模块级 logger |
| **日志级别** | `DEBUG` 仅开发环境；生产环境最低级别 `INFO` |

### 6.2 敏感信息脱敏

在 `infrastructure/logger.py` 中实现统一脱敏工具函数：

```python
def mask_secret(value: str, visible_chars: int = 4) -> str:
    """
    脱敏处理：只保留前4位和后4位，中间用 **** 替代。
    示例: "abcdefghijklmnop" -> "abcd****mnop"
    """
    if len(value) <= visible_chars * 2:
        return "****"
    return f"{value[:visible_chars]}****{value[-visible_chars:]}"
```

### 6.3 脱敏红线 (Review 一票否决)

以下字段在**任何日志输出、异常消息、通知推送**中，必须经过 `mask_secret()` 处理后才能打印：

- `api_key`
- `api_secret`
- `webhook_url`

**违反此规则的 PR 将被架构师直接打回。**

## 七、 后端并行开发方案 (两人工作量拆分)

为了确保两位后端开发同事可以完全**零阻塞**地并行工作，我们将整个系统沿着**"I/O 边界"**与**"纯粹计算边界"**将任务劈开。

双方协同的唯一桥梁是提前定义好的 **Pydantic 数据模型 (Domain Models)**。

### 🧑‍💻 开发人员 A：基础设施与数据管道 (Infrastructure & I/O 专家)

**角色定位**：处理所有与外部系统打交道的"脏活"，如网络请求、异步调度、解析配置、推送消息。
**工作目录**：`src/application/`、`src/infrastructure/`、`src/interfaces/`、`config/`

**开发任务**：
1. **统一模型定义 (与 Dev B 共同敲定)**：基于 Pydantic 设计全局核心对象（详见第四节），落地到 `domain/models.py`。
2. **配置管理器 (`application/config_manager.py`)**：
   - 加载并用 Pydantic 强校验 `core.yaml` 和 `user.yaml`。
   - 实现合并逻辑：`core_symbols` 不可被覆盖或删除。
   - API Key 权限检测：检测到 `trade` / `withdraw` 权限时抛出 `FatalStartupError(F-001/F-002)`。
   - 启动时将合并后的所有生效参数（脱敏后）打印到控制台。
3. **数据接入层 (`infrastructure/exchange_gateway.py`)**：
   - 编写 CCXT REST 异步预热脚本（拉取前 100 根 K 线并写入缓存）。
   - 编写 CCXT.Pro WebSocket 订阅系统，按需只推送 `is_closed=True` 的完整 K 线。
   - 实现 WebSocket 自动重连与指数退避机制，超过最大重试次数抛出 `ConnectionLostError(C-001)`。
   - 编写定时轮询资产的 `BackgroundTask`，更新到全局内存中 `AccountSnapshot`。
4. **统一日志 (`infrastructure/logger.py`)**：日志格式配置、`mask_secret()` 工具函数。
5. **报警触达层 (`infrastructure/notifier.py`)**：
   - 飞书/企业微信 Webhook 发送封装。
   - 将 `SignalResult` 格式化为纯文本 Markdown 消息。
   - 实现系统级故障告警（包含错误码与截断后的堆栈信息）。

**A 的单元测试策略**：
- 不要真实连接交易所！使用 `pytest` + `aioresponses` 库 Mock 交易所返回的 JSON。
- 测试覆盖点：配置合并边界、权限检测拦截、断线重连逻辑、消息格式化、脱敏输出。

---

### 🧑‍💻 开发人员 B：领域专家 (业务策略与风控计算)

**角色定位**：专注数学模型与纯逻辑。不关心数据从哪里来，不关心告警通过什么发出去，只关心"输入数据 -> 策略判定 -> 输出结果"。
**工作目录**：`src/domain/`（除 `models.py` 与 Dev A 共同维护外，其余文件为 B 独占）

**开发任务**：
1. **异常定义 (`domain/exceptions.py`)**：按第五节的异常分级表实现所有异常类。
2. **指标计算器 (`domain/indicators.py`)**：基于 numpy/pandas 或原生 Python 实现高效的 EMA 流式计算与状态更新。
3. **策略判定引擎 (`domain/strategy_engine.py`)**：
   - 实现无颜色的 Pinbar 几何结构判别器。
   - 实现同周期 EMA60 过滤器，输出 `TrendDirection` 枚举。
   - 实现 MTF (多周期) 层级校验 (15m→1h→4h→1d→1w)，输出 `MtfStatus` 枚举。
   - 实现四种组合逻辑门：`if (pinbar_match && (!trend_switch || trend_ok) && (!mtf_switch || mtf_ok))`。
4. **风控试算模块 (`domain/risk_calculator.py`)**：
   - **强制要求**：所有参数使用 `decimal.Decimal`，严禁 `float`。
   - 核心公式：`Position_Size = (Balance × Loss_Percent) / Stop_Loss_Distance`。
   - 杠杆熔断：计算结果不得超过 `MAX_LEVERAGE` 对应的最大仓位。
   - 输入 `AccountSnapshot` + `KlineData` + 配置参数，输出完整的 `SignalResult`。

**B 的单元测试策略**：
- **极端边界测试**：手动构造极端 K 线数组（极端插针、伪 Pinbar、实体异常等），断言判定结果。
- 编写 MTF 时，伪造不同级别的 EMA 数据来测试"多空过滤"是否精准生效。
- 风控计算必须精确到 `Decimal`，测试用例中断言必须使用 `Decimal` 比较，禁止 `assertAlmostEqual`。

## 八、 审查 (代码 Review) 与模块整合验收流程

作为架构师，我将把控进度、Review 代码、并主导两层模块的最终结合。

### 1. 代码 Review 标准 (架构师验收卡点)

当 A 或 B 提交 Pull Request 时，除了基础语法，架构师将重点审核：

**【对 Dev A 的审查重点】**：
- CCXT 预热过程是否会阻塞 `asyncio` 事件循环？
- API Key / Secret / Webhook URL 是否存在被日志意外打印的泄漏风险？（一票否决）
- 冷启动合并配置是否覆盖了所有异常路径？
- 异常抛出是否使用了统一的异常类和错误码？

**【对 Dev B 的审查重点】**：
- `domain/` 下文件的 `import` 列表是否绝对干净？（不允许 ccxt/aiohttp/requests/yaml 等）
- 是否全局严守了 `Decimal` 类型？（有 float 立刻打回）
- Pinbar 及 MTF 的计算性能复杂度是否在 O(1) 或 O(N)？
- 策略输出是否完整填充了 `SignalResult` 的所有字段？

### 2. 联调与集成测试计划 (Integration Test)

当两人各自单元测试（Coverage > 80%）通过后，架构师将统筹整合：

1. **沙盒联调测试 (Playground Test)**：不连交易所网络。将 A 准备好的一批真实 CCXT 录播 JSON 数据输入到系统中，观察系统是否能够成功完成：**假数据流入 → B 引擎识别 → B 风控试算 → 组装成 Markdown → A 模拟发送** 整个闭环。

2. **网络压力与混沌测试 (Chaos Test)**：开启 Binance Testnet WebSocket 连接，架构师手动断开网络，验证 A 的恢复机制和熔断告警能否正确输出带有错误码和堆栈摘要的飞书报警。

3. **零执行红线检验**：刻意在 `user.yaml` 注入带有执行权限的 API Key，架构师启动系统并查验系统是否立即触发 `FatalStartupError(F-001)` 并拒绝启动。
