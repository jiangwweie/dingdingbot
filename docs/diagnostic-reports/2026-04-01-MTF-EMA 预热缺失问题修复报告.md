## 二、原因分析

### 2.1 初步假设

| 假设 | 可能性 | 验证方法 | 验证结果 |
|------|--------|----------|----------|
| WebSocket 数据未同步 | 中 | 检查 WebSocket 订阅状态 | ✅ WebSocket 正常 |
| 1h K 线未闭合 | 中 | 检查 K 线闭合判断逻辑 | ✅ 逻辑正确 |
| MTF EMA 未预热 | 高 | 检查 `_mtf_ema_indicators` 初始化 | ❌ **根因** |

### 2.2 根因定位（5 Why 分析）

```
Why 1: 为什么 MTF 过滤器返回 higher_tf_data_unavailable？
  → _get_closest_higher_tf_trends 返回空字典 {}

Why 2: 为什么返回空字典？
  → EMA.is_ready = False，不满足返回趋势的条件

Why 3: 为什么 EMA.is_ready = False？
  → EMACalculator 需要 60 个数据点才能 is_ready=True，但只更新了 1 次

Why 4: 为什么只更新了 1 次？
  → EMA 在第一次调用时才懒创建，之前没有预热

Why 5: 为什么没有预热？
  → _build_and_warmup_runner 方法只预热了 Runner 内部指标，
    没有预热 _mtf_ema_indicators
```

### 2.3 代码分析

**问题代码位置**: `src/application/signal_pipeline.py` 第 312-342 行


### 2.4 问题本质

**Warmup 盲区**: 启动时的 Warmup 只预热了 `StrategyRunner` 内部的状态（如 ATR、形态检测等），但遗漏了 `_mtf_ema_indicators` 这个独立维护的 EMA 指标字典。

**懒创建的陷阱**: `_mtf_ema_indicators` 采用懒创建模式（第一次调用时才初始化），但在第一次调用时，只能用当前 K 线更新一次，无法满足 EMA 需要 60 个数据点的要求。

## 四、修复方案

### 4.1 方案设计

**目标**: 在 `_build_and_warmup_runner` 方法中，用历史 K 线数据预热 `_mtf_ema_indicators`。

**预热范围**: 1h/4h/1d 高周期（用于 MTF 过滤的时间周期）

**预热方法**: 遍历 `_kline_history`，为每个高周期创建 EMACalculator 并用 `history[:-1]`（排除当前运行中的 K 线）更新。

### 4.2 代码修改

**文件**: `src/application/signal_pipeline.py`
**位置**: `_build_and_warmup_runner` 方法（第 344-375 行）

**修改内容**:

```python
# 在 Runner warmup 后添加 MTF EMA 预热逻辑
# MTF EMA warmup: pre-warm higher timeframe EMA indicators
# This ensures MTF filters have ready EMAs on first signal check
# Note: key format is "symbol:timeframe" where symbol may contain ":" (e.g., "BTC/USDT:USDT:1h")
mtf_warmup_count = 0
mtf_debug_keys = []
for key, history in self._kline_history.items():
    parts = key.split(":")
    # 从后往前解析 timeframe，因为 symbol 可能包含 ":"
    timeframe = parts[-1] if parts[-1] in ["1h", "4h", "1d", "1w"] else parts[-2]

    # 只预热高周期 EMA (用于 MTF 过滤)
    if timeframe in ["1h", "4h", "1d"]:
        ema_key = key
        if ema_key not in self._mtf_ema_indicators:
            self._mtf_ema_indicators[ema_key] = EMACalculator(period=self._mtf_ema_period)

        ema = self._mtf_ema_indicators[ema_key]
        mtf_debug_keys.append(f"{ema_key} ({len(history)} bars)")

        # Warmup EMA with historical K-lines (exclude currently running kline)
        for kline in history[:-1]:
            ema.update(kline.close)
            mtf_warmup_count += 1

# Log MTF EMA warmup completion
logger.info(f"MTF EMA warmup: checked {len(mtf_debug_keys)} keys, warmed {mtf_warmup_count} data points across {len(self._mtf_ema_indicators)} indicators")
if mtf_warmup_count > 0:
    logger.info(f"MTF EMA warmup complete: {mtf_warmup_count} data points across {len(self._mtf_ema_indicators)} indicators ready")
elif self._kline_history:
    logger.info("MTF EMA warmup skipped: no higher timeframe data available yet")
```

### 4.3 方案对比

| 方案 | 优点 | 缺点 | 工作量 |
|------|------|------|--------|
| A: 在 `_get_closest_higher_tf_trends` 中批量更新 EMA | 代码改动小 | 每次调用都要检查，性能差 | 1h |
| B: 在 `_build_and_warmup_runner` 中预热（推荐）| 一次性预热，性能好 | 需要修改 warmup 逻辑 | 2h |
| C: 启动时单独调用预热方法 | 逻辑清晰 | 增加额外方法，复杂度高 | 3h |

**选择方案 B**: 性能好，逻辑内聚，符合 Warmup 设计意图。

## 六、经验总结

### 6.1 技术教训

1. **懒创建的陷阱**: 懒创建（Lazy Initialization）虽然节省资源，但在有状态依赖的场景下可能导致初始化时序问题。

2. **Warmup 完整性**: 系统启动时的 Warmup 必须覆盖所有有状态的组件，尤其是跨方法/模块共享的状态。

3. **日志可见性**: 关键初始化过程应该有清晰的日志输出，便于问题排查。本次修复增加了 `MTF EMA warmup complete` 日志。

### 6.2 架构反思

**问题根源**: `_mtf_ema_indicators` 作为独立于 `StrategyRunner` 的状态字典，在 Warmup 设计时被遗漏。

**改进建议**:
1. 将 `_mtf_ema_indicators` 集成到 `StrategyRunner` 内部，统一由 Runner 管理状态
2. 或者在 Warmup 方法中显式调用各个组件的 `warmup()` 方法

### 6.3 预防措施

1. **Warmup 检查清单**: 建立 Warmup 检查清单，确保所有有状态组件都被预热
   - [ ] StrategyRunner 状态
   - [ ] MTF EMA 指标
   - [ ] ATR 指标
   - [ ] 其他自定义指标

2. **启动自检**: 系统启动后自动检查关键组件的就绪状态，未就绪时记录警告日志

3. **监控告警**: 对 `higher_tf_data_unavailable` 等关键错误设置告警阈值
