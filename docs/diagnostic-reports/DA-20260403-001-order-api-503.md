2. lifespan 函数未初始化 `_repository` (api.py:284-295)
3. 启动命令 `uvicorn src.interfaces.api:app` 仅启动 FastAPI，未运行完整系统
4. 日志显示多个 API 返回 503：`/api/signals`, `/api/config` 等

## 根因分析（5 Why）

```
Why 1: 为什么 API 返回 503 "Repository not initialized"？
  → 因为 `_repository` (SignalRepository) 为 None

Why 2: 为什么 `_repository` 为 None？
  → 因为 FastAPI lifespan 启动时未初始化

Why 3: 为什么 lifespan 未初始化 `_repository`？
  → 因为 lifespan 函数只有 shutdown 清理逻辑，没有 startup 初始化逻辑

Why 4: 为什么 lifespan 缺少 startup 初始化？
  → 因为系统设计期望通过 `main.py` 启动完整系统，而不是直接启动 FastAPI

Why 5: 为什么用户直接启动 FastAPI 而不是 main.py？
  → 因为开发环境测试通常使用 `uvicorn` 直接启动，缺少对两种启动模式的明确区分
```

**根本原因**: FastAPI 应用有两种启动模式（standalone vs full system），但 lifespan 仅实现了 full system 模式的清理逻辑，未实现 standalone 模式的初始化逻辑。

**问题代码位置**: `src/interfaces/api.py:284-295` (lifespan 函数)

**影响范围评估**:
- 所有依赖 `_repository` (SignalRepository) 的 API 端点都会返回 503
- 包括：`/api/signals/*`, `/api/config`, `/api/diagnostics` 等
- `/api/v3/orders/*` 端点使用 `_get_order_repo()`，理论上不受影响
- 但 pending 问题可能与数据库锁或异步操作有关

## 修复方案

### 方案 A [推荐] - lifespan 完整初始化

**修改内容**:
```python
文件：src/interfaces/api.py
位置：第 284-295 行
当前代码：
    @asynccontextmanager
    async def lifespan(app: FastAPI):
        """Manage application lifespan events."""
        # Startup
        yield
        # Shutdown
        if _repository is not None:
            await _repository.close()

修改为：
    @asynccontextmanager
    async def lifespan(app: FastAPI):
        """Manage application lifespan events.
        Initialize repositories on startup, close on shutdown.
        """
        # Startup - 初始化所有 Repository
        from src.infrastructure.signal_repository import SignalRepository
        from src.infrastructure.order_repository import OrderRepository
        from src.infrastructure.config_entry_repository import ConfigEntryRepository
        
        global _repository, _order_repo, _config_entry_repo
        
        # 初始化 SignalRepository
        if _repository is None:
            _repository = SignalRepository()
            await _repository.initialize()
        
        # 初始化 ConfigEntryRepository
        if _config_entry_repo is None:
            _config_entry_repo = ConfigEntryRepository()
            await _config_entry_repo.initialize()
        
        # OrderRepository 按需创建，不在此初始化
        
        yield
        
        # Shutdown - 清理所有 Repository
        if _repository is not None:
            await _repository.close()
        if _config_entry_repo is not None:
            await _config_entry_repo.close()
```

**优点**:
- ✅ 解决所有 Repository 未初始化问题
- ✅ 支持 standalone 模式启动 FastAPI
- ✅ 与现有 `set_dependencies()` 兼容（注入优先）
- ✅ 无需修改 `main.py`

**缺点/风险**:
- ⚠️ 每次请求都会创建 Repository 实例（如果未注入）
- ⚠️ 需要确保所有 Repository 有 `initialize()` 方法

**预估工作量**: 0.5 小时

### 方案 C - API 端点懒加载

**修改内容**:
修改 `_get_repository()` 添加懒加载逻辑：

```python
文件：src/interfaces/api.py
位置：第 230-234 行
当前代码：
    def _get_repository() -> SignalRepository:
        """Get repository or raise error if not initialized."""
        if _repository is None:
            raise HTTPException(status_code=503, detail="Repository not initialized")
        return _repository

修改为：
    async def _get_repository() -> SignalRepository:
        """Get repository, initializing if needed."""
        global _repository
        if _repository is None:
            from src.infrastructure.signal_repository import SignalRepository
            _repository = SignalRepository()
            await _repository.initialize()
        return _repository
```

**优点**:
- ✅ 最小修改
- ✅ 按需初始化

**缺点/风险**:
- ⚠️ 需要将所有调用改为 `await _get_repository()`
- ⚠️ 可能导致多次初始化（需要加锁）

**预估工作量**: 1 小时
