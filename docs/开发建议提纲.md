## 致技术架构师的开发提纲（交底专用）

### 1. 架构选型建议

- **核心模式**：Clean Architecture / DDD（领域驱动设计）。确保领域层（策略/风控）纯净，无外部依赖。
- **并发处理**：采用异步 IO（如 Python `asyncio`），支持多币种、多周期的 WebSocket 并发订阅。
- **配置管理**：**拒绝热更新逻辑**。采用静态配置加载流，启动时完成参数验证与预热。

### 2. 核心逻辑实现要点

- **WebSocket 模块**：
  - 实现自动重连与指数退避机制。
  - 仅在 `is_closed=True` 的 K 线闭合点触发策略计算。
- **策略引擎模块**：
  - 解耦 Pinbar 判断算法。
  - 实现四种过滤组合的逻辑门：`if (geometric_match && (!trend_switch || trend_match) && (!mtf_switch || mtf_match))`。
- **风控模块**：
  - 使用 `Decimal` 类型进行所有金额与比例运算，严禁使用 `Float` 以防精度丢失。
  - 计算公式：`Position_Size = (Balance * Loss_Percent) / Stop_Loss_Distance`。

### 3. 安全与合规限制（高压线）

- **权限最小化**：API 密钥验证时，若检测到“交易/提现”权限，系统必须拒绝启动并报错。
- **日志规范**：启动时必须在控制台清晰打印当前所有生效参数（策略、风控、币种清单）。

### 4. 交付优先级 (Roadmap)

1. **Phase 1**：基础 WebSocket 订阅 + 简单行情打印。
2. **Phase 2**：Pinbar 形态识别逻辑 + 静态配置加载。
3. **Phase 3**：EMA 趋势过滤与 MTF 校验开关。
4. **Phase 4**：账户只读查询与风控试算推送。
