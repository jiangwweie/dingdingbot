# 阶段总结：执行恢复主线收口与第二阶段启动前夜

> 时间范围：近三天
> 记录目的：沉淀这三天真正对后续还有价值的决策、演进和纠偏，而不是重复流水账

---

## 1. 这三天我们到底完成了什么

这三天的核心成果，不是“又修了几个 bug”，而是把实盘执行链从一条不断打补丁的实现线，收口成了一条相对清晰的主线：

1. `ExecutionIntent` 已明确进入 PG 主线，成为执行意图真源
2. 恢复语义不再挂在 SQLite 或内存补丁上，而是正式收敛为 PG `execution_recovery_tasks`
3. `circuit_breaker` 不再被当成独立真源，而是被明确为恢复任务的派生保护状态
4. SQLite `pending_recovery` 过渡链已完全移除，恢复主链不再双真源
5. 启动恢复流程已经形成：
   - Phase 4.3：启动对账
   - Phase 4.4：由 PG active recovery tasks 重建 breaker 缓存

一句话概括：

**执行恢复链已经从“能跑”收敛到“主线清楚、真源清楚、恢复来源清楚”。**

---

## 2. 这三天最重要的架构决策

### 决策 A：执行/恢复/对账/熔断类状态，默认优先考虑 PG

这是这几天最重要的总原则。

之前一些最小补丁会自然滑向：

- 先放内存
- 再放 SQLite
- 之后再想办法迁 PG

这样短期快，但长期一定会制造主线分叉。

这几天我们把这个原则真正锁定了：

**凡是强执行语义状态，设计时先判断是否应该进入 PG 主线。**

只有在明确是过渡态，且不会形成长期技术债时，才允许临时落 SQLite 或内存。

---

### 决策 B：`pending_recovery` 不塞进 `orders`，也不粗暴塞进 `execution_intents`

我们认真比较过三种落点：

1. 塞进 `orders`
2. 塞进 `execution_intents`
3. 独立成 PG 恢复表

最后确认：

**恢复语义更像“恢复工单/恢复任务”，而不是订单事实或 intent 本体字段。**

所以正式方案定为：

- `execution_recovery_tasks` 作为 PG 正式恢复真源

这一步很关键，因为它让后面的恢复、重试、对账、运维语义都有了独立且清晰的归宿。

---

### 决策 C：`circuit_breaker` 不单独做成新 PG 表

第二阶段入口议题原本是：

**`circuit_breaker` 是否需要 PG 真源化**

最后架构判断结论是：

**不应该单独建 breaker 表。**

更合理的方式是：

1. `execution_recovery_tasks` 继续作为恢复真源
2. `circuit_breaker` 作为 active recovery tasks 的派生保护状态
3. 运行时保留内存 breaker 集合作为快速判断缓存
4. 程序启动后，由 PG active recovery tasks 重建 breaker

这个结论避免了又新增一个“看起来像真源，实际只是派生状态”的表。

---

## 3. 这三天最值得保留的纠偏经验

### 经验 1：不是所有“最小补丁”都值得继续演进

SQLite `pending_recovery` 一开始是合理的，因为它解决了“跨进程脚本看不到内存状态”的现实问题。

但后面继续扩它，就会和 PG 主线冲突。

所以这三天一个很关键的纠偏是：

**承认它曾经有价值，但及时把它降级为过渡态，并最终删除。**

这比“为了不浪费已经写的代码而继续沿错误方向优化”要健康得多。

---

### 经验 2：全局视野没问题，但当前层只能锁一个入口

这几天并不是没有看到更多事情：

- PG 继续迁移
- breaker 真源化
- recovery 重试/退避
- 回测精细化
- 前端重构

问题不是看不到，而是如果全部同时进入执行态，就会重新制造主线分裂。

所以后来采用的方式是：

**全局上看见后续多条线，但当前层只允许锁定一个入口议题。**

这也是第二阶段后来能稳定下来，而不是再次散掉的原因。

---

### 经验 3：主线文档必须轻，理由文档单独放

planning 文档之前已经明显过大。

这几天做的整理证明了一个很实用的经验：

- `task_plan.md` 写当前层
- `findings.md` 写还影响当前决策的结论
- `progress.md` 写近期完成和下一步
- 长历史全部归档

这样才不会每次开工都被旧内容淹没。

## 4. 当前架构快照

### 恢复主链现在怎么工作

#### 真源

- PG `execution_recovery_tasks`

#### 派生缓存

- 内存 `circuit_breaker_symbols`

#### 推进器

- `StartupReconciliationService`

#### 运行时判断

- `execute_signal()` 只查内存 breaker 集合做快速拒绝

#### 启动恢复

1. Phase 4.3：启动对账
2. Phase 4.4：由 PG active recovery tasks 重建 breaker

---

### 当前已经明确不再存在的旧链路

下面这些已经不应再被视为正式恢复主链的一部分：

1. SQLite `pending_recovery`
2. 依赖 orchestrator 内存 `_pending_recovery` 的恢复逻辑
3. 把 breaker 当作独立持久化真源的方案

---

## 5. 当前阶段的边界

### 当前主线

**执行恢复状态收敛与可运营化**

### 当前不是主线的事

1. 回测精细化
2. 参数继续搜索
3. 前端/API 扩张
4. 同时推进多张新 PG 表

这不是说这些不重要，而是：

**现在不应该和执行恢复主线抢注意力。**

---

## 6. 进入第二阶段前，我们已经具备了什么

进入第二阶段，不是从零开始，而是在这几个基础上继续：

1. 执行意图有 PG 真源
2. 恢复任务有 PG 真源
3. breaker 已明确为派生状态
4. SQLite 过渡链已移除
5. 主链从逻辑上已经不再分叉

这意味着第二阶段不再需要反复争论“恢复状态到底放哪”，而是可以在既定方向上继续推进。

---

## 7. 第二阶段的真正起点

第二阶段不是“继续随便修点东西”，而是要沿着已经锁定的方向推进：

### 主题

**执行恢复状态继续向 PG 收敛，并把执行链从“可运行”推向“可运营”。**

### 已经完成的第一步

- breaker 已改为由 PG active recovery tasks 重建

### 后续更适合展开的议题

1. recovery task 的 retry / backoff 语义是否要正式化
2. 运行时恢复和启动恢复的责任边界是否需要再压缩
3. 少量残余的初始化/注入一致性问题如何作为 P2 收尾

## 8. 这份“历程”文档以后适合记录什么

这个目录不应该变成第二套 planning。

我建议它只记录下面三类东西：

1. **阶段总结**
   - 一个阶段结束时，记录真正留下来的结论和转折

2. **关键纠偏**
   - 某条线走偏了，后来为什么改向，新的方向是什么

3. **阶段交接点**
   - 进入下一阶段前，当前已经具备什么、哪些不再讨论、哪些要继续带入

不适合记录：

1. 每日流水账
2. 细碎 bug 修复清单
3. 与当前主线无关的临时想法

换句话说：

**`docs/journey/` 记录的是“演进脉络”，不是“执行明细”。**
