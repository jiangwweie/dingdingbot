## 新功能

### 1. 动态风险头寸计算（方案 B）

**问题**: 之前的风险计算仅使用 `total_balance`，未考虑账户实际可用余额和现有持仓占用，可能导致过度风险暴露。

**解决方案**: 实现方案 B 动态风险调整机制：

- **使用 `available_balance`** - 基于实际可用余额计算风险，而非总余额
- **动态暴露限制** - 考虑现有持仓占用，当接近 `max_total_exposure` (默认 80%) 时自动降低风险
- **风险空间耗尽时返回 0 仓位** - 无可用风险空间时自动拒绝新信号

**核心公式**:
```
current_exposure_ratio = total_position_value / total_balance
available_exposure = max(0, max_total_exposure - current_exposure_ratio)
risk_amount = min(
    available_balance * max_loss_percent,
    available_balance * available_exposure
)
position_size = risk_amount / stop_loss_distance
```

**问题**: MTF 过滤器的多周期数据对齐逻辑需要优化，确保使用正确的 K 线计算趋势。

- 新增配置字段 `mtf_ema_period` 和 `mtf_mapping`
- MTF 趋势计算使用最后闭合的 K 线，而非运行中的 K 线
- 集成测试验证 MTF 趋势对齐逻辑

## 配置变更

### 新增配置项

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `max_total_exposure` | Decimal | 0.8 (80%) | 最大总暴露限制 |
| `mtf_ema_period` | int | 60 | MTF 趋势计算 EMA 周期 |
| `mtf_mapping` | Dict | {"15m":"1h","1h":"4h",...} | MTF 周期映射关系 |

### 示例配置

```yaml
# config/user.yaml
risk:
  max_loss_percent: 0.01  # 1%
  max_leverage: 10
  max_total_exposure: 0.8  # 80%

mtf:
  ema_period: 60
  mapping:
    15m: 1h
    1h: 4h
    4h: 1d
    1d: 1w
```

## 技术亮点

### 1. 测试驱动开发
- 先编写 35 个单元测试验证核心逻辑
- 再编写 16 个集成测试验证端到端流程
- 测试覆盖率 100%

### 2. Decimal 精度
- 所有金额计算使用 `decimal.Decimal`
- 禁止使用 `float` 进行金融计算
- 8 位小数精度支持小币种

### 3. 边界场景处理
- 零可用余额返回 0 仓位
- 超出暴露限制返回 0 仓位
- 止损距离为零抛出异常
