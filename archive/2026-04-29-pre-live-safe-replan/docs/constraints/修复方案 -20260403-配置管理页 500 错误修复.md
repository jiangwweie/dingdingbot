## 1. 问题背景与诊断摘要

### 1.1 问题症状汇总

用户反馈 7 个问题症状，经诊断归因为 4 个根本原因 + 1 个 UI 需求：

| 问题 ID | 症状 | HTTP 状态码 | 错误信息 | 根因分类 |
|--------|------|------------|----------|----------|
| 问题 1 | `/api/config/snapshots` | 500 | `__init__() missing 'error_code'` | D 类 - 错误响应格式 |
| 问题 2 | `/api/strategy/params/templates` | 404 | Not Found | C 类 - 前端路径错误 |
| 问题 3 | 订单列表 | UI 需求 | 需移除订单 ID 列，增加序号列 | UI 需求 |
| 问题 4 | `/api/strategies` | 500 | Pydantic 验证错误 | B 类 - 字符串引用 |
| 问题 5 | `/api/strategies/templates` | 500 | Pydantic 验证错误 | B 类 - 字符串引用 |
| 问题 6 | `/api/strategy/params` | 500 | `'NoneType' object has no attribute 'execute'` | A 类 - Repo 未初始化 |
| 问题 7 | `/api/strategy/params/preview` | 500 | `'NoneType' object has no attribute 'execute'` | A 类 - Repo 未初始化 |

### 1.2 根因分析

**A 类问题 - Repository 未初始化 (问题 6, 7)**

- **位置**: `src/interfaces/api.py:266-274` `_get_config_entry_repo()`
- **根因**: Fallback 创建 `ConfigEntryRepository` 实例后，虽调用 `await repo.initialize()`，但底层数据库连接可能未正确初始化
- **影响**: 所有依赖 `_get_config_entry_repo()` 的端点在特定条件下会抛出 `'NoneType' object has no attribute 'execute'`

**B 类问题 - Pydantic 字符串引用 (问题 4, 5)**

- **位置**: `src/interfaces/api.py:2198, 2237`
- **根因**: `response_model="StrategyListResponse"` 使用字符串引用而非类引用，与 Pydantic v2 不兼容
- **代码示例**:
  ```python
  @app.get("/api/strategies", response_model=StrategyListResponse)  # 正确
  @app.get("/api/strategies", response_model="StrategyListResponse")  # 错误 - 字符串引用
  ```

**C 类问题 - 前端路径错误 (问题 2)**

- **位置**: `gemimi-web-front/src/lib/api.ts:1643`
- **根因**: 前端调用 `/api/strategy/params/templates`，但后端仅存在 `/api/strategies/templates` 端点
- **影响**: 策略参数模板管理功能完全不可用

**D 类问题 - 错误响应格式 (问题 1)**

- **位置**: `src/interfaces/api.py:3669-3732` 等多处异常处理
- **根因**: 异常处理返回 `{"error": str(e)}` 而非 `ErrorResponse(error_code=..., message=...)`
- **违反规范**: `docs/arch/2026-03-25-系统开发规范与红线.md` API 错误处理规范

## 2. 架构方案设计

### 2.1 方案对比分析

#### A 类问题修复方案

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| A1 | 修复 `_get_config_entry_repo()` 初始化逻辑 | 简单直接 | 仅修复表面问题 | ⭐⭐ |
| A2 | 统一 Repository 初始化模式，移除 fallback 逻辑 | 架构一致性好 | 需要修改多处 | ⭐⭐⭐ |
| A3 | 引入依赖注入容器管理 Repository 生命周期 | 最佳实践 | 重构成本高 | ⭐⭐⭐⭐ |

**推荐方案：A2** - 统一 Repository 初始化模式

理由：
- 当前代码存在 `lifespan` 初始化和 `_get_*_repo()` fallback 两种模式，容易导致状态不一致
- 移除 fallback 逻辑，强制在 `lifespan` 中完成所有初始化，符合 FastAPI 最佳实践

#### B 类问题修复方案

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| B1 | 移除字符串引号，直接使用类引用 | 简单，零成本 | 需要检查所有端点 | ⭐⭐⭐⭐⭐ |
| B2 | 使用 `from __future__ import annotations` | 支持前向引用 | 非根本解决方案 | ⭐⭐ |

**推荐方案：B1** - 直接使用类引用

理由：
- Pydantic v2 完全支持在 `response_model` 中使用类引用
- 字符串引用主要用于前向引用场景，此处不适用

#### C 类问题修复方案

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| C1 | 修改前端 API 调用路径 | 快速修复 | 需要前后端协调 | ⭐⭐⭐ |
| C2 | 新增后端端点 `/api/strategy/params/templates` | 前端零改动 | 增加端点冗余 | ⭐⭐ |
| C3 | C1 + 后端 HTTP 308 重定向支持 | 兼顾兼容性 | 稍复杂 | ⭐⭐⭐⭐ |

**推荐方案：C3** - 前端修复路径 + 后端 HTTP 308 重定向

理由：
- 短期：后端添加重定向，现有前端代码可正常工作
- 长期：前端逐步迁移到正确路径

#### D 类问题修复方案

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| D1 | 逐处修改异常处理返回 | 简单直接 | 容易遗漏 | ⭐⭐ |
| D2 | 创建统一异常处理中间件 | 集中管理 | 需要重构 | ⭐⭐⭐⭐ |
| D3 | D1 + 静态分析检查 | 即时修复 + 预防 | 需要工具支持 | ⭐⭐⭐⭐⭐ |

**推荐方案：D3** - 立即修复 + 长期中间件方案

理由：
- 短期内逐处修复最快见效
- 长期应引入 HTTP 异常处理中间件，统一转换异常为 `ErrorResponse`

### 3.2 详细修复方案

#### A 类修复 - Repository 初始化

```python
# 当前问题代码 (api.py:266-274)
async def _get_config_entry_repo() -> Any:
    global _config_entry_repo
    if _config_entry_repo is None:
        from src.infrastructure.config_entry_repository import ConfigEntryRepository
        _config_entry_repo = ConfigEntryRepository()
        await _config_entry_repo.initialize()  # 可能未正确初始化
    return _config_entry_repo

# 修复方案 - 移除 fallback，强制 lifespan 初始化
async def _get_config_entry_repo() -> Any:
    """Get config entry repository.
    
    Raises:
        HTTPException: 503 Service Unavailable if repository not initialized
    """
    if _config_entry_repo is None:
        raise HTTPException(
            status_code=503, 
            detail="ConfigEntryRepository not initialized",
            headers={"X-Error-Code": "CONFIG-001"}
        )
    return _config_entry_repo
```

## 4. 接口契约

### 4.1 错误响应格式规范

所有 API 错误响应必须遵循以下格式：

```json
{
  "error_code": "CONFIG-001",
  "message": "ConfigEntryRepository not initialized"
}
```

**错误代码命名规范**:
- `CONFIG-xxx`: 配置管理相关错误
- `STRATEGY-xxx`: 策略管理相关错误
- `REPO-xxx`: Repository 层错误
- `VALIDATION-xxx`: 参数验证错误

### 4.2 端点路径规范

| 功能 | 正确路径 | 错误路径 (已废弃) |
|------|----------|------------------|
| 策略模板列表 | `GET /api/strategies/templates` | `GET /api/strategy/params/templates` |
| 策略详情 | `GET /api/strategies/{id}` | - |
| 策略参数 | `GET /api/strategy/params` | - |

### 7.2 架构改进建议

**建议 1: 引入依赖注入容器**

```python
# 使用 FastAPI Depends 系统
def get_config_entry_repo() -> ConfigEntryRepository:
    if _config_entry_repo is None:
        raise HTTPException(status_code=503, detail="Repository not initialized")
    return _config_entry_repo

@app.get("/api/strategy/params")
async def get_strategy_params(repo: ConfigEntryRepository = Depends(get_config_entry_repo)):
    ...
```

**建议 2: 统一异常处理中间件**

```python
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    error_code = getattr(exc, 'error_code', 'UNKNOWN-001')
    return ErrorResponse(
        error_code=error_code,
        message=str(exc)
    )
```

**建议 3: 自定义异常类层次**

```python
class ConfigException(Exception):
    error_code = "CONFIG-001"
    def __init__(self, message: str):
        self.message = message

class RepositoryNotInitializedError(ConfigException):
    error_code = "CONFIG-001"
```
