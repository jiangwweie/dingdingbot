## 各阶段详细说明

### 【阶段 0】需求接收与澄清

**触发条件**: 用户提出功能需求

**负责人**: Product Manager (PdM)

**交互式沟通检查点** (强制):
- [ ] 提出至少 3 个澄清问题
- [ ] 复述需求并获得用户确认
- [ ] 调用 `brainstorming` 技能探索需求边界

**执行步骤**:

1. **判断任务复杂度**
   - 涉及前端 + 后端 + 测试 → 启动全自动工作流
   - 仅单一模块 → 直接分配对应角色

2. **需求澄清** (交互式)
   ```
   用户："我想加个止损功能"
   
   → PdM 提问:
      1. 止损触发条件是什么？(价格突破/时间截止/手动触发)
      2. 止损执行方式？(市价/限价/条件单)
      3. 是否需要分级止损？(部分止损/全部止损)
   
   → 复述确认:
      "我理解的需求是：当价格跌破 X% 时，自动以市价卖出 Y% 仓位。对吗？"
   ```

3. **调用 brainstorming** (复杂需求)
   - 探索需求边界
   - 识别隐性依赖和风险点

4. **输出**: 
   - `docs/products/<feature>-brief.md` (PRD)
   - 状态看板初始化 (`docs/planning/task-board.md`)

---

### 【阶段 1】契约设计（对齐的关键！）

**目标**: 创建《接口契约表》，作为后续所有工作的 SSOT

**负责人**: Architect (Arch)

**交互式沟通检查点** (强制):
- [ ] 提出至少 2 个技术方案选项
- [ ] 解释每个方案的 trade-off
- [ ] 获得用户确认技术方向

**执行步骤**:

1. **阅读 PRD**，理解需求

2. **技术方案设计** (交互式)
   ```
   → Arch 提供方案:
      方案 A: 使用状态机模式，显式状态转换
         优点：状态清晰，易测试
         缺点：代码量较大
      
      方案 B: 使用事件驱动，隐式状态管理
         优点：代码简洁
         缺点：状态转换不直观
   
   → 用户确认: "选择方案 A"
   ```

3. **创建契约文件**
   - 路径：`docs/designs/<task-name>-contract.md`
   - 使用模板：`docs/templates/contract-template.md`

4. **填写契约表** (精确到字段约束)

5. **更新状态看板**:
   ```markdown
   | 阶段 1 契约设计 | ✅ 已完成 | Arch | YYYY-MM-DD | - |
   ```

---

### 【阶段 2】任务分解与依赖

**目标**: 将需求拆分为可并行执行的独立任务

**负责人**: Project Manager (PM)

**交互式沟通检查点** (强制):
- [ ] 产品范围确认
- [ ] 技术方案确认
- [ ] 任务计划 (工时、排期) 确认

**执行步骤**:

1. **任务分解**
   ```python
   # 使用 TaskCreate 创建任务
   - Task 1: 后端 - 实现 API Schema 定义 (2h)
   - Task 2: 后端 - 实现业务逻辑 (3h)
   - Task 3: 后端 - 编写单元测试 (2h)
   - Task 4: 前端 - 定义 TypeScript 类型 (1h)
   - Task 5: 前端 - 实现 UI 组件 (3h)
   - Task 6: 前端 - 编写组件测试 (2h)
   - Task 7: 测试 - 编写集成测试 (3h)
   ```

2. **标注依赖关系**
   ```python
   # 使用 addBlockedBy 设置依赖
   TaskUpdate(taskId="4", addBlockedBy=["1"])  # 前端类型依赖后端 Schema
   TaskUpdate(taskId="5", addBlockedBy=["1", "4"])  # UI 实现依赖 Schema+ 类型
   TaskUpdate(taskId="7", addBlockedBy=["3", "6"])  # 集成测试依赖单元测试
   ```

3. **识别并行任务簇** ⭐
   ```
   第一簇 (先行):
   - T1: 后端 Schema 定义
   
   第二簇 (并行):
   - T2: 后端业务逻辑 ← 依赖 T1
   - T4: 前端类型定义 ← 依赖 T1
   
   第三簇 (并行):
   - T3: 后端单元测试 ← 依赖 T2
   - T5: 前端 UI 实现 ← 依赖 T4
   - T6: 前端组件测试 ← 依赖 T5
   
   第四簇 (串行):
   - T7: QA 集成测试 ← 依赖 T3+T6
   ```

4. **更新状态看板**:
   ```markdown
   | 阶段 2 任务分解 | ✅ 已完成 | PM | YYYY-MM-DD | - |
   ```

5. **请求用户确认** (强制)
   ```
   📋 任务计划已分解，请确认:
   
   - 产品范围：7 个任务，覆盖后端 + 前端 + 测试
   - 技术方案：方案 A (状态机模式)
   - 预计工时：16 小时
   - 关键路径：T1→T2→T3→T7 (7h)
   
   请回复"确认"开始执行。
   ```

### 【阶段 3】并行开发 ⭐

**目标**: 前后端并行开发，最大化执行效率

**负责人**: Team Coordinator

**自动调度逻辑**:
```python
# Coordinator 检测到用户确认后自动执行

# 1. 解析任务依赖图
任务依赖 = {
    "T1": [],
    "T2": ["T1"],
    "T3": ["T2"],
    "T4": ["T1"],
    "T5": ["T4"],
    "T6": ["T5"],
    "T7": ["T3", "T6"]
}

# 2. 识别可并行任务簇
并行簇 = [
    ["T1"],              # 簇 1: 先行任务
    ["T2", "T4"],        # 簇 2: 可并行
    ["T3", "T5", "T6"],  # 簇 3: 可并行
    ["T7"]               # 簇 4: 串行
]

# 3. 分派任务 (使用 dispatching-parallel-agents)
for 簇 in 并行簇:
    调用 Agent(subagent_type=对应角色，prompt=任务描述)

# 4. 等待完成，更新状态看板
```

**执行步骤**:

1. **启动并行簇 1**
   ```
   → 分配给 Backend: T1 (API Schema)
   → 状态看板更新：T1 🔄 进行中
   ```

2. **检测 T1 完成 → 自动启动簇 2**
   ```
   → 检测：T1 标记✅已完成
   → 自动分配:
      - Backend: T2 (业务逻辑)
      - Frontend: T4 (类型定义)
   → 状态看板更新：T2 🔄, T4 🔄
   ```

3. **检测簇 2 完成 → 自动启动簇 3**
   ```
   → 检测：T2✅, T4✅
   → 自动分配:
      - Backend: T3 (单元测试)
      - Frontend: T5 (UI 实现)
      - Frontend: T6 (组件测试)
   → 状态看板更新：T3 🔄, T5 🔄, T6 🔄
   ```

4. **定期同步状态** (每 30 分钟)
   ```markdown
   📊 并行开发进度同步:
   
   已完成：T1, T2, T4
   进行中：T3(60%), T5(30%), T6(0%)
   阻塞事项：无
   ```

5. **完成后进入审查阶段**
   ```
   → 检测：T3✅, T5✅, T6✅
   → 自动进入阶段 4
   → 状态看板更新：阶段 3 ✅ 已完成
   ```

### 【阶段 4】审查与验证

**目标**: 确保代码质量，契约对齐

**负责人**: Code Reviewer

**Coordinator 自动调度逻辑** ⭐:
```python
# 自动检测审查状态

# 1. 检测所有开发任务完成
if all(task.status == "✅" for task in 开发任务):
    # 2. 自动分配审查任务给 Reviewer
    调用 Agent(subagent_type="code-reviewer", prompt="""
        请对以下文件进行代码审查:
        - src/domain/xxx.py
        - src/application/xxx.py
        - gemimi-web-front/src/components/xxx.tsx
        
        审查重点:
        1. 对照契约表检查字段一致性
        2. 检查必填等级和约束条件
        3. 架构分层是否正确
        4. 无 float 污染 (使用 Decimal)
        
        审查完成后输出审查报告。
    """)
    
    # 3. 更新状态看板
    更新看板 ("阶段 4 审查验证", "🔄 进行中", "Reviewer")
```

**执行步骤**:

1. **Reviewer 介入审查**
   - 对照契约表检查 API 字段
   - 检查类型定义一致性
   - 代码质量审查

2. **审查问题处理**
   ```
   - 小问题 → 直接通知对应角色修复
   - 大问题 → 创建修复任务，重新进入审查循环
   ```

3. **审查通过 → 自动进入测试阶段** ⭐
   ```python
   # Reviewer 标记审查通过
   if 审查结果 == "批准":
       # 自动通知 Coordinator 调度 QA
       Coordinator.自动调度_QA()
   ```

4. **更新状态看板**:
   ```markdown
   | 阶段 4 审查验证 | ✅ 已完成 | Reviewer | YYYY-MM-DD | - |
   审查问题：0 严重 + 2 一般 + 1 建议 (全部修复)
   ```

### 【阶段 5】测试执行

**目标**: 验证功能完整性，确保无回归

**负责人**: QA Tester

**Coordinator 自动调度逻辑** ⭐:
```python
# 审查通过后自动执行

def 自动调度_QA():
    """阶段 4 完成后自动触发"""
    
    # ⚠️ 测试前必须通知用户确认 (测试非常耗时)
    通知用户 ("""
        📋 审查阶段已完成，准备进入测试执行阶段。
        
        测试任务:
        - 单元测试：tests/unit/ (预计 5-10 分钟)
        - 集成测试：tests/integration/ (预计 10-20 分钟)
        - E2E 测试：如需要 (预计 20-30 分钟)
        
        预计总耗时：30-60 分钟
        
        请回复"开始测试"确认执行，或"跳过测试"直接交付。
    """)
    
    # 等待用户确认
    if 用户确认 == "开始测试":
        调用 Agent(subagent_type="qa-tester", prompt="""
            请对以下功能进行测试:
            - 功能：[功能名称]
            - 契约表：docs/designs/xxx-contract.md
            - 代码路径：src/xxx, gemimi-web-front/xxx
            
            测试任务:
            1. 运行单元测试 (覆盖率≥80%)
            2. 运行集成测试
            3. E2E 测试 (如需要)
            4. 生成测试报告
        """)
        
        更新看板 ("阶段 5 测试执行", "🔄 进行中", "QA")
    else:
        # 用户选择跳过测试
        更新看板 ("阶段 5 测试执行", "⏭️ 已跳过", "QA")
        直接进入阶段 6()
```

**执行步骤**:

1. **运行单元测试**
   ```bash
   pytest tests/unit/ -v --cov=src --cov-report=html
   ```

2. **运行集成测试**
   ```bash
   pytest tests/integration/ -v
   ```

3. **生成测试报告**
   ```markdown

   | 测试类型 | 通过率 | 说明 |
   |---------|--------|------|
   | 单元测试 | 95% | 45/47 通过 |
   | 集成测试 | 100% | 12/12 通过 |
   | 覆盖率 | 85% | 目标≥80% ✅ |
   ```

4. **测试结果处理**
   - 全部通过 → 进入提交阶段
   - 有失败 → 创建修复任务，返回对应角色

5. **更新状态看板**:

## 异常处理机制

### 问题分级

| 级别 | 类型 | 示例 | 处理方式 |
|------|------|------|----------|
| **L1 简单** | 语法错误、小 Bug | 拼写错误、类型错误 | 角色自行修复 |
| **L2 中等** | 接口理解偏差 | 字段类型不匹配 | Coordinator 协调对齐 |
| **L3 严重** | 技术阻塞、需求变更 | 依赖缺失、架构冲突 | 标记 blocked，最后汇报 |

### 阻塞检测与恢复

```python
# Coordinator 持续监控

阻塞检测：
    for task in 所有任务:
        if task.阻塞时间 > 30 分钟:
            标记为 "🔴 阻塞"
            通知 PM 介入
    
阻塞恢复：
    if L3 问题解决:
        解除 blocked 状态
        重新分配任务
        继续流水线
```
