## 任务一：策略参数可配置化分析与梳理

### 1.1 需求背景

**当前状态**: 系统已实现动态策略引擎（`DynamicStrategyRunner`），但策略参数（如 Pinbar 的影线比例、EMA 周期等）硬编码在代码中，用户无法通过界面配置。

**问题陈述**: 高级交易员需要根据市场环境调整策略参数，当前必须修改代码并重启服务，使用门槛高且中断交易。

---

### 1.2 用户故事

#### 核心用户画像

| 用户类型 | 特征 | 技术能力 | 使用频率 |
|----------|------|----------|----------|
| **高级交易员** | 有丰富交易经验，熟悉技术参数 | 中 - 理解 EMA、ATR 等指标 | 高 - 每日调整 |
| **量化分析师** | 需要回测不同参数组合的效果 | 高 - 熟悉 Python/JSON | 中 - 周级调优 |

#### 用户故事列表

**US-1: 策略参数配置**
> 作为高级交易员，我希望在策略工作台配置 Pinbar、EMA 等策略参数，以便适应不同的市场环境。

**验收标准**:
- [ ] AC1: 用户可在 UI 界面编辑 Pinbar 参数（最小影线比、最大实体比等）
- [ ] AC2: 用户可配置 EMA 周期、MTF 使能状态
- [ ] AC3: 配置修改后通过热重载 API 即时生效，无需重启服务
- [ ] AC4: 配置保存后持久化，服务重启后自动加载

**US-2: 策略参数模板**
> 作为量化分析师，我希望保存和加载策略参数模板，以便快速切换不同市场策略。

**验收标准**:
- [ ] AC1: 用户可保存当前配置为命名模板（如"牛市激进"、"熊市保守"）
- [ ] AC2: 用户可从模板列表加载历史配置
- [ ] AC3: 模板支持导出/导入（JSON 格式）

**US-3: 参数预览与验证**
> 作为交易员，我希望在应用参数前预览效果，避免错误配置导致损失。

**验收标准**:
- [ ] AC1: 提供"Dry Run"预览功能，显示配置变更对比
- [ ] AC2: 参数超出合理范围时给出警告（如 EMA 周期<5 或>200）
- [ ] AC3: 预览成功后方可确认应用

|----------|------|
| **业务价值** | 高 - 降低使用门槛，提升用户留存 |
| **用户痛点** | 高 - 当前必须修改代码，体验极差 |

### 1.5 技术复杂度与依赖

| 维度 | 评估 | 说明 |
|------|------|------|
| **技术复杂度** | 中 | 需要设计配置表 schema，迁移现有配置 |
| **前端依赖** | 策略工作台组件（已有 80%） |
| **后端依赖** | ConfigManager 热重载 API（已完成） |
| **数据库依赖** | 新增 config_entries 表，需迁移脚本 |
| **测试依赖** | 单元测试 + E2E 测试 |

**架构决策** (2026-04-02):
- **配置存储方案**: 数据库存储 (SQLite) 替代 YAML 文件
- **运行时**: 配置直接从数据库加载，无需解析外部文件
- **YAML 角色**: 仅用于导入导出备份，不作为运行态存储
- **优势**: 启动无需加载外部文件、事务支持、与快照系统无缝集成

**技术风险**:
- 参数校验不充分可能导致策略异常 → 缓解：严格 Pydantic 验证 + 边界测试
- 热重载失败导致配置不一致 → 缓解：原子性更新 + 回滚机制
- 配置迁移失败 → 缓解：提供回滚脚本，保留原 YAML 备份

## 任务三：订单管理级联展示功能

### 3.1 需求背景

**当前状态**: 系统已实现订单链（Order Chain）概念，ENTRY 订单成交后动态生成 TP/SL 子订单，但前端仅展示扁平订单列表，用户无法直观理解订单间的层级关系。

**问题陈述**: 用户无法快速查看某笔开仓订单关联的所有止盈/止损子订单，导致复盘和风控分析困难。

---

### 3.2 用户故事

#### 核心用户画像

| 用户类型 | 特征 | 技术能力 | 使用频率 |
|----------|------|----------|----------|
| **交易员** | 需要追踪订单执行进度 | 中 - 理解 OCO 逻辑 | 高 - 每日多次 |
| **风控人员** | 需要分析订单链风险敞口 | 高 - 熟悉订单结构 | 中 - 日级审查 |

#### 用户故事列表

**US-1: 订单层级展示**
> 作为交易员，我希望在订单列表中展开查看父订单及其子订单（TP/SL），以便了解完整的订单执行链。

**验收标准**:
- [ ] AC1: 支持树形展开/折叠订单链
- [ ] AC2: 父订单显示子订单数量徽章
- [ ] AC3: 子订单缩进显示，视觉上区分层级
- [ ] AC4: 展开状态下可看到 TP1/TP2/SL 等子订单

**US-2: 订单链状态汇总**
> 作为风控人员，我希望在父订单上看到订单链的整体状态，以便快速评估风险。

**验收标准**:

## 优先级评估 (RICE 模型)

### RICE 评分标准

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| **R**each - 覆盖范围 | 25% | 影响多少用户 |
| **I**mpact - 影响力 | 25% | 业务价值大小 |
| **C**onfidence - 信心度 | 10% | 假设置信度 |
| **E**ffort - 投入成本 | 40% | 人天估算 |

**公式**: `Priority = (R × I × C) / E`

### 优先级决策理由

#### P0: 策略参数可配置化

**理由**:
1. **最高用户痛点**: 当前必须修改代码配置参数，使用门槛极高
2. **战略价值**: 降低用户使用门槛，提升产品竞争力
3. **技术可行性**: 已有热重载基础设施，实现风险低
4. **依赖关系**: 不依赖其他功能，可独立交付

**风险**: 延迟交付可能导致用户流失

---

#### P1: 订单管理级联展示

**理由**:
1. **高使用频率**: 所有订单用户都会受益
2. **数据基础**: 后端已有订单链概念，只需前端展示
3. **竞争对标**: 专业交易系统标准功能
4. **用户体验**: 显著提升订单复盘效率

**风险**: 延迟交付影响用户体验，但不影响核心交易

---

#### P2: 订单详情页 K 线渲染 (TradingView)

**理由**:
1. **锦上添花**: 提升专业体验，但非核心功能
2. **替代方案**: 用户可使用第三方图表工具
3. **技术复杂度**: 相对较高，学习成本大
4. **性能考量**: 大数据量渲染需优化

**风险**: 延迟交付影响较小，可后续迭代
