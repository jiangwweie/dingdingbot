## 问题清单

### P0-1: FilterConfig.params 类型安全缺口

**现状**:
```python
class FilterConfig(BaseModel):
    type: str
    params: Dict[str, Any]  # 类型逃逸洞
```

**风险**:
- 攻击注入：`{"period": "DROP TABLE signals"}`
- 后端无校验，信任边界破裂
- 每个 Filter 需手写字典校验，代码重复

**影响范围**: FATAL
**实施成本**: 中

**建议方案**:
1. 为每种 Filter 创建具名 Pydantic 类
   - `EmaFilterParams(period: int, source: str)`
   - `MtfFilterParams(higher_timeframe: str)`
   - `AtrFilterParams(period: int, threshold: Decimal)`
2. 使用 Discriminator Union 基于 `type` 字段自动路由
3. 与子任务 F 的 LogicNode 模式对齐

**验收标准**:
- 移除 `Dict[str, Any]`
- 每个 Filter 类型有独立参数模型
- 反序列化自动类型校验

### P0-2: 热重载并发锁粒度过粗

**现状**:
```python
async def on_config_updated(self, new_config):
    async with self._runner_lock:
        self._runner = create_dynamic_runner(new_config)
        await self._warmup_runner(self._kline_history)  # 200+ K 线
```

**风险**:
- Warmup 耗时：200 根 × N 策略 × M 币种 = 秒级阻塞
- 阻塞期间新 K 线到达无法处理
- 实盘场景下，阻塞 = 信号延迟 = 滑点

**影响范围**: CRITICAL
**实施成本**: 高

**建议方案**:
1. 双缓冲策略
   - 后台创建新 Runner 并 Warmup
   - 完成后原子切换指针
2. 使用 `asyncio.Semaphore` 限制并发 Warmup 数量
3. 增量 Warmup：仅回放新增 K 线

**伪代码**:
```python
async def on_config_updated(self, new_config):
    # 后台 Warmup
    new_runner = await self._create_and_warmup_runner(new_config)
    # 原子切换
    async with self._pointer_lock:
        self._current_runner = new_runner
```

**验收标准**:
- 主事件循环无阻塞
- K 线处理延迟 < 100ms
- 并发 Apply 不崩溃

### P1-3: 测试覆盖盲区

**现状**: 284 个单元测试，100% 通过

**未覆盖场景**:

| 场景 | 风险 | 优先级 |
|------|------|--------|
| WebSocket 断线重连 | 实盘崩溃无感知 | 高 |
| 交易所 API 限流 | 触发 C-001 但无降级 | 高 |
| 配置热重载时 K 线到达 | 竞态条件 | 中 |
| SQLite 写入失败 | 信号丢失无告警 | 中 |
| 通知渠道全部失败 | 用户收不到信号 | 高 |

**影响范围**: HIGH
**实施成本**: 中

**建议方案**:
1. 集成测试：模拟交易所异常、网络抖动
2. 混沌工程：随机 kill WebSocket、注入延迟
3. 覆盖率报告聚焦关键路径

**新增测试文件**:
- `tests/integration/test_exchange_failures.py`
- `tests/integration/test_websocket_reconnect.py`
- `tests/integration/test_notification_failures.py`

### P2-4: 递归深度限制可能成为瓶颈

**现状**: `LogicNode` 限制递归深度 ≤ 3

**风险**:
- 复杂策略如 `((A OR B) AND C) OR ((D AND E) OR F)` 超限
- 专业用户可能需要 5-7 层嵌套

**影响范围**: MEDIUM
**实施成本**: 低

**建议方案**:
1. 改为动态配置 `max_recursion_depth` (默认 3, 可上调至 7)
2. 前端 TraceTreeViewer 支持折叠/展开深度控制
3. 性能测试验证 7 层嵌套评估延迟

**验收标准**:
- 配置项生效
- 7 层嵌套延迟 < 50ms
- 前端渲染不卡顿

### P2-5: 历史数据兼容性未处理

**现状**: `SignalResult.tags` 改为动态数组

**风险**:
- 旧数据含 `ema_trend` 字段，新代码查询可能崩溃
- SQLite Schema 未版本化
- 前端需兼容新旧格式

**影响范围**: MEDIUM
**实施成本**: 低

**建议方案**:
1. 数据迁移脚本：旧记录补 `tags` 字段
2. 查询层格式统一（Legacy → Tags 转换）
3. SQLite Schema 引入 `PRAGMA user_version`

**验收标准**:
- 历史信号正常查询
- 新旧格式自动转换
- 前端无崩溃

### P3-6: 策略 Apply 无审计日志

**现状**: `POST /api/strategies/{id}/apply` 已实现

**缺失**:
- 谁在何时应用了哪个策略？
- 应用前后的配置差异？
- 回滚入口？

**影响范围**: LOW
**实施成本**: 低

**建议方案**:
1. 新增 `StrategyAuditLog` 表
2. 提供 `GET /api/strategies/audit` 接口
3. 前端展示"最近应用"时间线

**数据模型**:
```python
class StrategyAuditLog(BaseModel):
    id: str
    strategy_id: str
    operator: str
    timestamp: datetime
    old_config_hash: str
    new_config_hash: str
```
