### 1.4 关键发现

1. **orders 存在双轨**：`CORE_ORDER_BACKEND` 控制走 SQLite `OrderRepository` 还是 PG `PgOrderRepository`，运行时只写一个，但两个数据库可能同时存在历史数据

4. **execution_recovery_tasks 已 PG-only**：无 SQLite 实现，是唯一纯 PG 表
5. **API 层有 6 处直接实例化 `OrderRepository()`**（api.py），绕过工厂，始终写 SQLite

### 2.3 跨库点

| 跨库调用 | 场景 | 风险 |
|----------|------|------|
| Signal(SQLite) → ExecutionIntent(PG) | `signal_id` 跨库外键，无法数据库级约束 | 数据完整性依赖应用层 |
| Order(PG) 的 `signal_id` → Signal(SQLite) | 同上 | 同上 |
| `/api/runtime/signals` 读 SQLite | Console 查询信号 | 与执行链数据不在同库 |
| `/api/runtime/attempts` 读 SQLite | Console 查询尝试记录 | 同上 |

### 2.4 各对象写入路径详解

#### Orders 写入路径

```
Signal received
  │
  ▼
ExecutionOrchestrator.execute_signal()
  │
  ├── OrderLifecycleService.create_order()           [ENTRY]
  │     ├── OrderManager.create_order_chain()        [generate Order object]
  │     └── _repository.save(order)                  [PERSIST: CREATED]
  │
  ├── ExchangeGateway.place_order()                   [submit to exchange]
  │
  ├── OrderLifecycleService.submit_order()            [CREATED→SUBMITTED]
  │     └── _repository.save(order)                  [PERSIST: SUBMITTED]
  │
  └── OrderLifecycleService.confirm_order()           [SUBMITTED→OPEN]
        └── _repository.save(order)                  [PERSIST: OPEN]

Exchange WebSocket update
  │
  ▼
OrderLifecycleService.update_order_from_exchange()
  ├── _repository.get_order_by_exchange_id()          [READ]
  ├── state machine transition
  └── _repository.save(local_order)                   [PERSIST: new status]

ENTRY filled (WebSocket or reconciliation)
  │
  ▼
ExecutionOrchestrator._handle_entry_filled()
  ├── OrderManager._generate_tp_sl_orders()           [generate TP/SL objects]
  ├── _order_lifecycle._repository.save(prot_order)   [PERSIST: CREATED -- DIRECT BYPASS]
  ├── ExchangeGateway.place_order()                   [submit TP/SL to exchange]
  ├── OrderLifecycleService.submit_order()            [CREATED→SUBMITTED]
  └── OrderLifecycleService.confirm_order()           [SUBMITTED→OPEN]
```

**关键问题**：
- ExecutionOrchestrator 有 2 处直接访问 `_order_lifecycle._repository.save()`（lines 571, 988），绕过 OrderLifecycleService 的状态机流程
- API 层有 6 处直接实例化 `OrderRepository()`（始终 SQLite），绕过工厂

#### ExecutionIntents 写入路径

```
ExecutionOrchestrator.execute_signal()
  │
  ├── intent = ExecutionIntent(...)                   [创建对象]
  ├── intent_repo.save(intent)                        [PERSIST: PG INSERT]
  │
  ├── [后续状态变更]
  │     ├── intent_repo.update_status(intent_id, BLOCKED, ...)
  │     ├── intent_repo.update_status(intent_id, SUBMITTED, ...)
  │     └── intent_repo.update_status(intent_id, COMPLETED, ...)
  │
  └── SQLite 模式: intent_repo = None → 内存 dict（无持久化）
```

**已闭环**：PG-only，SQLite 路径返回空响应。

#### Positions 写入路径

```
                        PRODUCTION RUNTIME
                        ==================

  Exchange (Binance)
       │
       │  [READ-ONLY source of truth]
       ▼
  ExchangeGateway.fetch_positions()
  ExchangeGateway.get_account_snapshot()
       │
       ├── API endpoints (close position, runtime positions)
       ├── ReconciliationService (local position comparison)
       │
       │  [NO WRITE-BACK to any local DB]

  ExecutionOrchestrator
       │
       │  Creates ephemeral Position(domain) objects in-memory
       │  (lines 533, 791) -- used only for TP/SL generation
       │  NEVER persisted
       ▼
  OrderManager._generate_tp_sl_orders()
       │
       │  Generates Order objects (persisted via OrderLifecycleService)
       ▼
  OrderRepository (SQLite or PG) -- orders are persisted, positions are not


                        UNACTIVATED CODE
                        ================

  PositionManager (position_manager.py)
       │
       │  create_position()  →  PositionORM  →  SQLite (NEVER CALLED)
       │  reduce_position()  →  PositionORM  →  SQLite (NEVER CALLED)
       ▼
  SQLite `positions` table (ALWAYS EMPTY at runtime)

  PgPositionRepository (pg_position_repository.py)
       │
       │  save()  →  PGPositionORM  →  PG (NEVER CALLED)
       ▼

```

**设计缺口**：系统跟踪 orders 和 execution_intents，但不持久化 resulting positions。持仓只能从订单状态推断或直接查询交易所。

## 4. 风险清单

### R1: Orders 双真源风险 [P0]

- **现状**：`CORE_ORDER_BACKEND` 环境变量控制写入 PG 还是 SQLite，但两个数据库可能同时存在历史数据
- **场景**：切换环境变量后，旧数据留在另一库，新数据写入新库，查询结果不完整
- **影响**：订单丢失、重复、状态不一致
- **根因**：工厂模式设计为双轨切换而非单源收敛

### R2: Positions 无本地持久化 [P0]

- **现状**：`PositionManager` 未激活（main.py 中 `position_manager=None`），PG `PgPositionRepository.save()` 无调用者，运行时完全依赖交易所 `account_snapshot`
- **场景**：交易所 API 不可用时，无法查询持仓；重启后无本地仓位恢复能力
- **影响**：风控判断缺失、保护订单无法挂载、Trailing TTP 无法推演
- **根因**：Phase 3/4 设计中 Position 持久化未完成

### R3: Signal → ExecutionIntent 跨库外键 [P1]

- **现状**：`execution_intents.signal_id` 引用 SQLite `signals` 表，无数据库级约束
- **场景**：Signal 被删除后，ExecutionIntent 成为孤儿记录
- **影响**：数据完整性依赖应用层，调试困难
- **根因**：执行链迁移到 PG 时，信号域未同步迁移

### R4: Console 观察口径偏差 [P1]

- **现状**：`/api/runtime/signals` 和 `/api/runtime/attempts` 读 SQLite，而执行链数据在 PG
- **场景**：前端展示信号→意图→订单的完整链路时，需跨库关联
- **影响**：无法在一次查询中获取完整执行链路
- **根因**：Console API 分批开发时未统一数据源

### R5: OrderAuditLog 仅 SQLite [P2]

- **现状**：`order_audit_logs` 表仅在 SQLite 迁移脚本中定义，无 PG 对应
- **场景**：订单审计记录与订单主体不在同一库
- **影响**：审计追溯需跨库查询
- **根因**：审计日志在 Phase 2 添加时未考虑 PG 迁移

### R6: Reconciliation 独立 SQLite [P2]

- **现状**：对账数据存储在独立的 `data/reconciliation.db`
- **场景**：对账报告与执行链数据分散
- **影响**：运维复杂度
- **根因**：对账模块独立开发

### R7: API 层绕过工厂 [P2]

- **现状**：api.py 有 6 处直接实例化 `OrderRepository()`（始终 SQLite），绕过 `create_order_repository()` 工厂
- **场景**：PG 部署时，这些端点仍写 SQLite
- **影响**：回测订单和单次查询端点的数据与主流程不一致
- **根因**：API 层开发时未统一使用工厂

## 5. 两个架构方案

### 方案 A：最小切面完成执行主线 PG 闭环

**目标**：让 4 个执行主线对象（orders, execution_intents, positions, execution_recovery_tasks）的读写全部收敛到 PG，消除双真源。

**变更范围**：

| 变更项 | 内容 | 工作量 |
|--------|------|--------|
| A1: Orders PG-only | 移除 `CORE_ORDER_BACKEND` 工厂切换，硬编码使用 `PgOrderRepository`；SQLite `OrderRepository` 保留但不再作为运行时选项 | 0.5d |
| A2: Positions 写入激活 | 在 `ExecutionOrchestrator` 中注入 `PgPositionRepository`，在开仓/减仓/平仓流程中调用 `save()`；`RuntimePositionsReadModel` 增加 PG fallback | 2d |
| A3: Positions Schema 补齐 | 当前 PG `positions` 表缺少 `watermark_price`, `total_fees_paid`, `current_qty` 等字段（对比 SQLite PositionORM），需补齐列 + 迁移脚本 | 0.5d |
| A4: Console API 对齐 | `/api/runtime/execution/orders` 确保只读 PG；`/api/runtime/positions` 增加 PG 数据源 | 0.5d |
| A5: 数据迁移脚本 | 提供一次性脚本将 SQLite orders 历史数据迁移到 PG | 0.5d |
| A6: API 层工厂统一 | api.py 6 处直接 `OrderRepository()` 改为使用工厂或 PG repo | 0.5d |

**不动的部分**：
- signals / signal_attempts 保留 SQLite
- config 全家桶保留 SQLite
- backtest 保留 SQLite
- reconciliation 保留独立 SQLite
- order_audit_logs 保留 SQLite（本次不迁移）

**风险**：
- A2 是最大风险点：positions 写入激活意味着需要确保与交易所状态的一致性
- 需要明确定义"谁写 positions"——建议在 OrderLifecycleService 订单 FILLED 时触发 position upsert
- PG positions 表 Schema 与 domain model 有差异（PG 用 JSONB `position_payload` 捕获额外字段），需评估是否改为显式列

**总工作量**：~4.5d

---

### 方案 B：执行主线 PG 闭环 + Runtime Observability 收口

**目标**：在方案 A 基础上，将 runtime console 所有执行链观察数据统一从 PG 读取，消除"前端看到的数据来自不同数据库"的问题。

**额外变更**：

| 变更项 | 内容 | 工作量 |
|--------|------|--------|
| B1: 方案 A 全部内容 | 同上 | ~4.5d |
| B2: Signal 摘要入 PG | 在 `execution_intents.signal_payload` JSONB 中已包含信号快照；新增 `signal_summary` PG 视图或物化视图，供 console 查询 | 1d |
| B3: Attempts 摘要入 PG | 在 `execution_intents` 中已有策略评估信息；新增 `attempt_summary` PG 视图，替代 SQLite `signal_attempts` 查询 | 1d |
| B4: OrderAuditLog 入 PG | 新建 PG `order_audit_logs` 表，`OrderLifecycleService` 写入 PG | 0.5d |
| B5: Runtime Console 统一查询 | `/api/runtime/signals` 和 `/api/runtime/attempts` 改为读 PG 视图（保留 SQLite 作为 fallback） | 1d |

**风险**：
- B2/B3 的"摘要入 PG"是只读视图，不影响写入路径，风险可控
- B5 需要前端适配新的响应格式（如果字段有变化）
- 增加的 PG 视图需要维护，与 SQLite 表 Schema 变化同步

**总工作量**：~8d

## 6. 推荐方案及理由

**推荐方案 A**，理由：

1. **最小变更原则**：方案 A 只解决"执行主线 4 对象 PG 闭环"这个核心问题，不动 signals/attempts/config 等非执行链模块
2. **Positions 是唯一真正需要开发的点**：execution_intents 和 execution_recovery_tasks 已 PG-only；orders 只需移除工厂切换；positions 是唯一需要"从零激活写入"的对象
3. **方案 B 的 Observability 收口可以后续迭代**：B2-B5 的 PG 视图方案是纯读取层优化，不涉及写入路径变更，可以独立排期
4. **风险隔离**：方案 A 的变更集中在 ExecutionOrchestrator 和 Repository 层，不影响 config/backtest/research 模块

### 方案 A 关键设计决策点（需用户确认）

| 决策点 | 选项 | 建议 | 理由 |
|--------|------|------|------|
| Positions 写入时机 | a) OrderLifecycleService FILLED 时触发<br>b) ExecutionOrchestrator 直接写 | **a)** | OrderLifecycleService 是所有订单状态变更的唯一入口，在此触发 position upsert 可确保一致性 |
| Positions Schema 补齐方式 | a) 显式列（watermark_price, total_fees_paid 等）<br>b) 继续用 JSONB position_payload | **a)** | 显式列支持索引和约束查询，JSONB 仅适合不常查询的扩展字段 |
| SQLite OrderRepository 处置 | a) 立即删除代码<br>b) 标记 deprecated，保留但默认 PG | **b)** | 回测仍使用 SQLite OrderRepository，立即删除会破坏回测流程 |
| CORE_ORDER_BACKEND 环境变量 | a) 移除环境变量，硬编码 PG<br>b) 保留但默认值改为 postgres | **b)** | 保留开发/测试时切换 SQLite 的灵活性 |

## 7. 明确不在本次范围内的模块
