# 子任务F进阶开发提示词：递归逻辑树引擎与强类型约束重构 (SubTask F - High Level)

**角色设定**：你是一位精通 Pydantic 高级特性（Union, Discriminator, Recursive Models）以及算法设计的资深后端架构师。你需要将当前的“平铺式”策略引擎升级为“工业级递归表达式树引擎”。

## 核心任务要求

### 1. 重构领域模型 (Recursive Pydantic Models)
在 `src/domain/models.py` 中，你需要实现以下强类型契约：
- **具体参数类**：为 `Pinbar`, `Engulfing`, `EMA`, `ATR`, `VolumeSurge` 等每个指标建立专有的 `BaseModel`（如 `EmaFilterParams`）。
- **递归逻辑节点**：
  - 定义一个 `LogicNode` 类，包含 `gate: Literal["AND", "OR", "NOT"]` 和 `children: List[...]`。
  - 使用 `Annotated` 与 `Union` 配合 `discriminator="type"`，将 `LogicNode`、`TriggerNode`、`FilterNode` 组合成一个多态实体。
  - **深度防御**：在 `LogicNode` 中通过 `model_validator` 检查嵌套深度，严禁超过 3 层以防递归溢出。

### 2. 重塑 Meta API (Schema Reflexion)
- **接口**：`GET /api/strategies/meta`
- **重构逻辑**：废弃手动维护的字典。直接调用 `StrategyDefinition.model_json_schema()`（或者其子部分）。
- **目标**：确保后端 Class 里加一个 `field`，前端界面通过 Schema 翻译能自动多出一个输入框。

### 3. 实现判定引擎升级 (Recursive Runner)
- **目标**：修改 `DynamicStrategyRunner`。
- **逻辑**：实现一个递归函数 `evaluate_node(node, context)`。
  - 如果遇 `AND` 门，对 children 进行 `all()` 判定并短路。
  - 如果遇 `Leaf`，调用具体的 `detect()` 或 `check()` 函数。
- **Trace 集成**：引擎返回的不再是一个平铺列表，而是一个与逻辑树同构的 **Trace 树**，记录每个分叉的胜败原因。

### 4. 新增“热预览”接口 (Dry Run Endpoint)
- **接口**：`POST /api/strategies/preview`
- **输入**：`{ "symbol": "...", "timeframe": "...", "strategy": { ...逻辑树JSON... } }`
- **逻辑**：不持久化、不热重载。临时拉取该标的的最新 K 线，跑一遍 `evaluate_node`，并将完整的 Trace 树返回。

## 行动指南
你不需要考虑 WebSocket 的并发问题（这属于子任务 A），你的核心精力在于**“把逻辑表达的自由度彻底放开，同时用 Pydantic 把类型安全扎得死死的”**。完成重构后，请确保单元测试覆盖递归逻辑门（如 AND 套 OR）的判定准确性。
