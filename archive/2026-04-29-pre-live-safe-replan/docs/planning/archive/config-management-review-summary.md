## 架构发现

### 发现 1: ConfigManager 存在 God Object 风险

**位置**: `src/application/config_manager.py` (~1600 行)

**问题**:
- 单一类承担过多职责：配置加载、缓存管理、热重载、观察者管理、快照集成
- 违反单一职责原则，可维护性差

**影响**:
- 测试困难（需要 mock 大量依赖）
- 修改风险高（改动可能影响多个功能）
- 团队协作困难（多人修改同一文件易冲突）

**建议方案**:
```
渐进式三层拆分：
第 1 层: 提取独立管理器 (RiskConfigManager, SystemConfigManager)
第 2 层: 迁移业务逻辑
第 3 层: 重构 API 层
```

**工时**: 16h (分阶段)

---

### 发现 2: P1-5 和 P1-6 存在依赖冲突

**问题分析**:
```
P1-5: ConfigManager 拆分 → 需要改变类的结构
P1-6: 依赖注入           → 需要改变类的使用方式

错误顺序：P1-6 → P1-5
- 依赖注入代码需要重写
- 测试代码需要重写
- 增加返工成本

正确顺序：P1-5 → P1-6
- 先完成类结构拆分
- 再注入拆分后的子管理器
- 避免重复工作
```

**决策**: 在 ADR-20260407-001 中记录，P1-5 必须在 P1-6 之前实施

---

### 发现 3: P1-7 异步化风险被低估

**位置**: `config_manager.py:331-401` 初始化逻辑

**当前实现**:
```python
async def initialize_from_db(self):
    # R9.3: 已有竞态修复
    if self._initialized:
        return
    
    # 这里有同步阻塞操作：
    # 1. os.makedirs() - 同步文件 I/O
    # 2. aiosqlite.connect() - 可能阻塞
    # 3. 数据库 schema 创建 - 同步
```

**风险评估**:
- ✅ R9.3 竞态修复已验证（双重检查锁 + 事件循环隔离）
- ⚠️ 同步操作存在，但未造成实际问题
- 📊 性能数据：启动时间 < 1 秒，可接受

**决策**: 暂缓 P1-7 实施，仅补充测试（P1-8），理由：
1. 破坏 R9.3 修复的风险 > 性能提升收益
2. 当前实现已在生产环境稳定运行
3. 无性能瓶颈的迹象

## 并行执行策略发现

### 文件隔离并行策略

**问题**: 多人修改同一文件会导致合并冲突

**解决方案**: 按文件边界分配任务

```
开发者 A (api.py):
├── P0-1: 删除重复端点定义
└── P1-4: 完善脱敏功能

开发者 B (api_v1_config.py):
└── P0-2: 实现快照列表查询
```

**效果**:
- 实际并行度: 2x
- 总耗时: 4 分钟
- 估算串行耗时: 6 小时
- 效率提升: 98.9%

**适用场景**: 无依赖关系的修改任务

## 关键技术决策

### 决策 1: P1-3 权限检查使用增强方案

**背景**: 当前系统为单用户系统，无多租户需求

**方案对比**:

| 方案 | 工时 | 安全提升 | 复杂度 |
|------|------|----------|--------|
| JWT 完整认证 | 8h | 高 | 高 |
| 签名验证增强 | 4h | 中 | 低 |

**选择**: 签名验证增强方案

**理由**:
1. 当前系统无用户管理模块
2. 无登录/注册流程
3. JWT 引入复杂依赖和攻击面
4. 增强方案可满足当前需求（80% 安全提升）

**实施**:
- 签名验证 + 时间戳防重放
- 审计日志记录
- 5 分钟时间窗口

---

### 决策 2: 渐进式重构策略

**问题**: 大爆炸式重构风险高

**策略**: 分层渐进式重构

```
第 0 层: 质量保障（P1-8 并发测试）
第 1 层: 紧急修复（P1-1, P1-2）
第 2 层: 架构重构（P1-5, P1-6）
第 3 层: 安全增强（P1-3）
观望层: 暂缓（P1-7）
```

**优点**:
- 每层都有可工作的系统
- 风险递增，易于控制
- 出问题可快速回滚
